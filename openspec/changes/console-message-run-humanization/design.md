# 设计：console-message-run-humanization

## 方案

### 1. 组件边界与文件形态

新增三个独立复合组件，文件名在实现时保持以下边界：

- `agent-message.tsx`：agent Markdown 摘要提取、中文角色/stage 映射、折叠与展开。
- `run-block.tsx`：运行头部、最小步骤模型、无步骤降级和原始输出展开。
- `run-outcome.tsx`：运行结局到固定中文主文案的映射、机器详情展开。

每个组件旁放置自己的 `*.test.tsx` 与 `*.stories.tsx`。调用点在本切片内直接引用文件；不修改共享 `src/index.ts`，也不把组件接入 `operator-console.tsx`。

### 2. agent 消息解析与覆盖优先级

`AgentMessage` 接收原始 Markdown、角色标识以及可选显式 `stage`、`conclusion`、`handoff` 展示字段。字段解析遵循一致的优先级：

1. 调用方显式字段；
2. 原始 Markdown 自动提取；
3. 不泄露机器协议的中性降级文案。

自动解析规则保持窄而确定：

- 结论：读取 `## 结论` 标题之后、下一个二级标题之前的首个非空内容段；
- stage：读取末尾合法 `agent-moebius:stage=...` marker，仅把 `in-progress`、`plan-written`、`code-verified` 映射为「进行中」「方案已写好」「代码已验证」；
- 交棒：读取 `## 下一步` 段中的合法 `交棒：...` 或 `等待真人：...` 行，在折叠态把合法角色句柄映射为中文角色名；
- 不完整或未知原文不抛错，折叠态使用中性降级，展开层仍完整保留原文供查证。

折叠态使用 `<details>` / `<summary>` 或等价原生可访问披露控件，默认关闭；标题和摘要不使用等宽字体，不展示英文 stage marker。展开层按原始 Markdown 文本完整呈现，协议行和机器词不做删除或改写。

### 3. 运行块最小展示模型

`RunBlock` 自有展示模型，不导入 runtime 类型：

- 角色：使用与 agent 消息相同的中文角色映射；
- 耗时：优先使用调用方传入的已格式化人话字符串；值缺失、为空或纯空白时确定性显示「耗时未知」，组件不推断计时状态；
- 中断：接收 `onInterrupt`，始终渲染可访问的「中断」按钮；
- 步骤：`title`、`status`、`summary`、`rawOutput`，其中 `status` 仅为 `completed | running | pending`；
- 无步骤降级：优先渲染调用方传入的单行 `summary`；值缺失、为空或纯空白时确定性显示「正在运行，等待进展」，不能留空白卡；
- 原始输出：有步骤时跟随各步骤折叠，无步骤时跟随整体概括折叠。没有原始输出时不渲染空详情。

步骤视觉语义遵循 `ui-design.md`：已完成使用完成标记，进行中使用唯一持续的中性呼吸指示，未开始使用中性空标记。中断按钮使用 danger 描边但不把整张卡染红。

### 4. 运行结局人话化

`RunOutcome` 只接受 `failed | stuck | interrupted | dead-letter` 四种展示状态，并使用需求侧确认的固定主文案：

| 状态 | 用户可见主文案 |
|---|---|
| failed | 运行失败 |
| stuck | 运行长时间无响应 |
| interrupted | 运行已中断 |
| dead-letter | 多次尝试仍失败，已停止自动重试 |

机器原因与原始输出作为 `rawReason` / `rawOutput` 进入默认关闭的详情区。折叠态不得渲染 `exit`、`idle-timeout`、`dead-letter` 等原始串。展开后必须逐字可查，不用人话文案覆盖原值。

### 5. 视觉与可访问性

- 复用现有 `Card`、`Button`、`Avatar`、`Badge` 等源码组件和 `tokens.css` 语义类，以 `accept-card.tsx` 为组合活参考。
- agent 折叠态保持无卡片、低视觉权重；展开内容和运行块可使用扁平细边白卡。
- 不新增裸色值、渐变、默认阴影或装饰动画；状态色只表达事实，中性层占主体。
- 所有披露控件可用键盘操作并暴露可理解的 summary；中断按钮具有明确可访问名称。
- 机器原文仅在详情层使用等宽排版，用户可见概括继续使用系统中文字体。

### 6. Story 与验证矩阵

Story 至少覆盖：

| Story | 必查行为 |
|---|---|
| agent 默认折叠 | 中文角色、中文 stage、结论、人话化交棒行可见，原文不可见 |
| agent 展开 | 完整原始 Markdown、stage marker、原始交棒行可见 |
| 运行块有步骤 | 中文角色、耗时、中断按钮及三种步骤状态可见，每步原始输出可展开 |
| 运行块无步骤 | 单行人话概括不为空，中文角色、耗时、中断按钮可见，整体原始输出可展开 |
| 运行块状态丢失 | `steps`、摘要、耗时缺失或纯空白时显示「正在运行，等待进展」和「耗时未知」，不出现空白卡 |
| 四种运行结局 | 折叠态仅见固定中文主文案，展开后分别可见 `exit:42`、`idle-timeout:10ms` 等 fixture 原始串 |

组件测试覆盖显式字段优先、Markdown 自动提取、缺失字段降级、步骤/无步骤分支、空值与纯空白兜底、四状态文案映射和详情披露。交互测试必须以用户可见性为判据：验证 Enter 与 Space 各自单次切换 agent 消息、步骤输出和运行结局详情；验证鼠标与键盘单次激活中断按钮各只调用一次 `onInterrupt`；验证包含换行、尖括号、与号和 `exit:42` 的原始文本在折叠时不可见、展开后 `textContent` 保持原值。不得只用“文本节点存在于 DOM”代替折叠态不可见。Storybook 启动或静态构建用于验证所有 Story 可渲染；实现完成后按六条正式验收语句逐条生成证据。

## 权衡

- 选择窄 Markdown 提取器，而不是引入完整 Markdown AST 依赖：输入骨架稳定，本任务只需要三个字段；窄解析更容易测试，也避免扩大包体和依赖面。完整原文始终保留，因此解析降级不造成信息丢失。
- 选择显式字段覆盖自动结果，而不是让调用方必须预处理：默认开箱可用于现有 agent 原文，同时给未来 runtime 结构化数据留出无破坏接入点。
- 选择组件自有展示模型，而不是复用 `operator-console.tsx` 私有类型：独立 Story 可以稳定演示，后续整合时再由调用层适配，避免本任务越界绑定运行时。
- 选择单一 `RunOutcome` 状态映射，而不是四个近似组件：固定人话映射集中可测，原始详情契约一致，同时 Story 仍逐状态展示。
- 不在本 change 接入主界面或共享出口：父任务已明确把独立组件产出与集成工作拆开，保持 conflict group 的并行安全。

## 风险

- Markdown 原文若偏离输出骨架，自动解析可能只能降级。缓解：解析失败不隐藏原文，显式字段可覆盖，并用非标准 fixture 测试无崩溃。
- 调用方可能同时丢失步骤、摘要与耗时。缓解：组件负责 trim 后判空，分别显示固定中文概括与「耗时未知」，并用缺失、空串和纯空白三组 fixture 固定该分支。
- `<details>` 的默认 marker 和跨浏览器样式可能偏离旧原型。缓解：只定制必要的 caret 和间距，保留原生语义，并在 Storybook 浏览器中检查折叠/展开。
- Story fixture 若只检查文本存在，可能漏掉机器串在折叠态可见。缓解：组件测试分别断言详情默认关闭与展开后可查，并做 Storybook 视觉走查。
- 未来接入 runtime 时字段形态可能不同。缓解：本组件只声明最小展示模型，由未来集成层适配，不在组件中导入 local-console 类型。
- 回滚只需删除新增组件、Story、测试和本 change 文件；本切片不修改既有组件或调用点。
