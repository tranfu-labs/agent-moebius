# 设计：conversation-console

字符图见本目录 `wireframes.md`；本文件只承载设计推导与权衡。

## 心智模型迁移

| 旧世界（GitHub issue 操作台） | 新世界（本地对话操作台） |
|---|---|
| 监听的 repository | 项目（侧栏第一层） |
| 一个 issue | 一个会话（侧栏第二层） |
| issue body + comments 共享时间线 | 对话时间线 |
| 评论里 `@dev` 交棒 | 输入框 @ 补全 / 消息间交棒节点 |
| `eyes` reaction、运行黑盒 | 运行中活动块，实时可见、可中断 |
| 观察页只读旁观 | 观察与操作合一，此页即主界面 |

## 不能丢的四个本质（与 codex 的区别）
1. **多角色，不是单 assistant**：一群角色在同一条时间线接力，交棒是核心事件。
2. **账本是脊柱**：会话背后挂着目标 → 里程碑 → 任务状态机，有阶段、依赖、闸口。
3. **人是闸口**：「等待真人」是协议显式状态，人工验收决定阶段推进。
4. **证据链**：运行记录、验收 fact、产物。

## 核心推导

### 推导 1：会话 ≈ 账本任务的化身
旧世界 CEO 编排本来就是「每个账本任务开 child issue、目标有父 issue 承载采访 / 提案 / 集成验收」。映射「issue = 会话」下，会话天然分三种：**编排会话**（每目标一个，对应父 issue）、**任务会话**（对应 child issue，含修复子会话）、**自由会话**（不进账本的小事）。这不是新结构，是旧世界已验证结构的换壳。

### 推导 2：这是「它找我」的产品
codex 由人主导节奏；moebius 是 agent 网络自主接力、人是闸口。人打开页面的第一问是「有什么在等我、什么正在跑」，所以注意力信息前置（排序、顶栏计数），但不推翻 codex 式骨架。

### 推导 3：协议纪律 → UI 轨道
旧协议的硬格式（一条消息一个 mention、交棒行、stage marker、验收走查格式、提案确认 key）存在的唯一原因是界面是纯文本。界面自有后，纪律退化为控件：@ 补全且只允许一个角色、验收走查渲染成逐条勾选表单（提交时生成协议合规文本）、提案确认是按钮。协议未变，从「人要遵守的格式」变成「界面替人生成的格式」，消掉整类摩擦与解析失败。

### 推导 4：输出骨架 → 渐进披露
角色输出骨架（`## 结论`、`## 依据`、`## 下一步`）就是现成的结构化契约。agent 消息默认折叠为「角色 + stage + 结论 + 交棒行」，点开看全文；用户消息永远全文。治「一个任务会话十几条长评论翻不动」的信息淹没。

### 推导 5：运行过程从黑盒到直播
页面与 runner 同机，codex 输出流就在本地。运行中不是转圈图标，而是时间线里的活动消息块：哪个角色在跑、跑了多久、最近一行输出、中断按钮。中断从「发新评论触发中断检测」变成显式操作。

## 侧栏：收件箱语义，不是工作流语义
侧栏在人扫视的零点五秒里只回答一个问题：「要不要点进去」。谁在跑、处于什么阶段、会话什么类型是工作流语义，属于点进去之后（时间线、上下文面板）或账本视图。

- 会话行只有一行：状态图标 + 名称。
- 状态只有三态：**✋ 等你 / ⣾ 运行中 / 无（静止）**。
- **没有「未读」**：未读徽标是「要求人消费消息」的协作产品语义，本质是心理债务与阅读召唤。moebius 的成功标准恰是人读得越少系统跑得越好，UI 里不得存在任何催促阅读的元素。「等你」与「未读」本质不同：前者是阻塞（人不动线就停），后者只是新进展——后者被删除，前者保留。

### 排序：✋ > 目标 > ⣾ > 静止 > 已完成（折叠）
自上而下是「必须看 → 想看 → 可不看 → 没得看 → 不用看」，注意力义务单调递减：

1. **✋ 等你**：义务，永远最上。
2. **编排会话（目标）**：人的查收锚点、「我在让系统干哪几件事」的目录。运行中恰恰是最不需要人看的状态，凭什么排它前面——这一段是人类视角对工作流视角的胜利。
3. **⣾ 运行中**：车间在动，可不看。
4. **静止**：裸名。
5. **▸ 已完成**：任务会话验收通过自动沉入折叠组（任务会话是消耗品，不该永久占位）。

规则补充：
- **状态先于类型**：编排会话自己 ✋（提案等确认、集成验收）时升入 ✋ 段，回落归位。
- **排序键绑定低频状态迁移**，不绑最新活动：运行中的会话不因输出流动而移位，只在开跑 / 跑完 / 交棒到人时移动。列表动 = 有事发生，列表不动 = 天下太平——跳动从噪音变成信号。区内按进入该状态的时间，目标段按创建序。
- **冒泡规则：状态只从被折叠不可见处向上冒泡**。项目折叠时行尾聚合计数（`▸ tranfu-site ✋1`）；展开时子会话自己在列表里，编排会话与项目行不重复代言；已完成永不冒泡。

## 结构表达：侧栏管注意力，账本管结构，内容管关系
子会话派生是一棵真实的树（目标 → 任务 → 修复子会话，深度无界），但树不进侧栏——树状侧栏要求先理解结构才能找到东西，与注意力场景相拧，且叠加「项目 → 会话」后缩进层级失控。侧栏平铺，父子关系在三处表达：

1. 任务会话头部面包屑「↰ 属于：目标 · X」，点击回父会话；
2. 编排会话时间线里，派生动作本身就是一条可点击的事件消息（「已创建 3 个任务会话」）；
3. **账本视图**是唯一以树形展示全结构的地方（项目维度 goal → milestone → task，任务节点直通会话，闸口可见）。

这与旧世界一致：GitHub 的 child issue 也是列表平铺、靠 body 里 parent reference 链接成图。

## 查收点：编排会话 = 对话 + 事件流
「不放心、想去看看」是真需求，但它与未读有两点本质区别：方向相反（查收是人拉系统，不是系统推徽标召唤人）、粒度不同（人要的是「这件事到哪了」的目标粒度进展，不是消息计数）。设计替换：删未读徽标，把查收体验建在编排会话里——

- 时间线上半段是人与 CEO 的真实对话（采访、提案、确认）；
- 之后是**子会话里程碑事件自动回流的紧凑事件流**（任务创建、验收通过 / 失败、修复派生、运行起止），每条一行、可点击跳转子会话；
- 会话头常驻进展摘要（`任务 1✓ · 1⣾ · 1✋`）。

旧世界背书：验收 fact、join 等待、blocked 上报、集成验收本来就回流父 issue，「订阅父 issue 即掌握全局」已经成立，此处只是渲染得更紧凑。

终局状态：信任建立后，侧栏一片裸名、顶栏 ✋ 为 0——界面越安静，系统越健康，这才配得上「人类尽可能少介入」。

## 权衡（放弃了什么）
- **树状侧栏**：理解成本、宽度约束、深度无界、折叠状态管理，全让渡给账本视图。
- **未读徽标**：与少介入哲学正面冲突，删除而非弱化。
- **独立收件箱视图**：顶栏计数 + 侧栏排序两个入口够用，第一版不做，避免分散。
- **按最新活动排序**：多 agent 并发下秒级洗牌，换成状态迁移排序。
- **侧栏里的工作流细节**（「dev 运行中」文字、两行条目、「── 编排 ──」分组）：预支了点进去才需要的信息，全部退回图标 / 悬停 / 会话内。
- **编排会话类型图标**：数量少、命名自带辨识度（`目标：X`），先不加，实际找不到再补极轻前缀。

## 风险
- 本 change 纯设计：与 runner 的联动（消息写入通道、触发、中断传导、数据读取、进程边界）未设计。实施前必须先补对接设计——部分交互（中断按钮、实时输出流、秒级消息进时间线）依赖 runner 侧能力，存在倒逼调整线框的可能。
- 自由会话（不进账本）的运行时形态（持久化位置、与 role thread 的关系）未定义，属对接设计范围。
- GitHub issue 总线是否与本地操作台并存过渡、还是直接退役，本次不裁决。
- 回滚：设计文档废弃即可，零代码影响。
