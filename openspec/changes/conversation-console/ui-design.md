# UI 设计风格：conversation-console

线框与信息架构见 [wireframes.md](wireframes.md)，交互推导见 [design.md](design.md)；本文件只承载视觉风格与文案语气的决策及推导，供实现阶段作为样式层的事实源。

## 一、定位输入与已定决策

目标用户是**普通用户，不是程序员**。产品底层跑 codex / gh 是实现细节，不是产品气质；界面上不保留任何终端质感（等宽字体铺面、ANSI 式高饱和状态色、命令原文直接示人）。

与用户逐轮确认的决策：

| 决策点 | 结论 |
|---|---|
| 风格基调 | 安静工作台风，参照 Notion / Linear 一脉：低饱和中性底、细边框、靠间距与字重分层 |
| 明暗模式 | 双模式跟随系统，用同一套语义变量翻转 |
| 强调色 | 靛紫，全局唯一强调色 |
| 角色识别 | 统一中性首字母头像；头像是**可替换插槽**，后续另有替换方案 |
| 信息密度 | 舒适档（Notion 感），行高宽松、卡片有呼吸感 |
| 机器术语 | 全面转人话，原始术语与输出原文藏进展开层 |

### 1.1 竞品对照修订（第二轮追加决策）

对照同类 agent 操作台（Devin、Manus、Cursor、OpenAI Codex 网页版、Claude Code 网页版）复盘后发现：这一类产品已收敛出四条惯例——**评审时刻给结构化证据**（差异、产物预览）而非聊天原文；**计划步骤是时间线里的一等对象**（编号步骤逐条打勾推进）；**产物有专门的展示面**（右侧面板的主角是成果不是元数据）；**等待状态显性化**（输入框直接告诉人「现在说话会怎样」）。本设计首版只设计了注意力分配（何时看、看多少、什么颜色能响），没有设计证据结构（人做判断时看什么），低于行业基线，也没兑现 design.md 列为不能丢本质的「证据链」。由此追加决策：

| 决策点 | 结论 |
|---|---|
| 验收卡 | 证据先于裁决：卡内自带「改了什么 / 谁怎么验过 / 在哪亲眼看」 |
| 运行块 | 计划步骤清单逐条推进，每步可下钻原始输出；无步骤数据降级单行概括 |
| 右侧面板 | 产物预览置顶做主体，管理字段沉底；产物为空整区收起 |
| 顶栏「等你」计数 | 可点，展开跨项目清单浮层直达会话；「运行中」计数不可点 |
| 输入框 | 常驻等待状态文案，陈述发消息的后果 |

## 二、风格总纲：安静是功能，不是审美偏好

design.md 已经写死：界面越安静系统越健康，不得存在任何催促阅读的元素。视觉层把这条哲学翻译成三条铁律：

1. **颜色出现即信号，但信号色只剩事实层**。健康状态下，整个页面接近单色：中性灰阶 + 少量靛紫交互控件；红色出现 = 有东西失败、危险或被中断。等你事项不用专属色相，靠置顶排序、顶栏计数、清单浮层与验收卡位置承载注意力。颜色永远不做装饰。
2. **一屏至多一个实心强调按钮**。其余按钮一律描边或幽灵样式。多个实心按钮抢注意力，与「注意力义务单调递减」的排序哲学相拧。
3. **动效只服务状态感知**。全页面唯一的持续动效是运行块的呼吸指示；列表条目只在状态迁移时以短动画移位（列表动 = 有事发生）；禁止装饰性动画、禁止任何形式的角标脉冲。
4. **证据先于裁决**。凡要求人做判断的组件（验收卡、提案卡），必须自带判断所需的证据——改了什么、谁怎么验证过、在哪能亲眼看到——不得让人回读时间线取证。「少介入」的前提是想查证时查得到：信任靠可查证建立，不靠安静建立。

## 三、色彩系统

### 3.1 层次结构

色彩分三层，权责互不越界：

| 层 | 用途 | 色相 | 约束 |
|---|---|---|---|
| 中性层 | 背景、卡片、边框、正文、等你状态 | 灰阶 | 页面主体，占比 ≥ 90%；等你不占色相 |
| 强调层 | 主按钮、选中态、链接、焦点环 | 靛紫 | 交互专用，不表达状态 |
| 事实层 | 验收通过 / 失败、运行错误、中断 | 绿 / 红 | 只标注已发生的事实；红是唯一危险信号色 |

角色头像游离在这三层之外：统一中性灰底 + 中性字，只出现在头像圆圈内，不得渗透到按钮、文本、边框——保证「颜色出现即信号」不被身份色污染。

### 3.2 指示值（实施可微调，语义不可变）

浅色态：

- 页面底 `#F7F7F8`，侧栏底 `#FBFBFC`，卡片底 `#FFFFFF`，凹陷底 `#F3F3F4`，输入控件底 `#FFFFFF`
- 正文 `#1B1B1B`，次要文字 `#5D5D5F`，提示文字 `#8A8F98`
- 边框细线 `rgba(15,17,20,0.09)`，强边框 `rgba(15,17,20,0.14)`，选中态 `#ECECED`
- 靛紫强调 `#5E6AD2`，强调前景 `#FFFFFF`
- 事实绿 `#2F8F5B`，事实红 `#E5484D`（Linear 红）

事实绿 / 红只用在文字、图标或危险按钮描边上，不使用浅色填充块。深色态按同一语义翻转：底色落在近黑灰，文字翻为浅灰阶，靛紫与事实色提亮一档保证对比度。所有颜色一律通过语义变量引用（如 `--canvas`、`--card`、`--ink`、`--accent`、`--danger`），组件代码不得出现裸色值。

### 3.3 角色头像（首版）

| 角色 | 界面名 | 头像底 / 字 |
|---|---|---|
| ceo | CEO | 中性灰 |
| dev | 开发 | 中性灰 |
| qa | 测试 | 中性灰 |
| dev-manager | 技术负责人 | 中性灰 |
| product-manager | 产品 | 中性灰 |
| hermes-user | 用户代表 | 中性灰 |
| 真人用户 | 你 | 中性灰 |

头像是插槽：首版渲染「中性圆 + 首字」，后续替换方案（形象、图片等）只换插槽内容，中性圆保留作降级。

## 四、字体与排版

- **全界面使用系统中文字体栈**（PingFang SC / Microsoft YaHei / 系统默认），不引入外部品牌字体，不使用等宽字体做界面元素。
- 唯一例外：展开层里的**命令输出原文、日志原文**按等宽排版——那是内容的本来面目，不是界面风格；折叠态的概括行（「正在运行测试」）仍是普通字体人话。
- 字号层级（舒适档）：页面标题 18、卡片标题 / 会话名 15、正文 14、辅助信息 12，最小不低于 12。字重只用两档：常规与中黑（500），不用粗黑。
- 行高 1.6 上下；消息卡内边距宽松（约 12×16），侧栏条目单行但行高 36 左右，保证普通用户的点击容错。

## 五、形状、层级与图标

- 圆角：控件 8，卡片 12。不做直角，也不做胶囊化滥用（胶囊只给计数徽记与状态签）。
- 边框：0.5–1px 细线为主，靠底色差与间距分区，不靠粗框线。
- 阴影：常态卡片零阴影或极轻;只有浮层（菜单、@ 补全面板、抽屉）用一层柔和投影，同屏浮层至多一层。
- 图标：线性描边风格单一图标库，粗细统一;不混用实心图标，不用表情符号做功能图标（线框里的 ✋ / ⣾ 在实现层替换为同语义的描边图标与呼吸动效点）。

## 六、关键组件的风格落点

对照 wireframes.md 逐个锚定：

- **侧栏会话行**：单行、无卡片感，悬停淡灰底，选中淡中性底。「等你」条目：中性 hand 图标 + 常规文字（不整行染色——行级染色会让多条「等你」把侧栏变成警报墙）。运行中条目：中性色呼吸点，不占色相。
- **顶栏计数**：「N 等你」是中性文字 / 普通胶囊，可点开清单；「N 运行中」是纯文字。永远不用红点。
- **用户消息**：白底卡片、全文显示，视觉分量略高于 agent 消息（人是闸口，人说的话是锚点）。
- **agent 折叠消息**：无卡片、缩进文本行——「角色头像 + 中文角色名 + 人话 stage + 时间 + 结论一行 + 交给谁」。展开后进卡片，原文与原始术语在卡片内。折叠态的低视觉分量本身就在说「这些你可以不读」。
- **运行块**：白底卡片 + 中性呼吸指示 + **计划步骤清单**（✓ 已完成 / 呼吸点进行中 / ○ 未开始，进行中步骤配人话概括「正在运行测试 · 已进行 3 分 12 秒」）。中断按钮是红色描边幽灵按钮——事实层颜色只在按钮上，不染卡片。点开任一步看该步原始输出（等宽排版的内容层）；拿不到步骤数据时降级为单行人话概括，宁可单行也不得空白。
- **右侧上下文面板**：产物预览置顶做主体（缩略图 / 文件卡，点开大预览），验收语句、运行记录次之，任务号 / 里程碑 / 阶段等管理字段沉底——普通用户关心成果，不关心编号。产物为空时整区收起，不放「（暂无）」占位。
- **交棒节点**：消息间的细分隔线 + 居中小字「开发 → 测试」，是标点不是事件卡。
- **「轮到你了」验收卡**：普通中性白卡 + 中性 hand 图标，靠出现在时间线底部、侧栏置顶、顶栏计数三件结构手段表达义务。卡头先给证据三要素——「改了什么」（人话摘要）、「已自测」（agent 走查记录入口）；每条验收语句下挂证据行（关联产物直达 + 自测结果），产物与右侧面板指向同一对象，无证据可挂时该行降级为裸语句。逐条通过 / 不通过是成对描边按钮，选中后分别只用事实绿 / 红文字；不使用浅色填充块。「提交验收结果」是靛紫实心按钮。
- **顶栏「等你」清单浮层**：点「N 等你」计数展开跨项目清单，一行 = 中性 hand 图标 + 会话名 + 一句义务描述 + 项目名，点行直达会话。它是跳转菜单不是收件箱页；「N 运行中」纯文字不可点——义务给出口，车间不给召唤。
- **输入框状态行**：输入框下方常驻一行提示文字（提示文字色），陈述当前发消息的后果（「发消息不会打断运行」「等你验收，也可以先回复别的」），消除「说话会不会打断它」的顾虑。
- **CEO 提案卡**：白卡 + 靛紫描边确认按钮；「确认提案」可用靛紫实心（它是推进型操作，不是阻塞型义务）。
- **@ 补全面板**：浮层列出「头像 + 中文角色名 + 一句职责」，选中生成协议合法 mention；用户界面上看到的是中文名，协议层句柄由控件代管。
- **账本树**：缩进树 + 状态签（✓ 绿字 / 呼吸点 / 中性等你签），节点行悬停出现「打开会话」链接按钮。
- **空态**：一句邀请式文案 + 输入框，不放插画堆砌;语气按 8.2 节。

## 七、术语人话化对照表

界面层展示人话，协议层保持原文；对照表是实现时的映射事实源：

| 协议原文 | 界面展示 |
|---|---|
| dev / qa / dev-manager / product-manager / hermes-user / ceo / secretary | 开发 / 测试 / 技术负责人 / 产品 / 用户代表 / CEO / 秘书 |
| plan-written | 方案已写好 |
| code-verified | 代码已验证 |
| in-progress | 进行中 |
| 运行输出尾行（如 `running pnpm test`） | 概括为「正在运行测试」等人话短语，原文进展开层 |
| 交棒 → @qa | 交给「测试」审查 |
| 等待真人：… | 等你：…（中性呈现，靠排序、计数、浮层承载） |
| 验收走查 `N. 通过 — 依据` | 逐条勾选控件（每条挂证据行），提交时生成协议文本 |
| 运行步骤事件 | 步骤清单人话短语（✓ / 呼吸点 / ○），每步原文进展开层 |

概括短语生成不到位时降级显示原文，宁可露术语也不得空白或误译。

## 八、语气与文案

1. 句子说人话、动词开头、不用感叹号：「已交给测试审查」而不是「交棒成功！」。
2. 错误先说发生了什么、再说能做什么：「运行被中断。可以重新发消息让开发继续。」
3. 永远不催促：文案里不得出现「快去看看」「你有 N 条未读」这类召唤;「等你」的措辞是陈述义务（「等你验收」），不是营销推送。
4. 空态是邀请不是道歉：「描述你的目标，@ 一个角色开始」，不写「这里空空如也」。
5. 等待状态要显性陈述：界面主动说清「现在发消息会怎样」，不让人猜（「发消息不会打断运行」）；「谁在等谁」永远是一句可读的陈述句。

## 九、设计走查清单（哲学的可验证断言）

实现或改版后逐条走查，全部应当成立：

1. 打开一个没有任何「等你」事项的项目 → 整页除头像圆与靛紫交互控件外无彩色，无任何角标、红点、脉冲动画。
2. 任意一屏截图 → 实心强调按钮数量 ≤ 1。
3. 全局搜索界面文案 → 无「未读」字样，无催促式措辞。
4. 出现一条「等你」 → 该会话置顶、顶栏等你计数可点、会话内出现验收卡或义务说明；等你相关元素仍为中性呈现，不引入专属色相。
5. agent 消息折叠态 → 不出现 stage 英文原文与命令原文;展开后可以找到原文。
6. 运行中的会话在侧栏 → 输出流动十分钟，条目位置不变;运行结束或交棒到人 → 条目以一次短动画移位。
7. 切换系统深浅模式 → 页面即时翻转，无裸色值残留（对比度抽查正文、等你签与危险按钮）。
8. 用鼠标走完「验收三条 + 提交」 → 全程零手写格式，提交产物为协议合规文本。
9. 打开任一验收卡 → 不回读时间线即可完成全部走查：改动摘要、自测记录、逐条证据入口都在卡内；无证据的条目是裸语句而不是编造的证据。
10. 展开任一运行块 → 可见步骤推进与每步原始输出；拿不到步骤数据时是单行人话概括，不是空白卡。
11. 会话有产物 → 右侧面板顶部有预览，验收卡证据行与它指向同一对象；无产物 → 该区整体消失，无占位文案。
12. 点顶栏「等你」计数 → 跨项目清单浮层，点行直达对应会话；点「运行中」计数 → 无响应（不可点）。
13. 任一有运行 / 有等待的会话 → 输入框下方有等待状态文案，且内容与实际状态一致。

## 十、待定与后续

- 头像替换方案：用户明确后续另出，本文件只保证插槽边界（圆形、尺寸两档、中性头像降级）。
- 「在我电脑上重跑验收命令」入口依赖与 runner 的对接设计（design.md 风险节），本版验收卡只保留证据行位置，不承诺重跑能力。
- 运行步骤数据的来源与粒度（计划事件、工具调用事件如何映射为人话步骤）属对接设计范围；界面层只约定三态渲染与降级规则。
- 深色态的靛紫 / 事实色具体提亮值：进实现时以对比度检测定稿。
- Electron 窗口本体（标题栏、系统边框）与页面风格的融合，属桌面壳范畴，另行处理。
- 本文件在归档时随 wireframes.md 一并回流 `docs/wireframes/`（或并入未来的设计规范目录），由归档流程裁决。
