# 设计：agent-teams-implementation

## 方案

### 执行编排

15 个任务分四层推进，层内可并行：

```
T1 数据模型与磁盘布局
 ├─ T2 内置种子打包与指纹覆盖播种
 └─ T3 IPC 通道 + 数据容器 + 加载态
      ├─ T4 团队首页横行陈列
      └─ T5 团队详情骨架 + 成员选择器 + AGENT.md 编辑区
           ├─ T6 主 Agent 选择器与切换 ──┬─ T7 新建团队草稿 + 添加 Agent
           │                              └─ T11 复制 / 删除 Agent 与删除团队
           ├─ T8 内置团队只读 + 复制并编辑 ─┘
           ├─ T9 @slug 提及与自动补全
           ├─ T10 Finder / 平台文件管理器入口
           └─ T12 外部修改冲突
T4 + T5 ─→ T13 需要修复态与修复入口 ─→ T14 跨页面异常反馈 + 会话侧约束
T7 + T13 ─→ T15 新建对话团队预选
```

T1 是全局前置，T3 之后并行度最高（T4/T5 两条主线），T5 之后可同时开五个分支。

### 关键决策

**1. 内置团队走指纹覆盖，与 `agents/` 的「存在即跳过」并存**

`buildSeedCopyPlan` 的跳过语义保留不动（`agents/` 与 `config.toml` 依赖它保护用户修改）。内置团队新增一条独立路径：把 `seed/teams` 的内容指纹写入 `<dataRoot>/teams/.system/.teams-seed.marker`，启动时比对，不匹配则清空并重解包 `.system/`，匹配则整个跳过。用户区 `<dataRoot>/teams/<user-team-id>/` 不在这条路径的作用范围内。

判据是「谁拥有写权」：用户可写的内容永不覆盖，产品独占的内容随版本更新。两条规则相反，所以必须是两条路径，而不是给 `buildSeedCopyPlan` 加参数。

**2. `.system` 而非 `.builtin`，对齐 codex 命名**

codex 实测布局为 `~/.codex/skills/.system/`，内含 `.codex-system-skills.marker`。采用同一名称，让熟悉 codex 的用户在文件管理器里看到相同形状。点号前缀同时使内置区在常规列目录时不干扰用户团队。

**3. 内置团队解包到用户数据目录，而不是直读应用资源目录**

只读内容本可直接从 `.app/Contents/Resources/` 读取，省掉全部播种逻辑。选择解包是为了两点：PRD 要求内置团队提供「在 Finder 中打开」用于查看软件自带内容，指向应用包内部路径的体验差且 macOS 签名后不可写；内置团队与用户团队路径同构，读取、复制、有效性校验可以共用一套代码，只在写入口按归属拒绝。

**4. 显示名称与描述只存在于 `AGENT.md`，不维护第二份摘要**

PRD 第 202 行明确要求。团队首页横行、成员行、当前 Agent 标题都从 `AGENT.md` 解析，`team.json` 只存团队名称、一句话描述、主 Agent slug 与成员顺序，不缓存成员显示名。代价是首页需要读取全部成员文件；收益是不存在摘要漂移这一整类 bug。

**5. slug 是团队内唯一的稳定标识，与显示名称完全解耦**

创建时生成，此后不因改名而变化，产品不提供修改入口。`@slug` 提及在磁盘上始终是纯文本 `@<slug>`，编辑器只在渲染层显示为可读名称组件。这保证用户用外部编辑器打开 `AGENT.md` 看到的是稳定引用，而不是需要产品才能解析的私有格式。

## 权衡

**内置团队被新版本移除时不做迁移。** 整体覆盖 `.system/` 会让上一版存在、本版删除的内置团队直接消失。选择接受：引用它的「上一次使用」记录回退到第一支内置团队，已有会话的历史与创建时载入的版本不受影响（PRD 第 327 行已承诺）。替代方案是保留墓碑记录并提示用户，但这会为一个低频事件引入长期状态。

**不做每次启动的逐文件完整性校验。** 用户若手工修改 `.system/` 下的文件，在指纹变化前不会被还原。选择接受：只读由 UI 层入口保证，升级时被覆盖，PRD 第 263 行「外部修改不能绕过只读限制」在这个实现下依然成立。逐文件校验的启动开销与团队数量成正比，不划算。

**未保存草稿留在内存，不落临时文件。** PRD 要求多名成员可同时保留未保存草稿、切换成员不弹窗、离开时统一处置。全部用内存状态实现，应用崩溃会丢失草稿。选择接受：落盘草稿会引入「草稿文件也可能损坏」与「外部修改冲突要三方比对」两类新问题，复杂度远超收益。

**`@` 自动补全只匹配当前团队成员，不做跨团队。** PRD 非目标已排除跨团队共享 Agent。实现上不引入全局 Agent 索引。

## 风险

**播种时机与首屏竞态。** 指纹比对与重解包发生在主进程启动路径上，团队页首屏读取可能早于播种完成。缓解：播种完成前团队数据接口返回加载中状态，UI 走 PRD 已定义的「页面加载中」态；播种失败走「页面加载失败」态并保留重试，绝不把失败呈现为空列表。回滚：指纹比对失败时保留现有 `.system/` 内容继续可用，不清空。

**清空重解包的中断风险。** 若在清空 `.system/` 后、解包完成前进程被杀，下次启动会看到不完整的内置区。缓解：解包到临时目录后原子替换，而不是就地清空；指纹文件在替换成功后最后写入，写入前的任何中断都会导致下次启动重新执行完整流程。

**用户区与内置区的路径混淆。** 两者同构且相邻，写操作若未按归属校验，可能写进 `.system/`。缓解：所有写入口在数据层统一做归属判定，内置团队的任何写请求直接拒绝并返回明确错误，不依赖 UI 层禁用按钮作为唯一防线；T8 的验收必须包含绕过 UI 直接调用写接口的用例。

**T5 是五条分支的共同前置，阻塞面最大。** 其成员选择器与草稿状态契约一旦返工，T6/T9/T10/T12 全部受影响。缓解：T5 优先落定草稿状态与成员切换的对外契约，后续分支只消费不修改；loop 调度上不要在 T5 合并前抢跑其下游。
