## 背景

Agent 团队体系目前不是运行时实体，只是一个带文件管理的 `AGENT.md` 编辑器。用户选择团队这个动作，在存储、执行、界面三层都没有落点：

- **执行侧不认团队**：`src/local-console/server.ts` 的 `listAgentFiles` 默认落到 `<dataRoot>/agents/` 这个全局目录，且 `desktop/src/main.ts` 启动 local console server 时没有覆盖它。无论用户选哪支团队，运行时可用的 Agent 名单永远是全局那一套。
- **存储侧没有绑定**：`sessions` 表没有任何团队列，`src/local-console/` 全目录对 team 零命中。团队标识只写进 `last-used-team.json` 一条全局偏好。
- **界面侧张冠李戴**：`desktop/src/console-page/app.tsx` 把团队管理页的**浏览选择态**当作「当前会话的团队」传给操作台。用户在团队页点开一支需要修复的团队，所有会话的输入框都会被禁用；而会话真正依赖的团队损坏时，发送照常放行。

因此 PRD 中「已有会话继续使用创建时载入的团队版本」「团队当前是否需要修复实时决定该会话能否继续发送」这两条反复出现的前提，在实现中一条都不成立。相关验收此前被判为通过，是因为测试直接把 `selectedAgentTeamKey` 当 prop 灌入组件，绕过了真实接线。

同批暴露的另外三处缺陷与上述主干独立，但都会在修复过程中被触碰，一并收口：

- **团队记录的身份模型把标识和位置绑死**。`resolveTeamLocation` 按 `teamId` 拼路径，而重新定位只改记录里的 `directoryName`、`id` 保持不变。于是「在文件管理器中打开」和外部修改检测这两条链在团队被重新定位后必然失效，且失败被静默吞掉。记录只存 `directoryName`（basename），结构上无法表达 `teams/` 以外的位置，而 PRD 场景明写团队可能「被外部移动」。
- **主内容区缺少视图路由规则**。进入 Agent 团队页后，点击侧边栏会话、新建对话、搜索跳转都不会把主区带回对话视图——选中态移动了、数据也切了，但用户看到的仍是团队页。
- **团队详情页若干交互欠账**：外部冲突时返回按钮静默失效、返回列表时滚动位置恢复时机错误、`AGENT.md` 编辑器缺少输入法组字处理。

## 提案

按「团队 → 会话 → 执行」的顺序补齐绑定链路，并修掉环绕它的三处缺陷。四块的实施顺序是 B → A → C → D，因为 A 需要一条可信的团队路径解析作为地基。

**A. 让团队成为运行时实体**
`sessions` 表记录会话绑定的团队标识；创建会话时一并写入；执行侧改为按会话解析可用 Agent 名单，无绑定的存量会话回退全局目录；界面上把「当前会话的团队」与「团队页浏览选择」拆成两个独立状态；团队健康度纳入既有轮询，实时决定能否发送。

**B. 让团队标识与磁盘位置解耦**
团队记录改为可表达受管目录与任意外部路径两种位置；所有用户团队的路径解析统一走记录，不再按 `teamId` 拼路径；外部修改检测的失败给出用户可见反馈。

**C. 建立视图路由规则**
把「任何把用户带向某段对话的动作都必须将主区拉回对话视图」抽成单一入口，四个出口统一走它，而不是逐个补状态复位。

**D. 补齐团队详情页交互**
外部冲突时返回给出明确阻止说明；滚动恢复等目标 DOM 就位后再执行；编辑器处理输入法组字；未保存草稿时从侧边栏切走触发既有三选项提示。

同时按产品裁决删除团队记录中缓存的成员摘要：不可用态的团队横行显示占位，不再回落到可能与磁盘不符的旧名单。

## 影响

**业务域**：`desktop-shell`（团队存储与路径解析）、`local-console`（会话绑定与执行侧名单解析）、`console-ui`（视图路由、需修复态传播、外部冲突、成员摘要）。

**对外行为变化**：

- 新建会话起，可 `@` 的 Agent 名单收窄为该会话绑定团队的成员；存量会话（无绑定）行为不变，仍使用全局目录。
- 团队页浏览行为不再影响任何会话的发送能力。
- 团队被重新定位后，「在文件管理器中打开」与外部修改检测恢复可用。
- 需要修复的团队横行不再显示成员名称。

**数据迁移**：`sessions` 表加列（可空，存量为 NULL）；团队记录文档 v1 → v2（`directoryName` 读为受管位置）。两处都向后兼容，不需要用户干预。
