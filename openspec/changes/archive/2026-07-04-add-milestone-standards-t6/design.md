# 设计：add-milestone-standards-t6

## 方案

### `docs/roadmap/milestone-standards.md`

文档结构采用可直接执行的标准说明，而不是流程宣言：

1. 开头说明文档用途：给人和未来编排者在拆里程碑时判断是否形成可验收垂直切片。
2. 写四条原则，每条都包含：
   - 标准定义。
   - 正例。
   - 反例。
   - 使用时的检查问题。
3. 写质量基准三档表：
   - `demo 级`：验证交互通路或叙事可信度，可用假数据 / 人工流程，但必须标清假设和不可承诺的能力。
   - `数据正确级`：核心状态、计算和持久化必须对真实 happy path 正确，可接受非核心体验粗糙。
   - `成品级`：面向真实使用的可靠性、异常路径、安全性、可观测性和体验完整度都进入验收。
   每档都写定义、适用场景、验收要求和反例；补充规则：新增档位必须显式写入里程碑文档。
4. 写抽象支付产品 worked example：
   - M0：假账扫码支付通路 demo。
   - M1：单用户真实 happy path。
   - M2：账户安全与异常路径。
   - M3：商户侧。
   每级明确“为什么这是垂直切片”、质量基准和验收方式。

worked example 只表达拆解方法，不写真实支付宝架构、牌照、清结算、资金合规或线上承诺。

### `docs/roadmap/milestone-task-issue-template.md`

做最小泛化：

- 标题从“里程碑 1”改为适用于任意里程碑。
- 占位符说明补充 `<里程碑文档>` / `<里程碑名>` 一类表达，消除 `docs/roadmap/milestone-1-acceptance-loop.md` 的硬编码。
- 在“使用规则”靠前位置加入：开 issue 和写验收语句前先参考 `docs/roadmap/milestone-standards.md`，确保任务是垂直切片、质量基准显式、验收语句可机械执行。
- 保留现有协作流程、边界说明和触发目标规则，不改成编排系统设计。

## 权衡

- 不把标准拆到多个文档：T6 的目标是提供一把复用决策尺子，集中在一个 roadmap 标准文档更利于人和未来编排者引用。
- 不新增质量顶层档位：product-manager 已确认固定三档；额外维度只作为每档内部说明。
- 不把模板重写成通用编排器协议：本任务只要求最小泛化与标准引用，重写协作流程会扩大范围。
- 不创建新的 OpenSpec spec domain：当前 `openspec/specs/` 只有 `github-issue-runner` 运行域，本次不改变运行时行为。把 roadmap 标准再复制到 spec 会制造双事实源。

## 风险

- 风险：标准文档如果只写原则，仍可能被执行者理解成抽象口号。
  - 缓解：每条原则必须有正例、反例和检查问题，并用 worked example 展示拆解落点。
- 风险：抽象支付产品示例容易被误解为真实支付宝方案。
  - 缓解：文档明确这是抽象支付产品示例，并排除真实支付宝架构、牌照、清结算或合规承诺。
- 风险：模板泛化时误改协作流程。
  - 缓解：只替换硬编码来源和标题口径，保留现有流程条目。
- 回滚：删除新增标准文档，并把模板中标准引用和泛化措辞恢复到原文即可；不涉及运行时数据迁移。
