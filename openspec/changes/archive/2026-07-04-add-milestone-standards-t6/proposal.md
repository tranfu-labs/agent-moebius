# 提案：add-milestone-standards-t6

## 背景

Milestone 2 的 T6 要交付一份里程碑设定标准，作为人类和未来编排者共用的决策尺子。当前 roadmap 中已经有各里程碑和任务模板，但缺少一份集中说明：里程碑应如何避免水平分层、隐含质量标准和不可机械验收的模糊任务。

这些缺口会增加需求方和执行方的上下文税，并让交付结果难以追责：任务看似拆完，实际每层都无法端到端验收；质量基准靠执行方猜；验收语句无法机械执行。

## 提案

新增 `docs/roadmap/milestone-standards.md`，作为里程碑拆解与任务粒度判断的标准文档。文档至少包含：

- 垂直切片原则：里程碑必须代表一次能力解锁或最大风险消除，并以可被角色端到端验收的切片收尾；每条原则给出正例和反例。
- 风险优先排序：最不确定的风险先通过 spike 消除，并引用 M1 T4 作为范例；每条原则给出正例和反例。
- 显式质量基准：固定三档 `demo 级`、`数据正确级`、`成品级`，每档写定义、适用场景、验收要求和反例；未来新增档位必须显式写入里程碑文档。
- 任务粒度标准：一个任务等于一组可机械执行的验收语句，并与兄弟任务保持最小文件重叠；每条原则给出正例和反例。
- worked example：用抽象支付产品示例说明 M0 到 M3 的里程碑拆解，每级都标注为什么是垂直切片、质量基准和验收方式；不涉及真实支付宝架构、牌照、清结算或合规承诺。

轻量泛化 `docs/roadmap/milestone-task-issue-template.md`：只消除里程碑 1 的硬编码，并在开任务 / 写验收前能被执行者看到的位置加入对标准文档的引用；不重写模板结构，不改协作流程，不扩展成编排系统设计。

## 影响

- 受影响文件：
  - `docs/roadmap/milestone-standards.md`
  - `docs/roadmap/milestone-task-issue-template.md`
- 不改运行时代码、不改 `agents/`、不改 GitHub issue runner 行为。
- 不新增 `openspec/specs/` 行为域：本次交付的事实源就是 roadmap 标准文档本身，创建额外 spec 会让同一标准出现双事实源。
