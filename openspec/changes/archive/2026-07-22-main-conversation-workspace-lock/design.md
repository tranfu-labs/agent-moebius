# 设计：main-conversation-workspace-lock

## 这是一次拆除，不是一次新增

本片的全部风险都在「拆干净」上。B 片为运行中切换工作空间铺了一条完整链路：UI 下拉 → 确认弹层 → 桌面乐观态 → HTTP → `workspace_pending_mode` 落库 → 步骤收尾时提升。锁定之后这条链路整条不可达。

**拆一半比不拆更危险**：留着后端待生效提升而只关掉 UI，等于留一条无人经过但仍在维护成本里的分支，以及一份仍写着「MUST 允许改选工作空间」的 spec——下一个实现者读 spec 会把它重新做出来。这正是本片存在的原因（PRD 改了、实现没跟、也没人知道），不能在同一个地方重犯。

## 锁定判据取「是否已有消息」，不取「会话是否存在」

PRD 的措辞是「发出第一条消息后」，而经过 A 片之后会话本来就是首条消息的副产品——两者在正常路径上同时发生。但它们在两种情况下会分叉：

- 子会话由编排创建，其首条消息由系统写入；
- 创建失败回滚的中间态。

统一取「该会话已有消息」作为判据，与项目锁定当前使用的条件（`operator-console.tsx:914` 的 `messages.length === 0`）保持同源，避免两项上下文用两套判据解锁而在边缘情况下不同步。后端以会话消息数为准，不信任前端传入的标志。

## 后端拒绝而不是静默忽略

`switchLocalSessionWorkspace` 在会话已有消息时返回明确错误，而不是静默丢弃。理由是这条命令在锁定之后只可能由陈旧客户端或将来的误接线触发，静默成功会让调用方以为切换生效、界面与实际 cwd 从此不一致——这正是「产品无法回答改了什么」的那类故障。错误文案走用户可理解的措辞，不暴露列名或内部标识。

## `workspace_pending_mode` 列保留但停写

不做删列迁移。已有库里该列可能残留非空值（B 片期间产生的待生效切换），删列需要一次不可逆迁移，收益只是省一列。改为：

- 写路径拆除，任何新的非空值不再产生；
- 读路径把残留值视作无效并按生效值解析，使存量库不会在升级后突然切换工作空间；
- 步骤收尾的提升逻辑只处理团队。

代价是 schema 上留一个死列，接受——它有 `Source` 可查，且比一次为清理而做的迁移安全。

## 独立工作空间的说明移到选择处

弹层删掉之后，「副本基于项目当前所在的提交，不包含你还没提交的改动」这句话不能一起消失——它是 PRD 明确要求必须说清的后果，只是发生的位置从「中途切换的确认弹层」变成「新对话页选择工作空间时」。同时删掉「此前已经在项目文件夹里产生的改动也不会被搬过去」：对话还没开始，不存在「此前的改动」，留着反而制造一个用户理解不了的前提。

注意当前新对话页只显示 `workspaceLabel` 文本、并不提供工作空间选择（`new-conversation-page.tsx:102`）。本片补上选择能力的同时补上说明，否则「只在发出第一条消息之前可选」这条会变成「哪里都不可选」。

## 测试改写而非新增

B 片的 `tests/local-console-pending-switch.test.ts` 有一组工作空间待生效用例，它们断言的正是本片要拆掉的行为。改写这些用例，不保留「旧行为仍可用」的兼容分支——两套语义并存是这次返工的根因之一。团队待生效的用例原样保留，它们是本片必须守住的回归护栏。

## 验收方式

- 纯逻辑走 Vitest：锁定判据、后端拒绝、存量残留待生效值的降级解析、同项目两会话模式互不影响的回归。
- 页面表现走真实桌面窗口（CDP `9222`）：新对话页可选并有后果说明；发出第一条消息后第二格变为不可点击文本；产品内找不到任何切回默认的入口。
