# 设计：restore-last-console-selection

## 方案

1. 新建 `selection-preference.ts`，以 renderer 已使用的本地存储保存 `{ projectId, sessionId }`，集中负责 schema 校验、安全读写、清除、相等判断与启动恢复判定。
2. `App` 首次挂载时读取偏好；没有偏好时只把 `local/default` 当作拉取项目列表所需的兼容请求，不把返回的兼容选择写回偏好。
3. 首次状态响应必须同时满足：项目 id 相同、会话 id 相同、返回会话是用户根对话，才结束 bootstrap 并恢复会话。任一条件不满足时清除偏好，打开未选择项目的新建对话。
4. 后续只在状态请求真正提交且返回根会话时去重写入偏好。选择请求失败、被新 mutation 抢占或只展开子会话时不写。
5. 创建会话成功后显式记住新会话；移除当前项目后清除偏好；归档或重绑后的有效相邻根会话由成功状态提交统一记住。

## 测试设计

### 单元测试

- 合法记录可读取并匹配同一根会话。
- 缺失、非 JSON、字段缺失、空字符串和存储异常全部返回无偏好。
- 项目不匹配、会话不匹配或返回子会话时拒绝恢复。
- 写入、清除和底层存储异常不打断操作台。
- 慢轮询、mutation 抢占和失败请求不能提交或推进选择；只有成功响应可成为记忆候选。

### AI 验证流程

1. 启动桌面应用，选择第二个项目中的非默认根会话，重启后确认恢复同一项目与会话。
2. 清除选择记录后重启，确认进入新建对话且项目未选择。
3. 写入不存在的项目/会话记录后重启，确认仍进入未选择项目的新建对话，侧栏第一个项目和最旧会话不被选中。
4. 打开父时间线中的子会话右栏后重启，确认恢复父根会话而不是子会话。

## 权衡

- 复用 renderer 本地存储而不是新增 SQLite 表或主进程偏好文件：选择和既有 Sidebar 可见性、未发送草稿同属界面恢复状态，改动边界最小；代价是本地存储被禁用时只能安全回退到新建对话。
- 不修改 local-console state API 的兼容回退：终端、本地测试和旧调用仍可使用 `local/default`；renderer 通过对比请求与返回识别回退，避免把兼容行为升级成产品默认。
- 不按“最近更新时间”推断目的地：PRD 要求恢复用户选择，而不是猜测最新活动；推断会让后台 Agent 输出改变启动位置。

## 风险

- 首次响应的 bootstrap 与周期刷新并发可能覆盖目的地；沿用现有 refresh lease，且 bootstrap 只允许首个可提交响应消费一次。
- 当前项目移除或最后会话归档后可能留下无效记录；对应成功路径必须清除，启动验证仍作为最终兜底。
- `app.tsx` 当前有其他未提交改动；实现只在选择初始化、状态提交和相关成功动作附近增量修改，不覆盖 Agent frontmatter 工作。

回滚时可移除选择偏好模块与 App 接线，并删除对应本地存储 key；本地会话、项目和消息数据不受影响。
