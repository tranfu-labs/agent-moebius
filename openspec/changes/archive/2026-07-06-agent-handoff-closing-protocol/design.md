# 设计：agent-handoff-closing-protocol

## 核心决策

### 1. 收尾行语法只定义一次，两侧逐字咬合

交棒行的合法形式在所有文件中使用同一段定义（角色 persona 是"产出方"，ceo.md 是"核查方"，两侧措辞必须一致，否则核查方会漏判）：

```text
交棒：@<合法角色> <请其做什么>
```

或

```text
等待真人：<等什么、请谁做什么>
```

约束：

- 位置固定：位于 `## 下一步` 节内、stage marker 之前；二选一、恰好一条。
- 一条评论至多一个合法 agent mention（沿用现有 v0 协议），mention 只出现在交棒行里。
- `等待真人` 行不得含任何合法 agent mention（否则会被 mention trigger 误触发）。
- 采访提问轮是"等待真人"的特例（`等待真人：回答上述采访问题`）。

### 2. CEO「第 0 裁决」置于一切业务场景之前

审查顺序改为：先裁决交棒完整性，再做既有的格式纠偏 / 验收治理 / 阶段回流等场景。规则表：

| latestResponse 状态 | CEO 动作 |
|---|---|
| 含合法交棒行（@角色） | 进入后续场景判断；若无其他违规 → `no_change` |
| 含合法等待真人行 | 同上，不重复催办 |
| 两者皆无 | `no_change` 非法；必须 `append` 路由：能套现有剧本（plan-written→@qa、code-verified/QA通过→发起角色）就套；发起者是真人 → append 裸写请真人按验收清单逐条验收 |
| 栏位存在但内容空泛（如"下一步：待定"） | 与缺失同等对待 |

「code-verified：识别发起需求角色」第 3 条与「qa 交棒兜底」的真人分支同步收窄：真人等待不再允许静默 `no_change`，除非最新评论已含等待真人行。

### 3. 统一输出骨架：三个通用节 + 角色专属节收编

```markdown
## 结论
<一句话先行：本轮做成了什么 / 判断是什么>

## 依据
<证据引用：文件路径、命令退出码、截图、评论位置——空泛视为缺失>

<角色专属必填节，收编现有约定，不新发明：
  dev@plan-written：「验收语句」；dev@code-verified：「验收证据」
  qa：固定六项审查 +「QA 结论：」行
  product-manager / hermes-user：逐条验收（每条 通过/不通过 + 依据）>

## 下一步
<交棒行，语法见上，恰一条>

<!-- moebius:stage=... -->
```

权衡：

- **为什么是骨架而不是单条规则**：issue #10 时间线上质量最高的输出恰是被模板锁住的部分（QA 六项审查逼出 D1–D3 真缺陷），断掉的恰是唯一自由发挥区（交棒）。栏位标题即强制回答的问题；栏位存在性是纯文本匹配，CEO 核查零语义成本。
- **防填空疲劳**：骨架只留三个通用节；角色专属节全部收编现有约定、不新增；"空泛=缺失"写进 CEO 裁决。
- **模板不靠记忆**：模板全文在 agents/*.md 内，runner 每轮组装提示词自然重发——与"词表每轮重发"同一原则，不受长会话漂移影响。

### 4. 为什么只动提示词层

- 最快见效：runner 每轮重读 persona，改完即生效，不需要发版或重启。
- 机械兜底（runner 停滞唤醒，`activeIssueNoChangeCount` 从"降级冷落"改"先唤醒再降级"）与治本（流程看板：runner 持权威看板、agent 输出增量、事实验证器）留作后续 change，本次不混入。

### 5. scope-locked 修复

product-manager.md 阶段 C 的收尾 marker `stage=scope-locked` 不在 `ALL_STAGES` 枚举内，运行时被静默解析为 `unknown`。改为 `in-progress`（语义：停下等人类确认，等价于"等待下一步指示"），私有阶段的正式表达留给看板机制。

## 风险

- CEO 守护是 LLM，第 0 裁决仍可能漏判——本 change 只把"漏判概率"从语义理解降到文本匹配级别，不宣称消除；机械保底在后续 runner change。
- 骨架对存量 issue 线程是新约定，旧线程里 agent resume 时可能混用旧格式一轮；CEO 第 0 裁决会兜住（无交棒行 → append 路由），可接受。
