# 线框：main-conversation-session-context

基线：`docs/product/pages/main-conversation.md#页面结构`（上下文条部分）与 `#区域与信息`。该页面 PRD 已接管主内容区版式，本文件只记录本次 change 相对基线**新增或改动**的部分；归档时回流到该页面 PRD，不写入 `docs/wireframes/`。

## pages/main-conversation.md

### 上下文条常态（沿用基线，分支改为真名）

```text
│   ┌──────────────────────────────────────────────────────┐  │
│   │ ▱ 项目 A  [默认工作空间 ⇄]  main  [◇ 开发团队 ▾]     │  │
│   │ 说点什么，或 @ 一个成员…                          ↑ │  │
│   └──────────────────────────────────────────────────────┘  │
```

第三格是真实分支名（`main`、`feat/x`、detached 时显示兜底值），不可点击。基线里写的「当前分支 / 会话分支」这类字面词被移除。

### 工作空间下拉

```text
   ┌──────────────────────────────────────┐
   │ ✓ 默认工作空间                       │
   │   直接改项目文件夹里的文件           │
   │ ──────────────────────────────────── │
   │   独立工作空间                       │
   │   把改动隔离在一份副本里             │
   └──────────────────────────────────────┘
```

项目文件夹不是 git 仓库时：

```text
   ┌──────────────────────────────────────┐
   │ ✓ 默认工作空间                       │
   │   直接改项目文件夹里的文件           │
   │ ──────────────────────────────────── │
   │   独立工作空间                  （灰）│
   │   这个项目文件夹不是 git 仓库，       │
   │   无法隔离改动                       │
   └──────────────────────────────────────┘
```

原因写在被禁用的项下面，不做成单独的提示气泡——用户是在这里做决定的，解释就该在决定的位置。

### 切到独立工作空间时的说明

```text
   ┌──────────────────────────────────────────────┐
   │ 换成独立工作空间                             │
   │                                              │
   │ 副本基于项目当前所在的提交，不包含你还没     │
   │ 提交的改动；此前已经在项目文件夹里产生的     │
   │ 改动也不会被搬过去。                         │
   │                                              │
   │                      [取消]  [换过去]        │
   └──────────────────────────────────────────────┘
```

两件事都要说：基于哪个提交、已有改动不搬走。PRD 点名了「界面必须把这两件事都说清楚」。

### 切回默认工作空间时的说明

```text
   ┌──────────────────────────────────────────────┐
   │ 换回默认工作空间                             │
   │                                              │
   │ 之后的改动会直接落在项目文件夹里。           │
   │                                              │
   │                      [取消]  [换回去]        │
   └──────────────────────────────────────────────┘
```

### 团队下拉

```text
   ┌──────────────────────────────────────┐
   │ ✓ 开发团队                           │
   │   营销团队                           │
   │ ──────────────────────────────────── │
   │ 这段对话用的是开始时载入的那份团队    │
   │ 内容，之后在团队页的修改不影响它      │
   └──────────────────────────────────────┘
```

说明放在下拉里而不是常驻在按钮上：常驻会占掉上下文条本就紧张的横向空间，而这个疑惑只在用户想动团队时才产生。

### 有成员正在工作时切换

```text
│   ┌──────────────────────────────────────────────────────┐  │
│   │ ▱ 项目 A  [默认工作空间 ⇄]  main  [◇ 营销团队 ▾]     │  │
│   │ 当前这一步跑完后换成营销团队                         │  │
│   │ 说点什么，或 @ 一个成员…                          ↑ │  │
│   └──────────────────────────────────────────────────────┘  │
```

按钮已显示新值，下面一行说明它何时真正生效。不写出来的话，用户会以为切换没生效而反复点。工作空间切换用同样的形式。
