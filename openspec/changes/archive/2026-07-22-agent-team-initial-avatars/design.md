# 设计：agent-team-initial-avatars

## 方案

新增一个无数据依赖的 `AgentInitialAvatar` 组件，输入 `displayName`、`slug` 与尺寸档位，内部调用纯函数推导单字符字形：

1. 对 `displayName` 做 `trim()`；非空时取首个 Unicode code point。
2. 名称为空时对 `slug` 做 `trim()` 并取首个 Unicode code point。
3. 拉丁字母使用大写，其他字符保留原样。
4. 两者都为空时使用中性的“协”作为最终防御性兜底。

组件只使用既有 `bg-ava-bg` / `text-ava-fg` 语义令牌，保持圆形、中性色与 `aria-hidden`。可读身份仍由相邻名称文本和现有按钮/标签的 accessible name 提供，避免屏幕阅读器重复朗读首字。

三处接入保持现有信息层级不变：

- 团队首页：头像进入现有成员胶囊左侧，主 Agent 标签与 `＋N` 逻辑不变。
- 成员选择器：头像进入每个 tab 左侧，选中、不可用、未保存和主 Agent 状态不变。
- 当前成员标题：头像与名称/描述组成一行身份块，成员操作菜单位置不变。

## 权衡

**从显示名推导，不新增元数据。** 头像只承担快速识别，不是独立身份事实；新增 `avatar` 会引入图片存储、复制、播种、路径安全与损坏降级等无关范围。

**共享组件而非三处复制。** 字形降级、尺寸和颜色语义只维护一份，避免团队首页与详情再次漂移。

**保留最终内部兜底。** 合法团队成员一定有 slug，但组件边界仍对空输入防御，避免测试 fixture 或异常中间态出现空白头像。

## 风险

- 复合 emoji 的首个 Unicode code point 可能不是完整 grapheme；当前身份名称以中英文文本为主，本次不引入 `Intl.Segmenter` 兼容分支。若未来允许 emoji 作为显示名首字符，再独立升级 grapheme 规则。
- 成员胶囊增加头像后横向占用略增；保持现有三成员上限与 `＋N` 收敛，并在窄布局测试中确认不破坏横向结构。
