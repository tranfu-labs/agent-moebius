# 设计：main-conversation-timeline-truth

## 方案

### 四种事实由数据层显式声明，界面不再猜

当前界面靠字符串匹配把系统消息映射成状态（`systemSummary` 读 `pending / running / completed / displayed`，`safeRunSummary` 用正则挡机器串）。这条路的问题不是不好看，是**不可判定**——runtime 换一句文案，界面就会静默降级。

改成数据层显式声明：`session_messages` 新增 `system_event_kind`，取值覆盖 PRD 点名的四种事实（没跑起来 / 卡住 / 用户按了停 / 反复重试仍未成功），加一个中性的「其他系统记录」。runtime 在写系统消息时同时写 kind；界面据此决定渲染成哪种表现、给不给「重试」，不再读正文。

这同时解决了「重启后四种事实仍可见」——kind 落库了，不依赖任何内存状态。

### 状态点判据换源

红蓝闪的派生集中到 `packages/console-ui/src/console/status-dot.ts`，判据全部换源：

| 点 | 新判据 | 旧判据（作废） |
| --- | --- | --- |
| 红 | 存在未处理的 `system_event_kind` ∈ {没跑起来, 卡住, 反复重试仍未成功}，**或**该会话处于三种不可继续状态之一 | `awaits_human_reason` 非空 |
| 蓝 | 没有成员在工作、最后一条消息未提及任何人、且用户尚未查看这些新结果 | `unread_since` 非空 |
| 闪 | 有成员在工作 | 不变 |

蓝点保留 `unread_since` 作为清除机制——`main-conversation.md` 给的是何时**出现**，`main-sidebar.md` 给的是何时**清除**（打开并成功展示最新结果后）。只按出现条件实现会让蓝点永不消失，因为那两个条件在用户打开对话后都不改变。

红点与三种不可继续状态的只读表现**不互斥，同时成立**：用户不在这段对话里的时候，红点是唯一能把他叫回来的东西；他回来之后看到的就是只读表现和对应的恢复入口。

优先级 `红 > 蓝 > 闪 > 无` 保持。三条必须显式实现的豁免与推论：

- **用户按了停不进红**：`用户按了停` 这个 kind 存在，但不在红点集合里。PRD 的理由是「那是用户自己刚做的动作，不需要产品回头提醒他」。
- **最后一条提及了某个成员时既不红也不蓝**：属于团队内部仍在传递。
- **正常完成不产生红点**：红点集合里只有异常，没有「完成」。

「等人回话」这条旧来源整条作废，不保留成第四种红点——PRD 把「成员主动声明我需要用户裁决」记入了待讨论，本片不得替它拍板。既有 `awaits_human_reason` 的值在迁移时映射：`exception` 归入「其他系统记录」并不触发红点，其余三种丢弃。

**红点必有可读的系统记录**：因为红点只由三种异常触发，而每种异常都对应一条带 kind 的系统消息，所以用户被红点带回来时页面上一定有说明发生了什么的记录。这是 PRD 点名要保证的性质，不是巧合——实现上不得引入任何不落系统消息就置红的路径。

### 机器信息在两处收口

- **数据层**：runtime 写的系统消息体改成用户语（当前是 `Skipped local run: ...`、`No valid agent mention found...`、`Local child session orchestration completed: <sessionIds>` 这类）。有了 `system_event_kind` 之后，正文只承担人话，机器细节不进正文。
- **界面层**：`machine-text.ts` 把过滤范围从 run summary 扩到所有渲染文本——agent 消息体、run step 标题、系统记录。agent 输出里的绝对路径是过滤的主要目标，因为它不由我们生成。

项目修复对话框里明文展示项目路径不在此列：那是用户自己选的文件夹、且他正需要凭它确认选对了。PRD 第 13 条约束的是「界面文案」，不是用户自己提供的路径回显。

### 三种不可继续状态收敛成一套

`src/local-console/session-status.ts` 统一判定，产出「能不能继续 / 为什么不能 / 恢复动作是什么」。三种情况共用同一种表现：历史只读、输入与发送禁用、对应上下文按钮标红；区别只在说明文案和恢复路径。

恢复条件按 PRD 逐条落实：

- 项目文件夹修复成功 → 恢复输入、发送和推进，历史与选择不变。
- 团队被删除后改选另一支可用团队 → 恢复推进，历史保留，新团队从当前上下文接手。**这条依赖 B 片交付的团队改选入口**。
- 团队修复完成 → **自动**恢复推进能力，不需要用户再操作一次。

「团队已删除」必须从「需要修复」里拆出来：当前 `withAgentTeamHealth` 拿不到 snapshot 就一律归为 `needs-repair`，导致界面只能引导用户去修一个已经不存在的团队。

运行期分流把团队健康纳入 abort 判据，与项目文件夹现有的分流规则合并成一条：已经拥有有效隔离副本、能够安全完成的执行跑完当前这一步；依赖已经不可用的项目文件夹或团队内容、无法安全继续的执行立即停止并留下可读记录。**产品不得在执行已经无法继续时仍显示成员正在工作。**

### 收紧绑定团队的名单来源

`team-runtime-binding.ts` 对已绑定会话只读取团队快照，并把 `primaryAgentSlug` 对应成员排在首位；绑定团队已删除或需要修复时显式返回健康错误，不得用共享目录顶替。按照已登记的 `desktop-shell` / `local-console` 规格，没有团队绑定的存量会话仍回退共享 `agents/`，`server.ts` 的默认名单解析也保留这条兼容路径。

## 权衡

**新增 `system_event_kind` 列 vs 继续靠正文字符串判定。** 选了加列。不加列的改动更小，但界面对 runtime 文案形成隐式耦合，而本片恰恰要大改那批文案——改文案的同时依赖文案，等于给自己埋一个静默失效。加一列换来「四种事实可判定、重启后仍可见」两个 PRD 硬要求。

**旧 `awaits_human_reason` 整条作废 vs 保留成第四种红点来源。** 选了作废。保留看起来更保守，但 PRD 把「成员主动声明需要裁决」明确记入了待讨论并写道「界面上不出现『等你回话』这类词」——保留等于在待讨论问题上替产品拍板。作废的代价是 agent 请求裁决时暂时没有红点，这是 PRD 已知并接受的现状。

**未绑定存量会话兼容 vs 把它视作「团队已删除」。** 取前者，因为前序已登记规格明确要求这类会话继续使用共享目录；「团队已删除」只适用于存在绑定、但该绑定目标真实消失的会话。

**「反复重试仍未成功」不给重试按钮。** 产品已经替用户试过多次，再给一个按钮等于把「你自己再点点看」的责任推回去。给出路（说话 / 换一个成员接手）比给重复动作诚实。PRD 的页面状态表里这一行的可用操作也只有「说话、提及成员」。

## 风险

- **状态点判据换源会让现有 spec 与测试大面积失效**。`console-ui/spec.md` 的 `Conversation status dot semantics` 是 `main-sidebar-implementation` 刚固化的，`conversation-sidebar.test.tsx`（513 行）大量依赖它。缓解：spec-delta 显式标注为替换而非补充；测试随判据一起重写，不保留两套语义并存。
- **runtime 文案大改可能漏掉某条路径**，导致某种系统记录没有 kind。缓解：kind 为必填，缺省值是中性的「其他系统记录」而不是 null，界面对它有确定的降级表现；单测覆盖每条写系统消息的路径都带 kind。
- **团队健康换源可能误伤未绑定存量会话**。缓解：团队名单与健康解析都显式保留未绑定回退，并用桌面装配测试覆盖；只有已绑定且目标消失的会话进入「团队已删除」。
- **把团队健康纳入运行期 abort 判据，可能误杀正在安全运行的执行**。缓解：判据与项目文件夹现有分流保持同一条——只有依赖已不可用内容、无法安全继续的才立即停；已有有效隔离副本的一律跑完当前步。
- **回滚思路**：`system_event_kind` 列可保留（新增列不影响旧读路径），界面与 runtime 的改动可整体 revert；本 change 不写不可逆的会话团队绑定迁移。
