# 设计：qa-cluster-sweep-before-report

## 方案

### 落点：`qa/AGENT.md`「检查方法」段

现有第 17~20 行四条规则不动其结构。做两处最小修改：

1. **改写第二条**，把「定向优先、全量兜底」显式化：
   > 根据改动影响选择检查范围，优先覆盖状态转换、失败恢复、并发边界、兼容性和其他模块的回归风险；**先跑与本次改动相关的定向测试，绿了再决定要不要全量兜底——小改动不配全量回归**。

2. **在原第二条之后新增一条**「同类横向穷尽」：
   > 发现某一类缺陷（并发一致性、状态转移、失败恢复、时间聚合等）时，先在本改动涉及的文件与调用点里横向扫完这一类的**姐妹场景**——同一入口的其它写路径、同一依赖的其它使用点、同一后台任务的其它异常边界——把整批缺陷合并汇报。**不要抓到一个就报一个**，那会把本可一轮讲完的问题拆成多轮返工。

### 关键概念界定（避免规则被误用）

- **「一类缺陷」**指的是**同一失效机制**，不是「同一份文件里的所有问题」。例：ingest.py 里 pending 心跳的时间倒退、锁前生成 recv、覆盖写入未取 max 属于同一类（时间单调性 + 锁边界）；同文件里的 SQL 拼接漏洞与它们不属于一类，不需要非要一次讲完。
- **「姐妹场景」**限定在本改动涉及的**文件与调用点**里横向扫；不要求 qa 遍历整个仓库找同类问题——那是 code review 的范畴。
- **「合并汇报」**不是等到所有类别都扫完才汇报。发现类别 A 就把类别 A 扫透后可以先出一版；类别 B 是后续再看代码时发现的，正常再报一轮。规则要压的是「同一类被切成多轮」，不是「多类别必须同批」。

### 与其它角色规则的兼容性

- **dev-manager「返工必须重新复核，三轮不收敛熔断」**：本 change 不动。qa 更充分地一次汇报，只是让「不收敛」发生概率降低。
- **dev「code-verified 后交回」stage marker**：不动。dev 仍按 stage marker 交回，只是拿到的反馈更完整。
- **qa 的 workspace_access: read-run**：不动。本 change 只增加检查**深度**的要求，不改写权限。

### 判定的可核查边界

规则落地后，判定「有没有按规则走」的方式：

- 一份 qa 报告里对某一类别列出的 blocking 数 ≥ 2 → 按规则；
- 一份 qa 报告里只报一个某类别的 blocking，dev 修复后下一轮 qa 又在**同一份文件的同一类别**里报出新的 blocking → 违反规则，触发 dev-manager 追问「上一轮为何没横向扫完」；
- 一份 qa 报告里跨类别只各报一个 blocking → 正常，规则不覆盖跨类别横向；
- 无 blocking → 与规则无关。

## 权衡

- **为什么不改 dev 的自检清单**：见 proposal 反证表。dev 层不是这次 bug 的守门人；把清单塞到 dev 一侧会让 dev-manager 反思闸门要额外验证「清单是否覆盖并发一致性」，反而增加编排负担，收益却不落在 session 事实上。
- **为什么不改 dev-manager 反思触发词**：那一层运行在「方案 vs 目标」上；本次 4 个 blocking 是「代码 vs 方案承诺」上的偏差，不属于反思闸门的职责区。加进去等于让 dev-manager 假装能做 code review，会稀释它现在极致轻的编排定位（占比只有 3%）。
- **为什么把「定向优先、全量兜底」并进本 change 而不单起**：改动落在同一段（qa/AGENT.md 检查方法），措辞可以顺手一起写；单独起一个 change 会让追溯反而变碎。
- **为什么不落到 spec-delta/agents-team 新域**：现有 agent 团队协作相关 spec 都在 `github-issue-runner` 域下（历史包袱，见 `2026-07-02-add-dev-manager-agent`、`2026-07-06-agent-handoff-closing-protocol` 等），本 change 沿用该归属，避免为一条规则新开域。

## 风险

- **规则被过度解读为「必须扫全仓库」**：靠 design.md「关键概念界定」小节的「限定在本改动涉及的文件与调用点」显式收紧。
- **规则被解读为「所有类别必须一次全报」**：靠「合并汇报不等于跨类别同批」显式收紧。
- **对 3 轮熔断的替代性**：本 change 让「一轮讲透」概率提高，但不消除多轮场景；熔断规则保留作长尾兜底。
- **回滚**：本 change 是纯文本追加，回滚即恢复 qa.md 到当前版本，无任何运行时数据 / API / schema 影响。
