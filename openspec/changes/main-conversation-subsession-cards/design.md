# 设计：main-conversation-subsession-cards

## 方案

### 卡片是主时间线上的一条记录，不是浮层

编排产生子会话时，runtime 在父会话时间线插入一条卡片锚点记录。卡片因此有确定的时间位置——它出现在「开发经理说要拆成三段」这句话之后，而不是悬在时间线顶部或底部。

卡片的每一行渲染子任务、负责成员、当前状态三列。**状态是必须的**，且这一点与本项目「能被对话内容本身表达的不给状态标记」的原则不冲突：那条原则的前提是内容能表达状态，而子会话的内容不在主时间线上。卡片是唯一的聚合入口，不给状态用户就只能挨个点开才知道跑完几个、有没有一个挂了。

状态取值与 C 片的四种事实对齐（进行中 / 没跑起来 / 已结束等），由服务端聚合接口给出，卡片不自己推导。

### 子会话从侧边栏摘除

`toSidebarProject` 过滤掉带父会话的会话，`conversation-sidebar.tsx` 的 lineage 一行小字随之删除。

这会正面推翻 `console-ui/spec.md` 的 `Flat session rail with persisted lineage`。该 Requirement 是 `main-sidebar-implementation` 固化的，当时的判断是「父子会话同级平铺 + 标出父会话」；PRD 的判断是子任务根本不该进侧边栏。后者是产品事实源，前者是实现规格，冲突时以前者为准。

注意只摘 UI，不动数据：`parent_session_id` 与 `session_edges` 继续服务运行时编排与恢复，这一点在既有 spec 里已经写明「不驱动 UI 树形层级」。

### 右侧展开区只做外壳

PRD 明确把展开区的内部结构、输入方式和操作划给该区域的独立需求。因此本片只交付：

- 打开：点击卡片行。
- 关闭：显式关闭动作。
- 父会话可见：宽窗下父会话右半被覆盖，左半仍在视野中。
- 关闭后回到父会话原来的滚动位置。
- 窄窗下覆盖整个主内容区。

内部先接现有会话视图。**不为它发明输入方式或操作集**——那会在独立需求建立之前先造一份要推翻的事实。

滚动位置恢复的实现要点：打开展开区时记下父会话滚动容器的位置，关闭时还原。不能依赖「重新渲染后浏览器自己保持」——主区分栏会改变容器宽度，进而改变换行与内容高度。

### 窄窗收敛顺序

上下文四项按 分支 → 工作空间 → 团队 → 项目 的顺序逐项收敛。分支最先收，因为它是四项里唯一不可点击的纯信息；项目和团队最后收，因为 PRD 点名它们最后收敛，而且它们是仅有的两项还能改的上下文。

## 权衡

**卡片锚点落成时间线记录 vs 由界面从子会话列表实时合成。** 选了落记录。实时合成不需要改 runtime，但卡片会失去确定的时间位置——用户看不出这次拆分是在哪句话之后发生的，而 PRD 的卡片图明确画在开发经理那句「我拆成三段并行推进」之下。落记录还顺带让卡片在重启后位置稳定。

**状态由服务端聚合 vs 界面遍历子会话自行推导。** 选了服务端。界面推导需要把每个子会话的完整状态都拉到前端，子任务多时开销明显；而且状态语义（什么算「没跑起来」）与 C 片是同一套判据，放在服务端只有一份实现。

**右侧展开区只做外壳 vs 顺手把内部也做了。** 选了只做外壳。PRD 明确划出了边界，越界的代价是：等该区域的独立需求真正建立时，已经存在的实现会成为「现状偏见」，让需求讨论被实现反向限制——`docs/product/AGENTS.md` 专门警告过这件事。

**推翻 `Flat session rail with persisted lineage` vs 保留平铺并额外加卡片。** 选了推翻。两者并存意味着同一个子会话有两个入口，且侧栏膨胀问题原封不动——PRD 摘除它的理由正是膨胀。保留只是回避冲突，不是解决它。

## 风险

- **推翻已固化的 spec 会撞现有测试**。`conversation-sidebar.test.tsx`（513 行）与 `operator-console.test.tsx` 有针对平铺 rail 与 lineage 的用例。缓解：spec-delta 显式标注为替换；用例随行为一起重写，不留两套语义并存。
- **主区分栏是本片唯一的布局骨架改动**，可能影响时间线的滚动跟随、自动到底与回到底部行为。缓解：分栏只改变可用宽度，不改变滚动容器的归属；分栏前后各验一次「有新内容且停留在底部时自动跟随、已向上翻阅时不打断」。
- **滚动位置恢复容易在内容高度变化时失效**。缓解：记录与还原基于父会话滚动容器，且在还原后再校正一次；真窗口验收里显式覆盖「展开期间父会话有新消息进来」这一情形。
- **本片依赖 B 片交付的 `composer-context.tsx`**。若 B 未归档，窄窗收敛无处可落。缓解：里程碑依赖序已固定 A → B → C → D。
- **回滚思路**：本片不改 SQLite schema（卡片锚点走既有消息表），回滚只需恢复侧边栏平铺与移除新组件。
