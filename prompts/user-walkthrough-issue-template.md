# 用户走查 issue 模板（tranfu-agents-manager 例行观察）

## 用法说明（这部分不要贴进 issue）

- 把下面「issue 正文」整段贴到 tranfu-labs/tranfu-agents-app 新建 issue 的 body 里，替换 `<占位符>`。
- 模板刻意不重复画像（`agents/tranfu-agents-manager.md`）里已有的内容：他关心什么问题、发言必须落在所见的具体内容上，都由画像本身约束。模板只负责画像管不到的事：这一轮干什么、去哪看、产出什么结构、收尾交给谁。
- 首轮验证目标有两个：一是确认 codex 能访问到有真实数据的页面；二是确认发现清单落在具体页面元素上，而不是复述画像偏好——后者若发生，先查画像红线是否失效，而不是往模板里加戒条。
- **不要把你自己已观察到的问题写进 issue**——它们是对照组。走查结束后核对发现清单是否覆盖了它们：覆盖不了，说明采证或任务框定有问题，先修模板再谈自动化。
- 跑通后此模板交给 cron 定时开 issue，`<上轮已提清单>` 由上一轮 issue 的收尾评论带入。

## issue 标题建议

```text
例行生态观察 <本轮日期>：/skills
```

## issue 正文

```markdown
@tranfu-agents-manager 本周例行看一眼生态。

这不是审计任务，不是"去找 BUG"。按你平时的习惯，用下面的页面回答你自己关心的那些问题。回答的过程中，凡是答不上来、要绕路才能答上、或者答得让你别扭的地方，就是你要提的；答得顺畅的地方一个字都不用写。

访问方式：<线上地址 https://tranfu-agents-app.tranfu.com/skills，或本地启动方式>。

## 输出要求

**先记录，后发现。**

第一部分是使用记录：你实际走的路径，每一步打开了哪个页面、点了什么、看到了什么。没有使用记录支撑的发现不算数。

第二部分是发现清单，每条按这个结构写：

1. **我当时想弄清什么**
2. **我在页面上具体看到什么**
3. **为什么这不是我要的**
4. **我期望看到什么**

## 收尾动作

- 够得上行动的发现，打包 @dev，让他收敛方案。
- 拿不准的归入"待观察"，一行带过，不展开。
- 最后维护一段"已提清单"：本轮提过的 + 下面上轮已提的合并列出，供下轮去重。

## 上轮已提清单（这些不用再提，除非情况有变化）

<上轮已提清单，首轮填"无">
```
