# 里程碑 5：主页面会话区

> **状态：方案已定，等待放行（2026-07-21）。** 依赖序：A 独立且为风险优先起点，gate 住 B/C/D；B gate 住 C（「团队已删除」的恢复路径就是 B 的团队改选）；D 相对独立但爆炸半径最大，排在最后。
> **与里程碑 4 的关系：** M4 把对话介质从 GitHub issue 迁到本地，交付的是「能跑通」；本里程碑把同一块界面对齐 `docs/product/pages/main-conversation.md` 定义的产品事实，交付的是「用户敢用、读得懂」。二者顺序依赖，不并行。

## 背景

`docs/product/pages/main-conversation.md` 已作为主页面会话区的产品事实源，定义了 21 条验收标准与 13 种页面状态。该文档末尾的「现状参考与产品缺口」列出了当前实现距离它的 11 条缺口。

对照代码核实后，这 11 条不是同一量级，且其中三条不是「缺少」而是**行为冲突**，必须拆除而非补齐：

1. **工作空间是项目级设置**。`worktree_mode` 落在 `projects` 表上，切换一次波及该项目下所有对话；PRD 要求它是每段对话各自的设置。
2. **侧边栏状态点的判据是「等人回话」**。现有 `sessions.awaits_human_reason` 表达的是 agent 请求人工裁决；PRD 要求红点只由「没跑起来 / 卡住 / 反复重试仍未成功」触发，且用户按下「停下」不触发。二者不是同一件事，是判据换源不是调参。
3. **子会话在侧边栏平铺**。`console-ui/spec.md` 的 `Flat session rail with persisted lineage` 明确要求父子会话同级平铺；PRD 要求子会话不进侧边栏，改由时间线中的卡片承载。

此外，「新建对话」当前是一个模态弹窗且进入即落盘一段空白会话，与 PRD「新对话不是一个动作，而是这个页面还没有任何消息时的样子」的立场直接相反。

## 核心设计立场（先行固化）

- **会话是发消息的副产品，不是一个独立动作**。进入新对话页不创建任何东西；发出第一条消息时会话才诞生。这条改变了 `state-sync.ts` 的选择态 mutation 模型，是本里程碑最先要消除的不确定性。
- **能被对话内容本身表达的，不给状态标记**。成员在回复就说明了交棒，成员在吐字就说明了运行中。只有四种事实升格为可见状态：没跑起来、卡住、用户按了停、反复重试仍未成功——它们全部来自机器故障或用户动作，不是团队协作的失败。
- **上下文四项都要说真话、都要能改**。分支显示真实分支名而非「当前分支」这类字面词；工作空间归属会话不归属项目；团队可在对话中改选，且要让用户看出用的是创建时载入的那一份。
- **切换不打断，跑完当前这一步再生效**。工作空间和团队都可以在推进中改变，但不中止正在跑的那一步、不留半截改动、不重放已完成的步骤。
- **子会话是团队的内部产物，不是用户主动管理的对象**。它不进侧边栏（否则侧栏长度随团队的拆分行为膨胀），改由主时间线上的卡片作唯一聚合入口；卡片必须带状态，因为主时间线表达不了子会话内部发生了什么。
- **PRD 的「待讨论」四项本版明确不作答**，实现一律留空，不得自行决定：用户看不见 Agent 动了什么、出错时的原始错误输出、排队补话、成员主动声明需要裁决。

## 成功标准（里程碑级）

一个真人从桌面台侧边栏顶部「新建对话」进入，完整走过：**项目未选定态写草稿 → 选项目 → 发出第一句（对话此时才诞生）→ 看着团队推进 → 推进中改选团队与工作空间（跑完当前步才生效）→ 遇到一步没跑起来时重试 → 点开子会话卡片看子任务再回主线**；全程 `docs/product/pages/main-conversation.md` 的 21 条验收在真实桌面窗口中逐条可演示，包括重启应用后草稿与四种事实仍然可见。

质量基准：**`成品级`**。会话页是用户和团队的唯一工作面，PRD 已定义 13 种页面状态与三种不可继续状态的恢复条件；这里出一次「被红点叫回来却看不出要我干什么」，用户就不敢再让团队动他的真实项目。不存在 demo 档的空间。

## 验收方式（里程碑级约定）

- **纯逻辑走 Vitest**：标题取首句规则、工作空间解析与迁移、状态点判据、子任务状态聚合。这些的边界条件截图验不出来，且是后续片次的回归护栏。
- **页面状态矩阵与响应式走真实桌面窗口**：`pnpm desktop` 开发态暴露的 CDP 端口 `9222`（见 [ADR-0002](../adr/0002-electron-cdp-dev-debug-channel.md)）attach 真窗口逐条比对。不引入 fake server 验收脚本——PRD 有大量「重启后仍然可见」「窗口变窄时逐项收敛」类判据，fake server 验不住。

## 本里程碑明确不承诺

以下来自 PRD「待讨论」，本版明确不作答，四片实现一律留空：

- 用户看不见 Agent 动了什么、也拿不回来（改动汇总、停下后的清理路径、独立工作空间的成果出口、首次使用默认值的提醒）。
- 出问题时按需展开的原始错误输出；「一步卡住」的精确判据与静默时长展示；「重试」到底重跑什么。
- 用户在团队工作期间「排队到当前这一步结束后再送达」的补话方式。
- 成员主动声明「我需要用户裁决」及其是否成为红点来源；验收裁决专用控件；用户修改对话标题。

以及产品层面已确认的非目标：交付结果的独立陈列区、右侧展开区内部结构、Agent 团队管理界面、对话归档与恢复、高保真视觉参数。
