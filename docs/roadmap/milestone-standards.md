# 里程碑设定标准

## 用途

本文是人类和未来编排者拆里程碑时共用的决策尺子。它用于在写里程碑文档、拆任务和写 issue 验收语句前判断：当前拆法是否能减少上下文税、是否可追责、是否能被角色端到端验收。

一个合格里程碑不是一组水平工程层的堆叠，而是一次能力解锁或最大风险消除。收尾必须形成可被角色端到端验收的垂直切片，并明确质量基准。

快速检查：

1. 这个里程碑结束时，是否有一个角色能从入口走到可观察结果。
2. 最大不确定性是否已被提前 spike 或实验证据消除。
3. 里程碑质量基准是否显式写成 `demo 级`、`数据正确级` 或 `成品级`。
4. 每个任务是否对应一组可机械执行的验收语句，并尽量减少与兄弟任务的文件重叠。

## 四原则

### 1. 垂直切片原则

标准：里程碑必须代表一次能力解锁或最大风险消除。它可以很小，但必须从某个角色的入口出发，穿过必要的界面、逻辑、数据或运维环节，到达一个可观察、可验收的结果。

正例：`M0 假账扫码支付通路 demo` 允许使用假账户和假订单，但用户能扫码、看到支付成功，商户侧能看到假订单变为已支付。它是 demo 级，但仍然是一条端到端支付叙事。

反例：`M0 完成所有数据库表设计`、`M1 完成所有接口`、`M2 完成所有前端页面`。这些都是水平分层，每一层单独结束时都没有角色能完成一条真实工作流。

检查问题：

- 谁是这个里程碑的验收角色。
- 这个角色从哪里进入。
- 角色做完后能看到什么状态变化。
- 哪些底层能力可以暂时假实现，但不能切断端到端路径。

### 2. 风险优先排序

标准：最不确定、最可能推翻方案的风险先处理。风险可以通过 spike、原型、故障注入或小范围端到端试跑消除，而不是等所有低风险工程层做完后再面对。

正例：M1 T4 先用 spike 验证本地 worktree 与无头浏览器截图路径是否可行，再决定后续 artifact publisher 和观察页任务如何拆。它没有先做完整产品功能，而是先消除最影响后续路线的技术不确定性。

反例：先把列表页、数据库、配置项和文案全部做完，最后才验证核心截图链路或发布链路是否可行。一旦关键路径不可行，前面的工作会大面积返工。

检查问题：

- 哪个假设如果错了，会推翻后续最多任务。
- 能否用一个 spike 在一两天内验证该假设。
- spike 的验收语句是否能证明风险已经被消除，而不是只证明“写了一段代码”。
- 低风险工作是否被提前做得过多。

### 3. 显式质量基准

标准：质量基准是里程碑的显式属性，必须写进里程碑文档。执行方不应靠猜测判断本轮是 demo、数据正确还是成品交付。

正例：`M0` 明确标为 `demo 级`，允许假数据，但验收必须看到完整扫码支付叙事；`M1` 明确标为 `数据正确级`，要求单用户 happy path 的订单、余额和流水持久化正确；`M2` 明确标为 `成品级`，要求账户安全、异常路径、幂等和可观测性进入验收。

反例：任务只写“实现支付系统第一版”。执行方可能交一个看起来能点的 demo，需求方却期待真实数据正确；也可能执行方过度建设成生产级，拖慢原本只需要验证叙事的阶段。

检查问题：

- 里程碑文档是否写明本轮质量档位。
- 验收语句是否匹配质量档位。
- 哪些能力是本轮明确不承诺的。
- 如果新增质量档位，是否已显式写入里程碑文档，而不是在执行中口头发明。

### 4. 任务粒度标准

标准：一个任务等于一组可机械执行的验收语句，加上与兄弟任务最小的文件重叠。任务不是按“层”拆，也不是按“人力感觉”拆，而是按可独立验收的结果拆。

正例：`artifact publisher 发现 worktree 验收截图 + run manifest 契约` 是一个任务，因为它有明确文件范围和机械验收：构造 worktree PNG 引用后应进入发布路径；任一 Codex run 后 manifest 应有 issue、role、stage、产物、时间字段。

反例：`完成后端`、`完成前端`、`补齐所有文档`。这些任务没有清晰验收语句，且容易与兄弟任务反复修改同一批文件。

检查问题：

- 这个任务能否用几条命令、文件检查或 UI 操作验收。
- 验收语句是否描述可观察结果，而不是描述执行过程。
- 任务文件范围是否和兄弟任务大量重叠。
- 如果验收需要多个角色协作，控制权移交是否写清楚。

## 质量基准

未来新增顶层档位必须显式写入对应里程碑文档；不得在执行中临时发明隐含档位。

| 档位 | 定义 | 适用场景 | 验收要求 | 反例 |
| --- | --- | --- | --- | --- |
| `demo 级` | 用最小真实或假实现证明一条能力叙事能走通。允许假数据、人工步骤或 mock，但必须标清假设和不承诺事项。 | 早期验证用户理解、交互通路、技术可行性或演示叙事。 | 验收必须能从角色入口走到可见结果，并能说明哪些数据或能力是假的。 | 只有静态截图、只有数据库表、只有接口草稿，角色无法走通端到端路径。 |
| `数据正确级` | 核心状态、计算、持久化和 happy path 对真实数据正确。非核心体验可以粗糙，但不能损坏核心事实。 | 验证业务核心是否成立，例如订单、余额、权限、状态机或产物发布契约。 | 验收必须覆盖真实数据写入、读取、刷新后仍一致，以及关键不变量不被破坏。 | 看起来能点通，但刷新后状态丢失；金额、订单状态或发布记录不可追溯。 |
| `成品级` | 面向真实使用的可靠性、异常路径、安全性、可观测性和体验完整度进入交付范围。 | 准备交给真实用户、真实运营流程或长期无人值守系统使用。 | 验收必须覆盖 happy path、异常路径、恢复路径、权限或安全边界、日志 / 观测和用户可理解反馈。 | 只测 happy path；失败时静默；无权限边界；无人值守后卡死但没有死信或观测。 |

## 任务验收语句写法

验收语句必须可机械执行。优先使用以下格式：

- UI 类：`打开 X → 做 Y → 应看到 Z`
- 非 UI 类：`跑 X → 应输出/退出码 Z`
- 文档类：`打开 X → 应看到 Y`

合格验收语句描述结果，不描述“应该努力做什么”。如果需要人工判断，也要把判断依据写成可观察对象，例如文件路径、页面状态、命令输出、截图 artifact 或 issue 时间线评论。

正例：

```text
打开 docs/roadmap/milestone-standards.md → 应看到垂直切片、风险优先、显式质量基准、任务粒度四原则，均附正反例。
```

反例：

```text
把里程碑标准写好。
```

## Worked Example：以“支付宝式项目”为抽象支付产品拆解

这里的“支付宝式项目”只是一个抽象支付产品拆解示例，用来说明里程碑设定方法。它不代表真实支付宝架构、牌照、清结算或合规方案，也不承诺任何真实资金处理能力。

| 里程碑 | 能力切片 | 为什么这是垂直切片 | 质量基准 | 验收方式 |
| --- | --- | --- | --- | --- |
| M0 | 假账扫码支付通路 demo：用户打开付款页，扫描商户二维码，用假账户完成一次支付；商户侧假订单显示已支付。 | 覆盖了付款用户、商户二维码、支付动作和商户可见结果。即使账户和订单是假的，角色仍能端到端理解“扫码支付”叙事。 | `demo 级` | 打开 demo → 扫描示例二维码并确认支付 → 应看到用户侧支付成功，商户侧假订单状态变为已支付，并看到假数据说明。 |
| M1 | 单用户真实 happy path：一个真实测试用户能登录，使用持久化余额或测试账本支付一笔商户订单，刷新后订单、余额和流水仍一致。 | 把 M0 的叙事推进到真实数据正确：付款、扣款、订单状态和流水形成单用户闭环。 | `数据正确级` | 跑单用户支付 happy path 验证 → 应看到订单已支付、余额扣减、流水记录一致，刷新或重启后数据仍可追溯。 |
| M2 | 账户安全与异常路径：加入支付口令或等价安全校验、余额不足、二维码过期、重复提交、网络重试和幂等保护。 | 在同一支付闭环上补齐安全和异常，解决真实使用前最容易破坏信任和数据正确性的风险。 | `成品级` | 跑安全与异常场景集 → 应看到错误口令被拒绝、余额不足不扣款、过期二维码不可支付、重复请求不重复扣款，且日志 / 状态可追踪。 |
| M3 | 商户侧：商户能生成收款码或订单，查看收款列表、订单详情和对账导出；付款用户完成支付后商户侧实时或可刷新地看到结果。 | 新增商户角色的端到端工作流，而不是只给已有用户流程加页面。商户从创建收款入口到确认收款和对账形成闭环。 | `成品级` | 打开商户后台 → 创建收款订单并让测试用户支付 → 应看到商户订单变为已支付，列表 / 详情 / 对账导出中的金额和状态一致。 |

这个拆解避免了“先做所有数据库、再做所有接口、最后做所有页面”的水平层做法。每个里程碑都能被对应角色验收，同时逐步提高质量基准和风险覆盖范围。
