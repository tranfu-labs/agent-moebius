# 0006. 本地会话拆分主理人控制车道与专业 Agent 执行车道

## 状态

accepted

## 背景

本地会话原先只有一个 session cursor、一个 `processingSessions` 互斥和一个 session 级 `activeRun`。这个模型把团队接力等价成单 Agent 顺序执行：用户在专业成员运行时发给主理人的消息只能等待当前 Codex CLI 退出；composer 的停止按钮也只能含糊地指向“当前成员”。

产品现已确认同一根会话允许一个主理人与多个专业 Agent 同时运行。composer 是用户与主理人的控制面，专业 Agent 是由主理人调度、可逐条观察和停止的执行者。继续维持 session 全局单车道会让主理人无法及时回应，也无法用 `runId` 区分多个停止目标。

## 决策

- 每个 session 建立一条最多一个 active run 的主理人控制车道。
- 每个专业成员建立自己的执行车道；不同成员可并行，同一成员保持串行。
- 所有 composer 用户消息先进入主理人车道。主理人忙时消息保持为持久化 pending，终态后 FIFO 发射。
- 活动运行以 `sessionId + runId` 为停止身份，state 返回全量 `activeRuns`；单值 `activeRun` 只保留为主理人兼容投影。
- 主理人回复中的 mention 负责启动或重定向专业成员；命中活动成员时必须等旧 run 真正终态后再启动新 run。
- SQLite/JSONL 继续是状态提交边界和可恢复事实源；renderer 不持有只存在内存中的待发射队列。

## 后果

正面：

- 用户可以在专业成员工作时继续与主理人沟通，主理人可以回复或决定中断成员。
- 多个专业成员各有可见运行和精确停止目标，不再由一个全局“当前 run”互相覆盖。
- pending 正文与附件沿既有事实写链持久化，刷新和重启不会丢失。

代价：

- session cursor 不再等价于唯一执行锁；runtime 必须区分控制调度与成员执行完成，并处理交错完成顺序。
- 同一会话的多个写成员共享工作空间，存在文件级竞争。该竞争由主理人派工与团队职责约束管理，本决策不把同一会话偷偷拆成多个工作空间。
- API 和 UI 需要在迁移期同时维护 `activeRuns` 与主理人 `activeRun` 兼容投影。
