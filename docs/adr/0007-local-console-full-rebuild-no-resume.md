# 0007. 本地会话正常步骤全量重建，仅同次未完成执行允许 resume

## 状态

accepted

## 背景

Codex CLI 支持两种执行模式：每次 `codex exec` 冷启动一个全新线程（full），或 `codex exec resume <threadId>` 在既有线程上续跑（resume）。仓库中两者都有实现（`src/codex.ts` 的 `CodexRunMode`），但使用者分裂：

- 本地会话运行时（`src/local-console/runtime.ts`）正常步骤硬编码 `{ kind: "full" }`——每轮把整条会话时间线重建成 prompt，喂给一个全新线程；`onThreadStarted` 记录 run 到 thread 的稳定关联。
- resume 分支的唯一调用方是已废弃的 GitHub 终端 runner 路径（`src/runner.ts` + `src/conversation.ts` 的增量规划）。

2026-07-23 对一条真实开发团队会话的排查证实了全量重建的实际形态：开发经理连续 9 轮运行，每轮均为全新 thread id；最后一轮（符合度核验）上下文约 9.7 万 token（模型窗口 25.8 万，占用约 37%），全轮累计输入 131 万 token 中缓存命中 92%——因为每轮 prompt 前缀就是上一轮时间线加增量，天然对 prompt cache 友好。

由此产生的疑问是：resume 是否值得接进本地会话？需要先澄清一个常见误解——resume 的主要价值不是省 cache（全量重建的缓存命中已接近上限），而是**工作记忆的延续**：线程里保留上一轮读过的文件内容、工具输出与中间推理，下一轮无需重新探索。上述核验轮跑了 61 个工具调用，其中相当部分是重读上一轮已看过的代码，这是全量重建的真实代价。

## 决策

本地会话的**正常步骤固定为全量重建**；仅当产品明确恢复同一次未完成执行时，允许对该 run 的唯一 thread 使用 `resume`：

1. 会话时间线是成员间唯一的共享事实源；普通新消息、成员交接和下一步骤都从完整时间线冷启动，不为 `session + role` 保存可续跑私有线程。
2. `resume` 只服务三种显式恢复入口：正常退出前已持久化的可恢复暂停、用户对卡住/未启动记录点击「重试」、用户对已停下记录执行「改一改重发」。恢复请求必须携带原 `runId`，只能解析该 run 已持久化的唯一 `threadId`。
3. 恢复前校验 session、原角色及团队快照、工作空间身份与 thread 关联。关联缺失、冲突、rollout 不可用或上下文不兼容时，安全降级为 full；不得按 role、时间或“最近 thread”猜测。
4. 正常退出先持久化恢复意图再中止 Codex；只有该意图允许启动时自动恢复。崩溃/强杀形成的无标记孤儿仍确定性落成 stuck，由用户点击重试。
5. 「改一改重发」仍追加公开用户消息；resume prompt 是对原执行的明确修正增量。它不修改历史，也不回滚已经发生的文件副作用。
6. `src/codex.ts` 保留受控的 full/resume 两种 driver 能力；业务层不得把恢复例外扩张为正常对话续线程。若未来「每轮重新探索」成为普遍瓶颈，仍优先让成员把结论写进共享时间线。

判据来自四点权衡：

- **事实源单一**：常规 resume 会让每个成员维护一条与共享时间线渐渐分叉的私有历史；把 resume 收窄到同一个未完成 run，且下一步骤重新 full，可避免形成长期私有记忆。时间线事实源不变式仍由 `tests/local-console-timeline-truth.test.ts` 约束。
- **上下文有界**：resume 线程只增不减，最终依赖 Codex 自动压缩兜底，压缩质量不可控；全量重建每轮上下文有界且可治理，规则遵守只取决于角色文本与时间线质量。
- **可恢复但不依赖**：线程文件存于 `~/.codex/sessions`，重启、清理、换机都可能断链；因此 resume 是优化路径，完整时间线 full 是永远可用的正确性兜底。
- **成本可接受**：缓存命中率 90% 以上时，全量重建的有效开销远低于名义输入量，换取上述三条属于划算交易。

## 后果

正面后果：

- 成员行为可治理：每轮冷启动意味着规则遵守不随会话时长漂移，调优只需要改角色文本或时间线内容。
- 会话对崩溃、清理、迁移仍然健壮；线程丢失只让恢复动作降级为 full，不阻断继续工作。
- 正常退出可以延续正在进行的文件探索和工具上下文，用户无需重复发送原消息；卡住和改写重发也能复用同次执行上下文。

负面后果与代价：

- 每轮重新探索的 token 与时延成本真实存在（观测到单轮 61 个工具调用、名义输入 131 万 token）；缓解手段是时间线持久笔记，而非切换执行模式。
- 成员没有跨轮私有记忆：上一轮未落进时间线正文的中间发现会丢失，对成员回复的信息密度提出要求。
- 依赖 prompt cache 控制成本；若缓存失效（前缀变动过大、缓存过期），单轮成本显著上升。
- 局部恢复新增了恢复意图、兼容性校验、退出时序和降级路径，必须用持久事实与测试防止误续到其他 run。
