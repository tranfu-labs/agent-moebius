# 0006. 会话内团队切换在步骤边界接管，并重组后续 prompt

## 状态
accepted

## 背景

本地会话在创建时载入一支 Agent 团队的内容快照。团队快照决定主 Agent、可用成员以及每名成员的 `AGENT.md` persona；runtime 在每一步运行前据此解析合法 mention、选择执行成员，并把团队上下文与完整会话时间线组装进 prompt。

因此，“已有会话能否切换团队”不只是界面 select 是否可用的问题。切换会改变后续 prompt 的 persona、主 Agent、成员名单和交棒范围，但不应改写已经开始运行的一步，也不应丢弃同一会话已经形成的上下文。需要明确以下边界：

- 正在运行的 Agent 不能因为改选团队而被中断或在半途中更换 prompt。
- 新团队必须看到同一会话的完整历史，不能通过新建会话或重跑旧步骤模拟接管。
- 会话使用的是选择时载入的团队内容；团队页之后的编辑不能隐式改变已经作出的选择。
- 项目、工作空间和改动基线属于会话，不随团队接管而变化。
- 团队被删除或需要修复时，改选一支可用团队仍是保留历史并恢复推进的出口。

讨论过把团队限制为只能在新对话时选择，首条消息后永久锁定。该方案可以减少“同一会话由不同团队先后处理”的认知成本，但并不能解决运行一致性之外的新问题；现有 pending/effective 快照边界已经避免了运行中 prompt 被改写。永久锁定还会失去按任务阶段换团队，以及团队不可用时在原会话恢复推进的能力。

## 决策

保留已有会话改选团队的能力。团队切换定义为**同一会话在步骤边界发生的后续控制权接管**，遵循以下规则：

1. **空闲时立即接管**：会话没有成员正在运行时，新团队绑定和内容快照立即成为 effective；下一条待处理消息使用新团队。
2. **运行时延迟接管**：会话有成员正在运行时，改选只写入 pending 团队绑定及其内容快照。当前步骤继续使用启动时已经组装好的旧团队 prompt，不中断、不重启。
3. **步骤收尾时原子落定**：当前步骤结束后，pending 绑定与快照一起提升为 effective 并清空 pending。进程重启不能丢失尚未落定的选择。
4. **选择时冻结内容**：改选时立即载入新团队当时的主 Agent、成员顺序和各成员 `AGENT.md` 内容。团队页之后的修改不影响该会话已载入的版本。
5. **后续 prompt 按新团队重组**：每个后续运行从 effective 快照取得被选成员的 persona、主 Agent 和可用成员名单，并继续附上同一会话的完整时间线。历史消息、项目、工作空间、分支和改动基线不因切换而重建或迁移。
6. **交棒按新名单解释**：新团队生效后的 mention 解析和无 mention 回交都以新成员名单为准。旧团队最后一条回复若没有指向新团队可用成员的合法 mention，控制权回到新团队主 Agent；新主 Agent可以直接承接收尾。
7. **界面披露生效时机**：运行中改选后，团队控件显示目标团队，并说明“当前这一步跑完后换成该团队”；不得暗示正在运行的 prompt 已经变化。

团队切换不重放已经完成的步骤，不把旧团队的运行归因给新团队，也不改变工作现场。它改变的是切换边界之后“谁读完整历史并负责下一步”的执行上下文。

## 后果

正面后果：

- 用户可以按目标阶段更换更合适的团队，同时保留同一项目现场和完整会话上下文。
- 正在运行的一步具有稳定 prompt；切换不会留下因中断产生的半截执行。
- 选择时快照让后续行为可复现，不会被团队页的外部编辑悄悄改变。
- 团队删除或需要修复时，可以通过改选团队在原会话中恢复，而不必放弃历史。
- 新团队主 Agent 能在旧成员完成当前步骤后自然接回控制权，维持主 Agent 收尾约束。

负面后果与代价：

- 同一会话可能先后出现不同团队的成员，界面和文档必须让“从下一步开始接管”足够清楚。
- 新团队需要理解旧团队留下的历史；两支团队的职责假设差异较大时，接管质量取决于时间线是否充分。
- 旧团队回复中的 mention 若不在新团队名单内，将不再构成合法交棒，而会回到新主 Agent；这是确定性边界，但可能与旧成员生成回复时的意图不同。
- 相同 slug 在两支团队中可以对应不同 persona；切换后使用新团队选择时冻结的内容，而不是沿用旧团队同名成员。
- runtime 与存储层需要长期维护 effective/pending 两套绑定和快照的事务一致性、重启恢复及覆盖测试。
