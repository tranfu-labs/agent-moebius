# 0004. 会话事实日志采用每会话 jsonl，SQLite 保留流转状态与索引

## 状态
proposed

## 背景

T3（#101/#102）把当时散落的可变 JSON 状态文件（`role-threads.json`、`agent-contexts.json`、`goal-ledger.json`、GitHub intake 与本地消息）统一进单一 SQLite 库，解决了单文件损坏互相牵连、新旧文件反向覆盖、事实源不明的问题。此后会话消息只存在 SQLite 的 `session_messages` 表中，磁盘上没有可直接阅读的会话记录文件；jsonl 只出现在每次运行的原始输出（`runDir/stdout.jsonl`）和 `.state/run-manifests.jsonl`。

产品侧随后出现了新的需求：用户要把一段对话的完整经过交给对话之外的工具或另一个 AI 做复盘（见 `docs/product/pages/main-left-sidebar.md` 的对话菜单）。会话记录若只存在数据库里，复盘必须先经过一次导出；而参照 Codex CLI 的实际存储布局（`~/.codex/sessions/YYYY/MM/DD/rollout-<ts>-<session-id>.jsonl` 为每会话追加日志，`state_*.sqlite` 等承载可变状态，`session_index.jsonl` 为索引），「会话记录本身就是磁盘上的追加文件」与「可变状态统一在 SQLite」可以并存，并不回到 T3 之前的碎片化状态——T3 反对的是一堆各管各的**可变** JSON 文件，而会话事实日志是**只追加、不改写**的记录。

约束与前提：

- 产品仍需要跨会话查询（侧边栏按项目列会话、红蓝闪点聚合、全局搜索），需要索引。
- 消息之外的可变状态（归档标记、路由裁决、未读、工作空间绑定、父子会话关系）需要事务性更新。
- 终端模式 GitHub runner 已废弃，不与桌面应用同时运行；产品状态的写者只有桌面应用主进程一个。
- 时间线的「运行中临时记录原地替换」是渲染语义，不要求存储可变：进度事件照常追加，界面取最新一条渲染。

## 决策

会话记录分两层存储：

1. **每会话一个只追加的 jsonl 事实日志**，作为消息与会话事件（用户消息、Agent 最终消息、进度事件、系统记录、子会话创建等）的唯一事实源。文件一经写入不改写、不删除行。子会话是另一个 session id 的独立 jsonl 文件，父会话日志中追加一条创建事件建立关联。
2. **SQLite 保留全部可变流转状态与索引**：归档、路由裁决、未读、工作空间绑定、会话列表索引、状态点聚合等。SQLite 中凡是可由 jsonl 推导的内容都定义为**可重建缓存**；两边不一致时以 jsonl 为准，通过重扫重建索引。

配套规则：

- **单写者**：每个会话的 jsonl 只由桌面应用主进程写入；Codex 子进程的原始输出继续写各自 `runDir`，不直接写会话日志。应用需保证单实例（或等价的会话级写锁）。
- **半行容忍**：进程崩溃可能留下不完整的最后一行；读取端必须容忍并截断半行，追加端在下次写入前校正。
- **原子提交点**：一条消息（含附件引用）以单行追加为提交点；附件先完成托管，再追加消息行。
- **迁移**：现有 `session_messages` 做一次性导出，按会话生成 jsonl 事实日志；迁移遵循 T3 同款规则——成功后记 marker，不删旧表数据，不允许旧数据反向覆盖新事实。
- ADR-0003 关于 SQLite driver 与 Worker 生命周期的决策不受影响，继续适用于保留在 SQLite 中的状态层。

## 后果

正面后果：

- 会话记录成为磁盘上可直接阅读、可被外部工具跟读的文件；产品的「复制对话记录路径」不再需要导出快照，路径指向持续更新的真实记录。
- 事实源单一且不可变，索引可随时重建；数据库损坏不再等于会话历史丢失。
- 与 Codex 的存储形态同构，降低理解与调试成本。

负面后果与代价：

- 引入双层存储的一致性纪律：写入顺序（先日志后索引）、重建路径和迁移都需要实现与测试。
- 会话记录文件暴露在磁盘上，用户或外部工具误改、误删该文件即损坏对话历史；产品按既有异常路径如实呈现，不承诺恢复。
- 跨会话搜索与聚合依赖索引层的维护；索引落后于日志时需要可见的重建手段。
- 历史会话需要一次性迁移，迁移期间新旧读取路径并存。
