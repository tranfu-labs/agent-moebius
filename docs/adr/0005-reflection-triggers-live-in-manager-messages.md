# 0005. 质量反思外部化：定向触发词由主理人以独立消息发出，成员自反思仅作粗筛

## 状态
accepted

## 背景

seeds 开发团队（`seeds/teams/development`）从旧 GitHub issue 体系（`agents/` 角色 + CEO 守护）迁移到本地会话体系。旧体系防「方案写漏 / 写多、实现执行少了 / 多了」靠两层：dev 的自反思（`openspec-driven-development` skill 步骤 4 / 6），加 CEO 守护的机械阶段回流（`plan-written` 强制交 qa 审查、`code-verified` 回流发起角色验收、验收语句缺失即打回）。本地体系有意去掉机械守护，由开发经理作为主理人承接兜底，于是要回答：反思介入以什么形态、放在哪一层。

关键输入：

- **实践观察**：让 agent 在自己回合里对自己念反思触发词，即便词句完全相同，检出率也明显低于同样的话由外部作为一条新消息发入。
- **研究证据**：无外部反馈的内在自纠不提升推理表现、有时反而降低（[Huang et al., ICLR 2024](https://arxiv.org/abs/2310.01798)）；自纠只在存在外部验证信号时可靠生效，模型无法可靠识别自身错误（[TACL 自纠综述](https://arxiv.org/html/2406.01297v3)）；模型系统性偏爱自己的产出——本质是偏爱低困惑度文本，而自己刚生成的内容困惑度最低（[Self-Preference Bias in LLM-as-a-Judge](https://arxiv.org/abs/2410.21819)）。
- **实践共识**：同一上下文里审查自己刚写的东西，得到的不是第二意见，而是第一意见的复述；独立会话 / 独立评审者检出更多（[多 agent 评审实践](https://www.mindstudio.ai/blog/automated-code-review-multiple-ai-agents)）。
- **机制解释**：外部消息享有角色显著性（模型对最新用户级消息的服从远高于对自己回合中段的自我指令）；回合边界迫使模型把先前产出当作被审对象重新读入；外部发问自带「有人怀疑」先验，把搜索姿态从确认翻转为找错。

候选方案：

1. 全部依赖 skill 内自反思——被上述证据与实践否定；
2. 复刻 CEO 机械守护（stage marker 触发、固定剧本模板）——与本地体系的轻量定位冲突，且旧体系已把「泛泛要求 agent 反思」列为空转失效模式；
3. 主理人判断式外部拷问——选定。

## 决策

反思按外部化程度分三层，归属固定：

1. **成员自反思**（skill 步骤 4 / 6）保留，仅作第一遍廉价粗筛，不作为质量采信依据。
2. **定向触发词是主防线，归属主理人消息层**：验收清单齐全后、放行前，开发经理以独立消息向 `@dev` 做双向核对——漏了哪些该做的、混进了哪些没要的（顺手重构、过度设计也算），要求逐条回应后再放行；改动很小时可并入放行消息。成员回合内的自查不能替代这条外部拷问。
3. **冻结验收清单是机械锚点**：方案时刻产出逐条可核查的清单，其后的采信、复核、验收都对照该清单做差异核对——多于清单即写多，少于清单即写漏。

配套规则（均已写入 `seeds/teams/development/members/*/AGENT.md`）：

- 拷问锚在清单与目标上（「清单第几条对应什么证据」），禁止「你确定吗」式裸质疑——外部质疑的已知副作用是谄媚式翻供，会把对的改错。
- 幂等重审：返工后的改动必须重新复核，上一轮「通过」不覆盖新改动。
- 熔断：修复与复核超过三轮不收敛即停止派发、上报用户。
- 「后面全自动」不取消清单闸门：团队里的自动衔接由主理人完成，方案停点照常；首次遇到时向用户提示该语义差异。

## 后果

正面：

- 防写漏写多的主防线建立在有证据支持的机制（外部回合）上，而不是寄望成员自觉。
- 团队形态由此获得单 agent + skill 结构性没有的资源——「外部回合」，这是本地团队相对单 agent 的核心差异化能力。
- 规则落在 seeds AGENT.md，论据落在本 ADR：未来任何想把触发词「精简」回 skill 自反思层的改动，须先推翻本条引用的证据。

负面与代价：

- 每份方案默认多一轮消息往返（小改动并入放行消息缓解）。
- 主理人的拷问质量成为单点：写成裸质疑反而诱发改口，需要靠 AGENT.md 的锚定句式约束。
- 三层职责分属 skill 与两份 AGENT.md，各自演进时需保持归属不漂移；skill 的自反思措辞与本决策解耦，不因本 ADR 修改。
