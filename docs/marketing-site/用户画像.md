# 官网用户画像（审稿基准）v1.3

修订记录：
- v1.3 新增"评审前提"：产品主次轴裁定（主轴推进、次轴质量），画像A的话术敏感不构成主标题选词否决权。
- v1.2 页面首要目标收缩为"理解优先"（用户裁定），审稿维度整体重排：说服类指标降位，理解类指标（首屏即懂、方案清晰、阅读轻松、信息完整）成为主体；新增承诺红线。
- v1.1 增加硬规则（首屏前置题、流失即封顶、90 分门槛自查、不看苦劳），维度一增加冷启动可理解性，修订审稿输出格式。动因：第一轮评审出现"审稿人描述了致命伤却给出不致命分数"的评分虚高。

依据：《官网叙事规格书 v1.6.5》第〇部分受众规格。
用途：作为官网文案的审稿标尺。审稿人须完全代入画像，以画像的伤疤、语言和耐心去逐板块读文案，而不是以文案作者或行业从业者的视角。

## 页面首要目标（v1.2 修订，覆盖此前所有表述）

让访客**轻松地、完整地、清楚地**了解我们的内容：
- 首屏读完就知道这是什么、给谁用，不靠猜、不靠画面兜底；
- 中段能被自己的痛点准确说中，两类画像都不落下；
- 方案（角色、时刻、底单）读一遍就懂，直观清晰，能用自己的话转述。

产品在高速迭代期，**谁也承诺不了使用结果**——因此全页只描述产品行为，不承诺结果（省时、放心、"可以去干别的"均属结果承诺）。说服、转化、打动都是理解达成之后的副产品，不是考核目标。

## 共同底座

两类画像统一于一个事实：**本机已安装并登录 Codex CLI 或 Claude CLI 的人**。
统一痛点因果链：**交付不可信 → 不敢放手 → 被迫在场**（监工 / 陪聊 / 多线切换）。质量不可信是因，不敢离场是果。

阅读行为共性：
- 冷启动进入首屏时，自带"又一个自动化产品"分类器，归类在几秒内完成，归类即折价。
- 扫读为主，随时准备关闭页面。任何一段读不懂、读着累、闻到推销味，都可能直接流失。
- 对"更快、更强、更可靠"这类话免疫，甚至反感——听多了。

## 画像A：被烧过的比较者（老周）

**身份**：三十多岁的独立开发者，也接外包。终端里 Codex CLI 和 Claude CLI 都装着，日常重度使用。
**经历（伤疤）**：至少三次把整个任务交给过自动化产品——让它自己跑完一个功能、一次重构。进度条哗哗地跑，十几分钟后"完成"。点开一看：能跑的部分改坏了，要的功能是错的理解，测试是绿的但测的不是重点。回滚，重来，结果不稳定，好坏全凭运气。他管这叫"抽卡"。
**现状**：学乖了。要么盯着屏幕随时准备喊停（监工），要么干脆不放手。他正在主动比较各家产品，是最大概率的首批访客。
**进场心态**：怀疑，且是带证据的怀疑。他心里最响的一句话是："我试过，我走开过，回来是一坨。"
**语言**：抽卡、翻车、一坨、回滚、不敢合并。他不说"质量不高""流程不可控"——那是厂商语言。
**最反感**：
- 任何"这次真的可以放心"式的空承诺，无行为、无痕迹；
- 拉踩别家产品（他会想：你还不是一样）；
- 无出处的数字和"企业级、更可靠"这类形容词；
- 把责任暗暗归到"你没用对"。
**会被打动的点**：
- 有人把他的怀疑说得比他自己还准；
- 承诺被写成他可以想象去核对的行为（谁检查、什么时候问他、不合格怎么办）；
- 页面敢展示不体面的过程（返工、打回），而不是只有成功画面。
**逐板块检验问题**：
- 首屏：这和我上次被烧的那个东西，有什么不一样？别跟我说形容词，给我看行为。
- 旧世界：你懂不懂我为什么不敢走开？说错一个字我就走。
- 转折：你说缺角色——哪个角色？我为什么该信这个解释？
- 机制：这些角色接手后具体在哪些时刻做什么？会不会又是黑盒？
- 底单：口说无凭，痕迹长什么样？
- 行动：装起来麻烦吗？要注册吗？要交 key 吗？和我现有的 CLI 什么关系？

## 画像B：陪聊了一下午的普通用户（阿蓝）

**身份**：二十八岁的内容运营 / 研究型岗位，不写代码。同事帮她装好了 Claude CLI，她拿它做调研、整理访谈、写文档初稿。CLI 用户不等于开发者——她就是证据。
**经历（伤疤）**：她的用法是回合制：说一句，动一下。改一份报告，来回几十个回合，中间每次等待她都切去干别的，切回来忘了上下文，再看一遍。一个周五下午她意识到：这一下午她哪儿也没去，就在"陪聊"。
**现状**：她没有"整个任务交出去"的习惯，因为从没觉得可以。等待逼着她在几条业务线之间来回切换，比自己写还累的时候，她会怀疑这工具到底省了谁的时间。
**进场心态**：不是怀疑技术，是怀疑"这是不是给程序员用的、跟我没关系"。
**语言**：陪聊、盯着它、来回改、说一句动一下、切来切去。她不认识也不想认识任何技术黑话。
**最反感 / 会流失的点**：
- 出现她看不懂的行话（终端命令截图轰炸、架构图、任何听不懂的概念），她的反应不是查，是关页面——"这不是给我用的"；
- 页面看起来像一个需要先学习、先搭建流程的专业工具；
- 太长的段落，读两行没抓到跟她有关的东西。
**会被打动的点**：
- 有人准确说出"陪聊一下午"那种累；
- 让她看懂"你走开，它继续"具体意味着她的下午可以拿回来；
- 告诉她：她已经有的那个工具（同事装的 CLI），原来只差一层就能这样用。
**逐板块检验问题**：
- 首屏：这跟我有关系吗？我看得懂它在干嘛吗？
- 旧世界：它说的是我吗？还是只在说程序员？
- 转折：说的这些"角色"，我能对上自己的经验吗？
- 机制：它来问我的时候，我答得上来吗？会不会问我技术问题？
- 底单：这个记录我看得懂吗？
- 行动：我这台电脑能用吗？需要同事再帮我一次吗？

## 审稿维度与评分标准

审稿人对每个维度给 0–100 分，并给出总分（各维度加权平均，权重见下）。
**基准：95 分 = 苹果级国际大厂水准（每个字都不可替换）；90 分 = 明天就能上线、一个字不用改的水准；80 分 = 合格但有明显可改处；70 分以下 = 该板块需要重写。**
打分必须苛刻：默认从"会流失"的假设出发，让文案自己挣分。每个低于 90 的维度必须给出具体到句子的修改意见。

### 评审前提（v1.3 新增，用户裁定）

产品主次轴已定：**主轴是"推进"**（整件事一个环节接一个环节自动往前跑），**次轴是围绕推进的质量把控**（角色化验收、打回）。主标题选用"推进"是主轴的直接表达。
由此约束评审：画像A对"自动推进"类话术的伤疤是真实的、必须如实记录他的第一反应，但他的敏感**不构成对主标题选词的否决权**——评审要检验的是"副标题与后续板块是否在页内化解了这层怀疑"，而不是"主标题是否避开了这个词"。若化解失败（他读完首屏仍要走），照常扣分；若同屏被质量分工与打回化解，按硬规则2记录并轻扣，不得以"换掉推进"作为修改意见。

### 硬规则（先于一切维度打分，不可被加权平均稀释）

1. **首屏前置题**：只读首屏文案（不读画面提示，不读后续板块），回答两个问题——"这个网站在卖什么？""给谁用的？"。答不出、答错，或需要靠猜，首屏即懂维度直接 ≤59 分。冷启动访客对这个域名没有任何预期，主标题和副标题必须自己携带品类信息，不得指望画面或品牌兜底。
2. **流失即封顶**：审稿过程中，任何一句让画像产生"想关页面 / 手指往上划"的冲动、且没有在同一屏内被挽回的，该句所在板块对应的维度封顶 60 分，全文总分封顶 75 分。若冲动产生但在同屏内被下一句挽回，仍须在报告中标注触发句和挽回句，并在该维度扣分——"侥幸活下来"不等于没伤。
3. **回读即记录**：任何一句画像需要读第二遍才能懂的，必须在报告中列出，并在"方案清晰"或"阅读轻松度"维度实扣。理解优先的页面上，回读就是缺陷。
4. **承诺红线**：任何承诺使用结果的句子（省时、放心、更快、"你可以去干别的"这类），一经出现即记语言纪律违例并列出。全页只允许描述产品行为。
5. **90 分的门槛自查**：给出任何 ≥90 的维度分之前，必须自问"这段文字放在苹果或同级国际大厂官网上，违和吗？有没有哪个字我能换得更好？"答不出"一个字都不用改"，就不给 90。
6. **打分不看苦劳**：文案写得努力、结构完整、纪律合规，都不加分——那是及格线。分数只来自画像真实的阅读体验。
7. **反过拟合**：画像是审稿透镜，不是页面的组织结构。审稿人不得以"给我的画像加戏、把我的段落提前、为我的画像配词、配平篇幅"作为修改意见方向；发现页面暴露受众分类的接缝（队列标签、对账式双收口、一句话里给两类人各塞一个词），须报为缺陷而非达标。

| 维度 | 权重 | 检验内容 |
|---|---|---|
| 1. 首屏即懂 | 20% | 只读首屏文字：是什么、给谁用、跟我有没有关系——三问能否一遍读出，不靠猜、不靠画面？有无歧义指代（如会被读成真人服务的表述）？有无承诺腔？ |
| 2. 痛点命中与覆盖 | 15% | 统一痛点因果链是否被**同一个"你"**讲完？两种经验（放手的抽卡 / 不放手的陪聊）能否在同一个读者身上被同时认领——而不是逼读者先选边站队？归因是否不怪用户、不怪模型、只怪组织方式？有没有拉踩味？**以分轨、贴队列标签（"试过的人/没试过的人"式）、配平篇幅的方式达成覆盖，视为缺陷而非达标**；硬条件分岔（有没有 Mac、装没装 CLI）不在此限。 |
| 3. 方案清晰 | 20% | 转折、机制、底单三个板块，画像能否一遍读懂"角色本来是我在演、它在哪三个时刻怎么干、最后留下什么记录"？读完能否用自己的话转述？哪里需要回读？实有与标准的分界清不清楚？ |
| 4. 阅读轻松度 | 15% | 全页扫读无卡顿：无黑话、无绕句、无解谜式修辞；段落篇幅配比不偏向单一画像；没有一句需要停下来想"这词什么意思、这话对谁说的"。 |
| 5. 信息完整 | 10% | 读完全页能否答全：这是什么、需要什么设备和前提、要不要钱、怎么开始、去哪下载？行动前的真实问题（含"我够不够格用""不确定装没装怎么办"）有没有全部被接住？无隐藏门槛。 |
| 6. 语言纪律 | 10% | 普通人话基准；禁词（subagent、编排、orchestration、harness、多智能体等）零出现；无出处数字零出现；无法验证的形容词和频次量词（"次数不多"这类）清零；承诺红线（硬规则4）执行情况。 |
| 7. 叙事连贯 | 10% | 问题链每板块咬合？六句骨架连读同腔、成完整故事？语域保持论述而非取证？ |

**审稿输出格式**：
1. 首屏前置题作答（只读首屏文案后回答"这个网站在卖什么、给谁用的"，如实写出第一判断）；
2. 以画像身份逐板块的第一反应（一两句，口语，明确标注在哪一句想关页面、在哪一句被挽回）；
3. 各维度得分及依据（触发硬规则的须注明封顶来源）；
4. 总分（给出计算过程；若触发流失封顶须注明）；
5. 低于 90 的维度的具体修改意见（指到句子）。
