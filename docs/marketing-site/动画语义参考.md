# 动效语义参考：业界惯用画法调研

## 背景与用途

Moebius 官网走新野兽派风格（奶油底、高饱和色块、粗黑描边、硬偏移阴影、黑底白字标签芯片），需要用纯 SVG/CSS 抽象示意动画表达一组机制语义。第一版全部自己臆造画法，结果被反馈"很多动画莫名其妙、语义看不懂"。

本文档的作用是：先总结业界在同类场景下**公认的视觉词汇**（形状、符号、运动方向），再落到我们自己的视觉 token 上。原则是——**先抄惯例，再换皮肤**，不要在没有惯例支撑的情况下自造符号。

我们项目里可复用的视觉 token（来自 `style-demo.html`）：

- 硬偏移阴影三档：小/中/大偏移黑色投影
- 描边两档：粗描边、细描边，颜色统一为墨黑
- 标签芯片：黑描边圆角矩形，内部纯色块（蓝/黄绿/橙/粉/描边留白）填充，常配一个小方点做状态指示
- 卡片：白底、粗描边、圆角、中等偏移阴影，悬停时整体位移且阴影收缩（模拟"按下去贴近纸面"的物理感）

## 1. 流水线 / 接力推进

任务从一个环节流向下一个环节。

### 业界惯用画法

- **步骤条（stepper）惯例**：圆形或方形节点 + 连接线。节点有三种状态外观——完成态用**实心填色 + 打勾图标**替换序号；当前态用**描边加粗或描边高亮 + 脉冲/呼吸动效**标出"正在进行"；未来态用**空心灰置、序号仍保留**，一眼就能看出"还没轮到"。
- **连接线的推进方式**：线段用 `scaleX(0) → scaleX(1)` 从左到右生长来表达"进度正在流动"，而不是直接切换透明度——生长动作本身就是"流向下一步"的语义来源。业界特别强调用 transform 而非 width 做这个动画，因为前者走 GPU 合成层、不触发重排。
- **DAG／有向无环图惯例**（CI/CD 流水线，如 CircleCI 可视化编辑器、GitHub Actions 的 Pipeline 视图）：任务节点用方框表示，依赖关系用带箭头的连线表示，允许并行分支再汇合，强调"方向性"和"节点间的因果连线"而非单纯罗列。
- **物流跟踪进度条惯例**：横向或纵向排列的图标节点（下单、在途、派送中、已签收），每个节点配一个专属小图标而非纯几何符号，强调"每一步是不同性质的事情"而不只是数值进度。
- **时间节奏**：入场用错峰（stagger），逐节点揭示，每个节点间隔约 30–50ms，总错峰时长控制在 300ms 内，让人感知"依次发生"而不是"同时炸出来"。

- [Beyond the Progress Bar: The Art of Stepper UI Design - Lollypop](https://lollypop.design/blog/2026/february/beyond-the-progress-bar-the-art-of-stepper-ui-design/)
- [Progress indicator – Carbon Design System](https://carbondesignsystem.com/components/progress-indicator/usage/)

### 对我们的落地建议
节点用我们的圆角方块芯片：完成态填色块+白色打勾，当前态描边加粗并叠加轻微脉冲阴影（小阴影↔中阴影交替），未来态用白底细描边灰字；连接线用硬直角折线（贴合野兽派的几何感）配合 `scaleX` 从左向右生长，颜色统一墨黑，不用渐变。

## 2. 对齐 / 澄清提问

开工前确认需求。

### 业界惯用画法

- **对话式追问惯例**：与其展示"报错/无法理解"，业界公认的更优模式是弹出**一句具体的澄清问句**（"你是指 A 还是 B？"），用对话气泡承载，气泡里直接给出可点选的分支选项而非要求打字——这样"提问"和"给出选择"被绑定在同一个视觉单元里。
- **分支选择惯例**：用二到五个并列的选项卡片/按钮表达"接下来要走哪条路"，选项数量刻意控制得少，避免选择过载；用户点选后其余选项淡出或收起，视觉上完成"收敛"。
- **表单确认惯例**：分步引导，一次只问一件事，问题以气泡或卡片形式逐条出现，避免一次性甩出长表单。
- **落点符号**：问号图标、气泡尾巴指向提问方，是这个语义最稳定的符号锚点，几乎不会被误读。

### 参考来源
- [Conversational UI Design — Patterns, Examples & Demos | AI Design Patterns](https://www.aiuxdesign.guide/patterns/conversational-ui)
- [AI Chatbot UI Design: 8 Patterns That Build User Trust](https://designpixil.com/blog/ai-chatbot-interface-design)
- [Onboarding UX: 10 patterns, best practices, and real examples](https://www.appcues.com/blog/user-onboarding-ui-ux-patterns)
- [Chatbot design challenges and tips | Jotform Blog](https://www.jotform.com/ai/agents/chatbot-design/)

### 对我们的落地建议
用一个黑描边气泡卡片（带小尾巴指向"提问者"图标），里面并排 2-3 个芯片按钮当分支选项；动效上气泡先弹入（ease-out 轻微过冲），选项芯片错峰浮现，用户视角的"选中"用其中一个芯片变实心色块、其余芯片渐隐收窄来表达收敛。

## 3. 验收 / 逐项检查

清单逐条核对打勾。

### 业界惯用画法

- **打勾动效惯例**：几乎所有成熟的 checklist/成功态动画都用 **SVG `stroke-dasharray` + `stroke-dashoffset`** 让勾号像"手写"一样被描出来，而不是直接淡入一个完整图标——"画出来"的过程本身就是"确认完成"的动作语义。常见时长 0.4–0.6s，配合外框圆圈也用同样手法画一圈。
- **逐项推进惯例**：清单条目自上而下依次点亮/打勾，每条之间有短暂错峰间隔，让人读出"正在一条一条过"而不是"整批瞬间通过"。
- **状态先后惯例**：未检查=空心方框/圆圈；检查中=描边填充过渡的中间态（可选）；已检查=实心打勾+文字变浅灰或加删除线，弱化已完成项、让注意力留在下一条未完成项上。

### 参考来源
- [Task Complete Checkmark Animation - CodePen](https://codepen.io/xkurohatox/pen/jPdrXz)
- [Success checkmark animation - CodePen](https://codepen.io/Johannakv/pen/XWzQKpe)
- [CSS-only Todo List Checkbox Animation - CodePen](https://codepen.io/shshaw/pen/WXMdwE)
- [CSS Animation: Circle loader with checkmark completed state - CodePen](https://codepen.io/scottloway/pen/zqoLyQ)

### 对我们的落地建议
清单条目用带黑描边的小方框芯片一条条纵向排列；每条从空心方框到实心色块打勾，用 SVG 描边动画"画出"勾号，条目间错峰 80–120ms；全部打完后可给最后一条加一次小阴影"落地"回弹，呼应野兽派按压感。

## 4. 打回 / 返工循环

不合格退回重做再验。

### 业界惯用画法

- **PR 审查"request changes"惯例**：主流协作工具（GitHub 等）把"打回"表达为流程图上的**反向箭头**，从"审查"节点指回"提交/修改"节点，形成一个闭环圈，而不是新开一条平行线——闭环形状本身传达"回到之前的状态重来"。
- **质检退回惯例**：常用**循环箭头图标**（两段弧线首尾相接指向对方，或单条打圈的箭头）叠加在流程节点上，表示"该节点会反复经过直到通过"。
- **循环节奏惯例**：循环箭头的动画常用旋转位移或"抖动一次后退回原位"的短促动作，配合节点本身短暂变色（黄/橙警示色）再变回，强调"这不是终态，是中间态"。
- **区别于终止态的关键**：打回循环必须让人明确看到"起点"和"回到起点"两端都存在，箭头首尾相连缺一不可；如果只画一个"×"或红叉，容易被读成"失败终止"而不是"退回重做"。

### 参考来源
- [The GitHub pull request review workflow - Graphite](https://graphite.com/guides/github-pull-request-review-workflow)
- [Reviewing proposed changes in a pull request - GitHub Docs](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/reviewing-changes-in-pull-requests/reviewing-proposed-changes-in-a-pull-request)
- [6 Practices for Effective Pull Requests | Pete Hodgson](https://blog.thepete.net/blog/2019/05/10/6-practices-for-effective-pull-requests/)

### 对我们的落地建议
画两个节点（"产出"、"审查"）+ 一条从审查指回产出的弧形回退箭头；触发打回瞬间"审查"节点短暂变橙色芯片再回归中性色，回退箭头本身做一次快速划过的位移动画（而非常驻旋转），明确传达"这一轮被退回，马上要重新过一遍产出"。循环箭头本身用粗黑描边，保持和其余线条一致的几何感。

## 5. 监工 / 盯着（负面场景）

人一直盯着屏幕不能走开。

### 业界惯用画法

这个语义在成熟插画体系（unDraw、Storyset 一类的"人物+电脑"组合）里没有单独的标准符号，但可以从相邻惯例拼出画法：
- **静止 + 长时间感**：用**没有变化的固定构图**本身来表达"钉住不动"——人物姿态、视线方向长时间不切换，是传达"被困在原地"最直接的手法；相反，若人物或视角频繁切换反而会削弱"盯着"的感觉。
- **视线焦点惯例**：插画体系常用眼睛/视线方向线直指屏幕，配合屏幕光晕表示"注意力被完全占用"。
- **时间流逝暗示**：可以叠加一个持续运行但不产生结果变化的元素（如光标持续闪烁、进度条空转），暗示"一直在等、一直在看，但没有进展"，这是负面语义的关键——单纯"盯着"不算负面，"盯着却等不到结果"才是负面。
- **负面基调手法**（见第 10 节通用惯例）：色相偏冷/偏灰，人物姿态略微佝偻或僵硬，避免使用鲜艳的强调色。

### 参考来源
- [unDraw - Open source illustrations for any idea](https://undraw.co/)
- [Storyset 插画风格说明](https://undraw.co/illustrations)（同源检索，Storyset 提供多风格人物插画库，可查"办公/等待"类）
- [UX Magazine：Designing Interface Animations](https://uxmag.com/articles/designing-interface-animations-the-book-excerpt)（负面动效的通用手法参考，见第 10 节）

### 对我们的落地建议
人物剪影固定不动，屏幕用一个持续闪烁的光标或空转的小圆点表示"卡在原地、没有产出"；背景色块用比常规更低饱和度的灰蓝，不用主色芯片，暗示这是"我们想避免的状态"而非正常流程节点。

## 6. 多线切换 / 混乱（负面场景）

一个人在多个任务间疲于奔命。

### 业界惯用画法

- **"多杂耍球/多任务"惯例**：插画体系里最通用的符号是**一个人物同时应对多个独立物件**（多个对话气泡、多个文件图标、多个屏幕），物件分布凌乱、方向不一致，人物姿态倾斜或伸展，用"物件数量超出人物能自然处理的范围"来表达过载。
- **运动惯例**：多个元素的动画**互相不同步、路径交叉**，制造视觉上的"抢注意力"感，与第 1 节流水线的"错峰但有序"形成鲜明对比——秩序感的缺失就是混乱语义的来源。
- **图标堆积惯例**：任务/通知图标以不规则角度堆叠、部分重叠出画面边界，暗示"处理不过来"。
- **色彩惯例**：常用多种高饱和色块并置且不协调排布（不是我们平时精心搭配的配色），刻意制造"杂"的视觉感受。

### 参考来源
- [unDraw / Storyset 插画库（多任务/杂耍类检索）](https://undraw.co/illustrations)
- [Person juggling multiple tasks - Vecteezy](https://www.vecteezy.com/vector-art/55631705-person-juggling-multiple-tasks-representing-multitasking-and-time-management)
- [Multitasking Overworked 插画合集 - Dreamstime](https://www.dreamstime.com/illustration/multitasking-overworked.html)

### 对我们的落地建议
中心人物剪影不动，周围 4-6 个不同色芯片（代表不同任务）以不规则角度、不同步的抖动/位移动画环绕出现，部分芯片可以短暂溢出画布边缘再弹回，制造"处理不过来"的视觉噪音；避免使用我们芯片体系里干净排列的样式，这里故意打破网格对齐。

## 7. 角色接管 / 交接

原来你做的事被别人接走。

### 业界惯用画法

- **接力棒交接惯例**：最通用、跨行业公认的符号是**接力棒（baton）从一只手传到另一只手**，捕捉"交接瞬间"两只手同时接触棒身的画面，传达信任和责任转移，且几乎不需要额外文字说明就能被理解。
- **交接方向惯例**：图形上强调"从左到右"或"从一侧到另一侧"的传递方向，与流水线的推进方向保持一致的阅读习惯（左进右出）。
- **抽象简化版**：不用完整人物插画时，也可以简化成**一个色块图标从 A 位置移动到 B 位置**，并在移动路径上做一次"交接点"的短暂高亮或轻微放大，模拟"手递手"的接触感。
- **区别于流水线的关键**：流水线强调"任务本身流动"，交接强调"执行者变了但任务不变"——画法上常用同一个任务图标不变形，但图标外圈的"持有者"标记（如头像/色块边框）发生切换。

### 参考来源
- [Baton Hand Off 插画合集 - iStock](https://www.istockphoto.com/illustrations/baton-hand-off)
- [Hands Passing Baton 插画合集 - iStock](https://www.istockphoto.com/illustrations/hands-passing-baton)
- [Passing the Baton When You Have That New Job - StickyMinds](https://www.stickyminds.com/article/passing-baton-when-you-have-new-job)

### 对我们的落地建议
用一个任务芯片在两个"持有者"图标（可以是简化的人形色块）之间做一次位移动画，芯片经过中点时轻微放大+阴影加深模拟"接触感"，落到新持有者一侧后边框颜色切换为对方的标识色，明确"内容没变、负责人变了"。

## 8. 过程底单 / 审计记录

一张逐行记录过程的单据。

### 业界惯用画法

- **小票打印惯例**：收据/小票类动效的公认画法是**纸条从顶部或底部逐行吐出**，每一行内容随纸条滚动依次显现，模拟真实热敏打印机"边打边出纸"的物理过程，比"整张收据一次性淡入"更有"正在记录"的实感。
- **时间线惯例**：审计日志类界面公认用**纵向时间线**——左侧一条竖线连接各条记录的圆点，每个圆点带时间戳，新记录从顶部或底部插入，历史记录整体下移或上移，符合"时间只往一个方向走"的直觉。
- **逐行铺陈惯例**：与流水线的"错峰"类似，审计记录也常用逐行淡入/滑入，但每行完全是**静态文本条目**而非可交互节点，强调"这是已发生事实的记录"而非"正在进行的流程"。
- **符号锚点**：小票用锯齿边/纸卷图形作为容器边界，是这个语义最容易被一眼认出的形状特征。

### 对我们的落地建议
用一张白底黑描边的纸条容器（顶边或底边做锯齿裁切模拟撕口），配合等宽字体逐行从顶部滑出/浮现，每行前面可加一个小时间戳或序号芯片；纸条本身可叠加轻微硬阴影强调"这是一张实体单据"而不是弹窗提示。

## 9. 终端窗口 vs 聊天软件

CLI 的图形表达惯例对比聊天界面。

### 业界惯用画法

**终端窗口（CLI）惯例：**
- 窗口顶部三个圆点（红/黄/绿，即 macOS 风格"交通灯"按钮）是终端类视觉最强的身份符号，即使不做任何功能也要保留，用户一眼识别"这是命令行"。
- 黑底（或深色底）+ 等宽字体 + 荧光绿/白字，是"终端感"的第二身份符号。
- `$` 或 `>` 提示符固定出现在每行输入起始位置，是"这里可以敲命令"的惯例标记。
- 光标用实心色块（不是竖线）以固定频率闪烁（display:none 切换或 opacity 0/1 交替），是终端最具辨识度的动效元素，几乎不需要其他装饰就能让人认出"命令行"。
- 内容偏好用 ASCII 字符/线框（如 `┌─┐│└─┘`）做装饰而非图标或渐变，强调"极简、面向文本"的调性。

**聊天软件惯例：**
- 用**圆角气泡**区分说话方，一侧对齐（自己）、另一侧对齐（对方），靠对齐方向而不是边框区分身份。
- 气泡之间靠间距而非分隔线区分消息，视觉更松散、更"对话感"。
- 常配头像、时间戳、已读回执等社交类元素，强调"双方在交流"。
- 输入框固定停靠在底部，是聊天类界面最强的身份符号，与终端"命令行紧跟提示符"的位置逻辑不同。

**两者对比的核心差异**：终端强调"单向、精确、面向指令"的调性（黑框+等宽+提示符），聊天强调"双向、随意、面向交流"的调性（气泡+头像+对齐错位）。这组对比本身就是很好的"专业工具 vs 日常聊天"隐喻素材。

## 10. 通用动效惯例

### 入场 / 强调 / 循环的节奏惯例

- **入场（entrance）**：统一用 ease-out（快进慢出），让元素"冲进来又稳稳停住"，比线性运动更有物理可信度；多元素入场一律错峰（stagger），单个间隔 30–50ms，总错峰时长不超过 300ms，超过这个时长用户会感觉"卡顿"而非"有节奏"。
- **强调（emphasis）**：给关键节点的动作分配更长时长、更明显的位移或缩放幅度；次要元素则同步、群组化处理，避免所有元素争夺注意力。
- **循环（loop）**：表达"持续进行中"的状态（如流水线当前节点、终端光标）用固定周期的呼吸/闪烁动画，周期要短而规律（如光标 1s 一闪），避免不规律的循环让人误以为是卡顿或异常。
- **退出/闭合**：与入场对称，用 ease-in（慢进快出），循环类动画结束时要有明确的"收尾帧"（例如打勾定住、进度条停在终点），不要让动画消失在中途状态，否则用户分不清是完成了还是卡住了。

### 负面场景的通用识别手法

业界公认让人一眼读出"这是负面/异常状态"的手法组合（通常叠加使用 2-3 种，单独一种容易被忽略）：

1. **抖动（shake）**：短促的左右或上下小幅位移，参考"摇头说不"的身体语言，是最强的负面动效信号，常用于错误提交、非法操作。
2. **变色（红/橙警示色）**：从中性色瞬间切到红色或橙色再淡出/保持，但要避免仅用颜色表达（色觉障碍用户会看不出区别），必须搭配抖动、图标变化等至少一种非色彩线索。
3. **堆积（pile-up）**：元素不规则堆叠、超出容器边界，用"装不下了"的视觉过载表达失控，适合多线切换类场景。
4. **残影/拖影（trailing/blur）**：运动物体后面拖一道模糊或半透明残影，表达"匆忙、来不及"，适合表达疲于奔命的场景。
5. **静止/停滞（freeze）**：与前四种相反，用长时间不变化本身表达"卡住了、没有进展"，适合监工/等待类场景——**动与不动都可以是负面信号，取决于语义本身是"失控"还是"停滞"**。
6. **节奏失序**：把原本该错峰有序的动画改成不同步、路径交叉，用"秩序感的缺失"本身传达混乱，不需要额外加警示色也能读出负面。
