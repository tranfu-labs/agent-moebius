# Agent 团队

> 状态：已确认，可开发
>
> 所属页面：应用级 Agent 团队管理
>
> 当前范围：定义团队陈列首页、团队详情、团队与成员管理、内置只读边界、文件异常状态，以及对 Sidebar、新建对话和会话页的跨页面约束。会话内团队选择与异常恢复的具体交互另行定义。
>
> 产品基线：[agent-moebius 产品需求文档](../prd.md)

## 页面目标

Agent 团队页让用户看见自己当前拥有的团队，理解每支团队负责什么、由谁默认接单、包含哪些专业成员，并在同一团队上下文中完成创建、复制、修改、修复和文件维护。

团队是本页的基本管理单位。每支团队严格包含自己的一组 Agent，并指定其中一名为主 Agent。Agent 不作为脱离团队的全局对象单独管理；相同角色出现在不同团队时，也视为彼此独立的成员配置。

本页不把团队呈现为组织架构或固定工作流。团队成员扁平并列，主 Agent 只是新会话中的默认接单者，不代表上下级、执行顺序或 DAG 关系。

主 Agent 的 `AGENT.md` 是团队接收新任务时的协作入口；其他成员各自的 `AGENT.md` 定义专业职责和自然语言交棒规则。产品只判断团队结构是否完整，不尝试理解或证明自然语言协作关系一定正确。

团队进入会话后，成员能够知道当前团队包含哪些 Agent 及其稳定标识，从而按各自 `AGENT.md` 的自然语言规则向同团队成员交棒。该成员清单不等同于固定执行顺序。

## 用户场景

- 用户从 Sidebar 进入 Agent 团队页，查看软件自带团队和自己创建的团队。
- 用户进入某支团队，确认团队用途、主 Agent 和成员构成。
- 用户复制一支只读的内置团队，再按自己的需要修改。
- 用户创建一支新团队，逐步添加 Agent，并让第一名成员自动成为主 Agent。
- 用户在同一团队详情内切换成员，编辑各自的 `AGENT.md`。
- 用户需要维护 `AGENT.md` 引用的其他文件时，在 Finder 或当前平台的文件管理器中打开团队或成员位置。
- 用户修复被外部移动、重命名、删除或变得不可读的团队文件。
- 用户将不再使用的用户团队移到系统废纸篓或回收站。

## 入口与去向

### 入口

- 从[主页面侧边栏](./main-sidebar.md)顶部的“Agent 团队”进入。
- Sidebar 入口只要发现至少一支团队处于“需要修复”，就在右侧显示一个红点；多支异常团队不叠加数量。
- 红点的悬停提示和辅助名称为“有 Agent 团队需要修复”，不能只依赖颜色传达含义。
- 未完成草稿是正常编辑状态，不触发 Sidebar 红点。

### 去向

- 点击团队首页的一整条团队横行，进入该团队详情。
- 团队详情中的成员行同时是成员选择器；点击后在当前页面下方切换对应的 `AGENT.md`，不进入独立 Agent 页面。
- “在 Finder 中打开”打开团队或当前成员对应的位置，供用户维护本页不负责编辑的相关文件。
- 返回团队首页时，保留此前的滚动位置；返回主页面时，恢复离开前的项目、对话和 Sidebar 状态。

│                 │  │ 3 名成员 · 开发经理带队      │              [→] │ │
│                 │  └──────────────────────────────┴──────────────────┘ │
│                 │                                                       │
│                 │  ┌──────────────────────────────┬──────────────────┐ │
│                 │  │ 我的开发团队                 │ [经理][开发][测试]│ │

## 区域与信息

### 页面标题与新建入口

- 页面标题固定为“Agent 团队”。
- “新建团队”是首页唯一常驻的创建操作。
- 页面不提供独立的全局 Agent 库入口。

### 团队横行

每条团队横行包含：

- 团队名称。
- 一句话用途描述。
- “内置”“未完成”或“需要修复”等必要状态。
- 成员数量和主 Agent 名称。
- 主 Agent 排在第一位的扁平成员示意。
- 进入详情的整体点击区域。

成员显示名称和一句话描述来自各自 `AGENT.md` 中的对应信息；团队横行和详情不再维护一份独立、可能漂移的成员摘要。

团队横行不常驻修改、复制或删除按钮。首页负责陈列、识别和进入，具体管理操作进入详情后完成。

### 扁平成员表达

- 不绘制连接线、箭头、环形轨道、中心节点、树形缩进或上下层级。
- 所有成员使用相同尺寸和视觉权重。
- 主 Agent 只通过排在第一位和“主 Agent”标签表达。
- 切换主 Agent 后，新主 Agent 自动移动到第一位。
- 其他成员不表达执行顺序，也不提供手动排序。
- 首页空间不足时显示前几名成员和 `＋N`；详情页保持单行并允许横向滚动。

### 团队身份

用户团队的“修改信息”只负责团队名称和一句话描述，不展开成员 `AGENT.md`。内置团队的身份信息只读。

本产品不存在可由用户设置的应用级“默认团队”。软件只记住上一次成功创建会话时使用的团队，供下一次新建对话预选。

### 主 Agent

- 每支可用团队必须且只能有一名主 Agent。
- 主 Agent 必须是当前团队成员。
- 主 Agent 是所选团队进入新会话后的默认接单者，不是其他成员的父节点或管理层级。
- 用户团队详情顶部常驻主 Agent 选择器，不把切换能力藏入“修改信息”或“更多”。
- 第一个成员自动成为主 Agent。
- 删除当前主 Agent 前，必须先指定另一名有效成员。
- 内置团队不能切换主 Agent。

### Agent 身份与说明

- 每个 Agent 在创建时取得一个团队内唯一的稳定 slug，用于团队内识别和协作引用；创建后不因修改显示名称而变化。
- Agent 的显示名称和一句话描述属于 `AGENT.md` 内容，可以与正文一起在编辑器中修改。
- 团队首页、成员行和当前 Agent 标题都读取 `AGENT.md` 中的显示名称和一句话描述。
- 本页不提供修改稳定 slug 的入口，也不把显示名称变化视为另一个 Agent。
- 团队详情中的成员信息持续显示并允许复制 `@<slug>`；显示名称负责帮助人理解，slug 负责稳定引用。

### Agent 提及与选择

- 在软件内编辑 `AGENT.md` 时，输入 `@` 和 Agent slug 可以匹配当前团队成员；匹配结果同时显示成员的可读名称和 `@<slug>`。
- 用户选中成员后，编辑器显示以可读名称为主的提及组件，不让 slug 成为正文里的主要视觉噪声。
- 提及组件底层仍保存 `@<slug>`；使用 Finder 或其他编辑器打开原始 `AGENT.md` 时，看到的是稳定的 `@<slug>` 文本。
- 提及组件悬停或获得键盘焦点时，可以查看并复制底层 `@<slug>`。
- 显示名称改变后，既有提及组件自动显示新名称，但底层 slug 和引用关系不变。
- 自动补全只帮助选择当前团队成员，不允许从团队外引入共享 Agent，也不验证整段自然语言交棒是否合理。

### 成员选择器与 Agent 编辑区

- 成员选择器和 Agent 编辑区都属于团队详情，不新增独立 Agent 页面。
- 点击成员只切换下方当前编辑对象。
- 软件内只编辑成员的 `AGENT.md`；它引用的其他文件不在软件内展示文件树或提供编辑器。
- 当前成员菜单承接“在 Finder 中打开”“复制 Agent”和“删除 Agent”。内置团队的成员不提供修改、复制单个成员或删除操作；用户需要复制整支团队后编辑。
- 团队之间不移动 Agent。需要复用时复制团队或复制当前团队中的 Agent，复制后独立演化。
- 删除成员前必须提示：其他成员的自然语言交棒规则可能仍引用该成员，产品不会自动理解或改写这些规则。用户确认后仍可删除；只要剩余结构继续满足可用条件，团队不因此自动进入“需要修复”。

### Finder 与平台文件管理器

- 团队“更多”菜单中的“在 Finder 中打开”打开整支团队对应的位置。
- 当前成员菜单中的同名操作打开当前成员对应的位置。
- Finder 是辅助文件维护入口，不作为页面主操作常驻展示。
- Windows 和 Linux 使用各自平台对应的文件管理器名称和行为。
- 内置团队的 Finder 入口只用于查看软件自带内容。通过外部工具修改内置文件不能绕过只读限制，也不会把内置团队变成用户团队；需要修改时必须先复制整支团队。

## 操作与反馈

### 新建团队与添加成员

- 新建团队先创建可恢复的草稿，不要求用户在一个长向导里一次写完整支团队。
- 草稿会保留在团队首页，直到用户继续完成或主动删除。
- “添加 Agent”在当前团队详情内切换到新成员编辑状态，至少包含创建后不再改变的稳定 slug，以及写有可编辑显示名称和一句话描述的初始 `AGENT.md`。
- 第一个成功添加的 Agent 自动成为主 Agent。
- 团队恰好有一名主 Agent、每名成员都有团队内唯一的稳定 slug，且所有成员的 `AGENT.md` 都可读时，就满足结构上的可用条件，可以供新对话选择。
- 单成员团队也可以使用，但只代表结构有效，不代表已经具备多 Agent 专业分工。
- 产品不分析 `AGENT.md` 的自然语言内容来验证交棒是否正确；协作质量由用户实际定义的 Agent 内容和后续会话结果体现。

### 修改团队信息

- 用户团队可以修改名称和一句话描述。
- 内置团队不提供修改入口。
- 修改团队信息不改变已经存在会话载入的团队版本。

### 切换主 Agent

- 用户从团队详情顶部的主 Agent 选择器中选择现有有效成员。
- 切换成功后立即保存并给出轻量反馈；不再要求进入额外编辑模式。
- 新主 Agent 移动到成员行第一位。
- 切换只影响之后使用该团队创建的会话；已有会话继续使用创建时载入的版本。

### 编辑与保存 `AGENT.md`

- `AGENT.md` 不自动保存。
- 修改后，当前成员显示未保存标记。
- 切换到其他成员时不弹窗，当前草稿继续保留；多名成员可以同时保留未保存草稿。
- “保存”只保存当前成员；支持标准的 `Command/Ctrl + S`。
- 保存成功后清除对应成员的未保存标记。
- 保存失败时保留编辑内容，在当前编辑区显示原因和重试。
- 离开团队详情时如果仍有未保存内容，统一提供“继续编辑”“放弃全部”和“保存全部并离开”。
- “保存全部并离开”逐个保存已修改的 `AGENT.md`。部分保存失败时，成功项保持已保存，失败项继续保留草稿，用户停留在当前团队详情并看到失败成员列表；产品不伪装成整体成功，也不回滚已经成功保存的文件。

### 外部修改冲突

- 本节只适用于可编辑的用户团队。内置团队即使通过 Finder 查看，也不能通过外部修改进入可编辑或已更新状态。
- 软件内没有未保存内容时，返回应用后重新载入外部更新的 `AGENT.md`，并给出轻量提示。
- 软件内已有未保存草稿时，不自动覆盖，保留草稿并提示文件已在外部修改。
- 用户只能选择“载入外部版本”或“用当前内容覆盖”。本页不提供逐行 diff 或自动合并。
- 其他引用文件的外部修改不触发本页刷新或冲突提示。

### 复制

- 内置团队的主操作是“复制并编辑”。
- 复制团队会创建一支独立的用户团队，复制团队对应的完整目录，包括团队信息、全部成员目录、各成员 `AGENT.md` 及其他相关文件，并进入新团队详情。
- 复制不会改变新建对话的“上一次使用”记录。
- 用户团队也可以从“更多”菜单复制。
- 用户团队中的单个 Agent 可以在成员菜单中复制；复制会包含该 Agent 的整个目录和全部相关文件，复制品仍属于当前团队。
- 在同一团队内复制 Agent 时，复制品取得新的团队内稳定 slug，避免与原成员的协作引用冲突；显示名称和其他文件内容可以从原成员开始再修改。
- 团队或成员存在未保存的 `AGENT.md` 草稿时，复制前必须让用户保存或放弃这些草稿；复制始终基于已经完整落盘的文件，不把内存草稿悄悄混入副本。

### 删除用户团队

- 内置团队不可删除。
- 用户团队和未完成草稿可以从详情页“更多”菜单移到系统废纸篓或回收站。
- 删除前说明团队包含的 Agent 和相关文件，并要求确认。
- 有未保存 `AGENT.md` 草稿时必须先处理草稿，不能直接删除。
- 删除成功后返回团队首页，并从新建对话团队选择器中移除。
- 被删除的团队如果是“上一次使用的团队”，下一次新建对话回退预选软件自带团队。
- 已有会话及其创建时载入的团队版本不因删除而被移除。
- 本产品不建设独立的已删除团队页面，恢复文件使用系统废纸篓或回收站。
- 从系统废纸篓或回收站恢复文件后，团队如何重新回到产品暂不在本页面承诺；文件层可恢复不等于自动恢复应用内记录。
- 已有会话在团队被主动删除后能否继续发送新消息，归会话交互 PRD 裁决；本页面只保证历史和已载入版本不会随删除一起消失。

### 新建对话中的团队预选

本页只定义跨页面约束，具体控件由[新建对话窗口](./new-conversation.md)负责：

```text
新建对话
  → 自动预选上一次成功创建会话时使用的团队
  → 用户可以在创建前改选
  → 进入会话
  → 由所选团队的主 Agent 默认接收消息
```

- 首次使用且没有历史选择时，预选软件自带的第一支团队。
- 上一次使用的团队已经删除或处于“需要修复”时，回退预选软件自带团队。
- 打开、编辑或浏览团队不更新“上一次使用”记录；只有成功创建会话后更新。

## 页面状态

| 状态 | 页面表现 | 是否可用于新建对话 |
| --- | --- | --- |
| 内置团队 | 标记“内置”，详情和 `AGENT.md` 只读，提供复制并编辑 | 是 |
| 正常用户团队 | 正常显示团队信息、主 Agent、成员和编辑区 | 是 |
| 未完成草稿 | 标记“未完成”，保留继续添加成员入口 | 否 |
| 需要修复 | 红色异常标识，显示具体不可用范围和修复入口 | 否 |
| `AGENT.md` 已修改 | 当前成员显示未保存标记，草稿留在内存 | 仍使用最近一次已保存版本 |
| 正在保存 | 当前编辑区显示进度并防止重复保存 | 仍使用最近一次已保存版本 |
| 保存失败 | 保留草稿，显示原因和重试 | 仍使用最近一次已保存版本 |
| 外部修改冲突 | 保留软件内草稿，等待版本选择 | 仍使用冲突前最近一次有效版本 |
| 页面加载中 | 使用保持团队横行结构的加载占位 | 等待加载结果 |
| 页面加载失败 | 保留页面框架，显示错误和重试 | 不把失败误判为没有团队 |

正常产品状态下一定存在至少一支软件自带团队，因此不设计“完全没有团队”的普通空态。内置团队也无法加载时，显示应用配置异常和重试，不引导用户把损坏状态当作正常空白重新创建。

### 团队位置不可用

团队整体位置被移动、重命名或变得不可访问时，团队继续显示在首页并标记“需要修复”。详情提供“重新定位团队”和“移除记录”，已有会话历史不消失。

```text
┌────────────────────────────────┬───────────────────────────────┐
│ 我的开发团队          需要修复 │ [开发经理] [开发] [测试]      │
│ 团队文件夹已移动或不可访问     │  暂时无法用于新对话            │
└────────────────────────────────┴───────────────────────────────┘
```

“重新定位团队”只能绑定能够识别为同一支有效团队的位置，不能用任意文件夹替换。

- 重新定位失败时保留原记录并说明选择的位置为何不被接受，不把团队错误绑定到新位置。
- “移除记录”只移除应用中已经失效的团队记录，不删除、移动或修改用户磁盘上的任何文件；执行前必须明确说明这一差异。

### 成员 `AGENT.md` 不可用

- 任一成员的 `AGENT.md` 缺失或无法读取，整支团队标记“需要修复”，暂时不能用于新建对话。
- 任一成员缺少稳定 slug，或同一团队内出现重复 slug，也属于“需要修复”。
- 用户可以在 Finder 中恢复文件，或将失效成员从用户团队删除。
- 失效成员是主 Agent 时，可以先指定另一名有效成员为主 Agent，再删除失效成员。
- 文件恢复后重新检查，并在所有成员恢复有效后解除“需要修复”。
- 软件不检查 `AGENT.md` 引用的其他文件是否完整。

### 跨页面异常反馈

- 只要至少一支团队“需要修复”，Sidebar 顶部“Agent 团队”入口显示一个红点；多支异常仍只显示一个。
- 红点的悬停提示和辅助名称为“有 Agent 团队需要修复”。
- 点击入口后，团队首页对应横行必须以“需要修复”状态明确标出具体异常团队。
- 未完成草稿不触发该红点。
- 如果已有会话当前选择的团队变为“需要修复”，会话历史仍可查看，但不能发送新消息；会话内团队按钮显示红色异常标识。
- 会话按钮、发送禁用、切换团队和修复后的恢复细节留给会话交互 PRD 定义。

这里区分两个事实：已有会话继续保留创建时载入的 Agent 内容版本；团队当前是否“需要修复”则实时决定该会话能否继续发送。

## 弹层与危险操作

- “新建团队”和“修改信息”只承载短字段，不把完整成员编辑塞进弹窗。
- 删除用户团队使用明确确认，说明团队文件和已有会话分别受到什么影响。
- 删除通过系统废纸篓或回收站保持文件层可恢复，不提供软件内永久删除；恢复回产品的方式暂不承诺。
- 内置团队不显示删除能力。
- 存在未保存草稿时，离开、删除或其他会丢失内容的操作必须先让用户选择保存或放弃。
- 外部修改冲突不得静默选择任何一方覆盖。

## 响应式与窗口行为

- 宽窗口下，团队横行左侧显示身份信息，右侧显示扁平成员行。
- 窄窗口仍保持“一支团队一条横行”，不改成多列卡片网格；成员行移动到描述下方。
- 首页空间不足时用 `＋N` 收敛成员数量。
- 团队详情中的成员选择器保持单行并允许横向滚动，不换行堆成多层。
- `AGENT.md` 编辑器始终占满成员行下方的可用宽度。
- 窗口缩放、成员切换和横向滚动不得丢失各成员尚未保存的草稿。

窄窗口示意：

```text
┌────────────────────────────────────────────────────────────┐
│ 开发团队                                      内置          │
│ 负责软件开发任务                                           │
│                                                            │
│ [经理 · 主 Agent] [开发] [测试] ＋2                        │
└────────────────────────────────────────────────────────────┘
```

## 关联流程

- 新建对话时预选上一次使用的团队，并允许用户改选。
- 所选团队的主 Agent 接收新会话第一条消息。
- 内置团队复制为用户团队后再修改。
- 团队文件异常后，从 Sidebar 红点或会话异常入口进入修复。
- 已有会话所选团队“需要修复”时，阻止继续发送，修复或切换团队后恢复。

这些流程尚未建立独立的 `flows/` 文档。会话内团队按钮、发送状态、切换团队以及运行中异常的具体行为明确留待会话交互 PRD 讨论。

## 指标与验收

1. 用户能从 Sidebar 顶部进入 Agent 团队首页，并用一条横行识别每支团队的名称、用途、主 Agent、成员和必要状态。
2. 团队成员始终扁平展示；页面不存在暗示组织层级、执行顺序或 DAG 的连接线和节点结构。
3. 内置团队全部只读、不可删除，并能通过“复制并编辑”连同所有成员和相关文件产生独立用户团队；Finder 不能绕过只读限制。
4. 用户能创建长期保留的团队草稿；第一名成员自动成为主 Agent，缺少有效主 Agent 时团队不可用于新建对话。
5. 用户在团队详情内即可切换成员并编辑多个 `AGENT.md`，不需要进入独立 Agent 页面；显示名称和一句话描述来自 `AGENT.md`，未保存草稿不会因成员切换丢失。
6. 用户能用 `@<slug>` 匹配当前团队成员，选择后看到可读名称组件；显示名称变化不破坏底层稳定引用，缺失或重复 slug 会让团队进入“需要修复”。
7. 软件内不提供相关文件编辑器；团队和成员均有不抢占主视觉的 Finder / 文件管理器入口。
8. 新建对话预选上一次成功使用的可用团队；没有历史选择或上次团队不可用时回退到软件自带团队。
9. 团队或成员文件不可用时，团队显示“需要修复”、不能用于新建对话，Sidebar 顶部“Agent 团队”入口出现红点。
10. 已有会话选择了“需要修复”的团队时，历史仍可查看，但发送被阻止；详细恢复交互由会话 PRD 承接。
11. 用户团队删除进入系统废纸篓或回收站，内置团队无法删除，已有会话载入的团队版本不被清除。

## 非目标

本页面明确不提供：

- 可视化工作流、流程节点、成员连线、组织架构或 DAG 编辑器。
- 脱离团队存在的全局 Agent 库。
- Agent 在团队之间移动、链接或共享同一份可变配置。
- 普通成员手动排序；只有主 Agent 固定排在第一位。
- `AGENT.md` 之外的内置文件编辑器、文件树、逐行 diff 或自动合并。
- 对自然语言职责和交棒关系做语义正确性验证，或在删除成员后自动改写其他 Agent 的规则。
- 团队发现、在线市场、远程安装、发布、版本依赖或更新协议。
- 可由用户设置的应用级默认团队。
- 会话内团队切换和异常恢复的完整交互。
- 软件内永久删除或独立的已删除团队管理页。
