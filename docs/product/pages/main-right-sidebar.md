# 主页面右侧栏

> 状态：讨论中草案
>
> 关联入口：[主页面会话区](./main-conversation.md)
>
> 验收对照：[main-right-sidebar.wireframe.html](./main-right-sidebar.wireframe.html)（浏览器打开，实现完成后用它对照视觉结果；需求改动一律先落本文，HTML 随后跟进，冲突时以本文为准）
>
> 范围：定义主页面第三栏——用户按需打开的证据区，承载「这段对话期间项目发生了哪些改动」「某个成员那一步的完整输出」「拆出的子任务」三类内容。主对话区的时间线、上下文选择与状态点由[主页面会话区](./main-conversation.md)定义，项目与对话的组织由[主页面侧边栏](./main-left-sidebar.md)定义。

## 页面目标

主对话区只留结论，右侧栏按需承载证据。

用户想核实「团队到底改了什么」「这个成员那一步究竟做了什么」「拆出去的子任务跑到哪了」时，在产品内就能看到，不必退回命令行或另开编辑器。

离开时用户应当得到的是一个判断：**这段工作可以接受，或者哪里不对**。右侧栏不负责让用户修改任何东西。

产品当前阶段尚未稳定，用户需要监督的不只是结果，还有团队的推进过程本身。右侧栏是这份监督需求的落点——把它集中在一个可以随手关掉的区域，主对话区才能保持安静。

## 用户场景

- 团队说这一步做完了，用户想知道它动了哪几个文件、每个文件改了哪几行，再决定是否继续。
- 某一步没跑起来，主对话区只给了一句人话，用户要读原始错误输出才能判断是环境问题还是团队走错了路。
- 团队把目标拆成几个子任务，卡片上显示其中一个「没跑起来」，用户要点进去看那一条到底怎么了。
- 用户想对照着看：左边是改动后的文件内容，右边是产生这个改动的那一步说了什么。
- 用户已经信任这支团队，什么都不想看——右侧栏保持关闭，界面回到两栏。

## 入口与去向

右侧栏**默认关闭**。开与关的状态跨对话切换和应用重启保留，是用户的工作习惯，不是某段对话的属性。

打开方式共四种，前三种同时决定打开后落在哪个标签上：

1. 主对话区在一轮工作结束时留下的**结果卡片**，点「查看」→ 打开并新建一个改动标签。
2. 主对话区某个成员那一步的**完整输出入口** → 打开并新建该步骤的过程标签。该入口的位置和文案由[主页面会话区](./main-conversation.md)定义，本页只规定点击后的结果。
3. 主对话区**子会话卡片**的某一行 → 打开并新建该子任务的标签。
4. 主内容区右上角的**显示/隐藏按钮** → 打开并恢复上次的标签条。

这段对话第一次打开右侧栏时，落在一个改动标签上。那是最常被要的东西，让用户开局就先做一次类型选择是多余的一步。此后标签条完全由用户自己组装。

关闭右侧栏后主内容区收回全部宽度，会话区回到关闭前的滚动位置。关闭不销毁已打开的标签：同一段对话内再次打开时，标签条恢复原样并停在关闭前选中的标签上。

切换到另一段对话时，右侧栏显示那段对话自己的标签条。标签属于对话，不跨对话携带，但**跨应用重启保留**：用户排查一个反复失败的步骤时开出来的几个标签就是他的工作现场，与未发送的草稿同理，重启后应当还在。逼用户回主对话区往上翻、把入口重新点一遍，恢复成本远高于留住它们的代价。

右侧栏不通往任何其他页面。它不提供在外部编辑器中打开文件的出口（记入「待讨论」）。

## 页面结构

### 默认关闭

```text
┌ 侧边栏 ────┬ 会话区 ────────────────────────────────┬─┐
│ ▾ 项目 A   │ 官网落地页验收                     [▥] │ │
│  » 对话 1  │ ────────────────────────────────────── │ │
│            │ (开) 开发                              │ │
│            │ 已补上空状态验收语句。                 │ │
│            │ ┌────────────────────────────────────┐ │ │
│            │ │ ✓ 已结束 · 2 个文件有改动  [查看] │ │ │
│            │ └────────────────────────────────────┘ │ │
└────────────┴────────────────────────────────────────┴─┘
                                          [▥] 显示/隐藏右侧栏
```

### 打开·改动标签

```text
┌ 侧边栏 ──┬ 会话区 ──────────────┬ 右侧栏 ───────────────────┐
│ ▾ 项目 A │ (开) 开发            │ 改动 × │ 开发 × │ ＋     │
│  » 对话1 │ 已补上空状态验收语句 │ ───────────────────────── │
│          │ ┌──────────────────┐ │ 这段对话期间，项目发生了  │
│          │ │ ✓ 已结束 · 2 个  │ │ 这些改动（项目文件夹）    │
│          │ │   文件    [查看] │ │ ───────────────────────── │
│          │ └──────────────────┘ │ ▾ sites/marketeam         │
│          │                      │     index.html    +12 -3  │
│          │                      │ ▾ tests                   │
│          │                      │   » landing.test.ts  +40  │
│          │                      │ ───────────────────────── │
│          │                      │  37   })                  │
│          │                      │  38 - 空状态没有断言      │
│          │                      │  39 + 空状态断言已补齐    │
│          │                      │  40   })                  │
└──────────┴──────────────────────┴───────────────────────────┘
```

文件树在上、被选中文件的内容在下，同属一个标签。变更行以增／删两种标记区分，未改动行照常显示以提供上下文。

### 改动·团队正在工作

```text
│ 改动 × │ 开发 × │ ＋                                │
│ ─────────────────────────────────────────────────── │
│ 团队正在工作，下面这份列表截至上一轮结束  [刷新]    │
│ ─────────────────────────────────────────────────── │
│ ▾ sites/marketeam                                   │
│     index.html                              +12 -3  │
```

### 改动为空

```text
│ 改动 × │ ＋                                         │
│ ─────────────────────────────────────────────────── │
│ 这段对话还没有开始，团队尚未工作                    │
```

三种空态使用不同措辞：对话还没开始、团队跑过但文件没有变化、内容还在读取中。用户第一次打开右侧栏往往正好是第一种，把它写成结论会让他判定这一栏没用。

### 过程标签

```text
│ 改动 × │ 开发 × │ ＋                                │
│ ─────────────────────────────────────────────────── │
│ 开发 · 这一步的完整输出                  跟随最新   │
│ ─────────────────────────────────────────────────── │
│             ↑ 向上滚动加载更早过程                  │
│ ── 第 1 次执行 · 本轮输入 ──────────────────────── │
│ 你                                                  │
│ 请检查落地页空状态。                                │
│                                                    │
│ 开发经理                                            │
│ 先核对现有实现和验收口径。@dev                      │
│ ── 本轮执行过程 ────────────────────────────────── │
│ 开发                                                │
│ 我先检查相关实现和测试。                            │
│                                                    │
│ ⌘ 读取文件                                          │
│ sites/marketeam/index.html                          │
│                                                    │
│ ⌘ 运行命令                                          │
│ pnpm test -- landing                                │
│ PASS  tests/landing.test.ts                         │
│                              [↓ 3 条新内容 / 到最新] │
```

过程标签第一次打开时定位在最底部；停在底部时跟随正在工作的步骤实时追加，用户向上阅读后暂停跟随，新内容只给「到最新」提示，不抢走阅读位置。向上滚动按需加载更早过程，界面只挂载可视区域附近的内容。同一步骤被重试时，每次执行在同一个标签内依次追加并标明第几次；每次执行都还原它启动时看到的公开时间线，再接这一轮自己的执行过程。

Codex 自己的过程记录文件找不到时，不拿主对话区最后一段回复冒充完整输出：

```text
│ 开发 · 这一步的完整输出                             │
│ ─────────────────────────────────────────────────── │
│ Codex 过程记录文件已不可用                          │
│ 这一步的最终回复仍保留在主对话区。                  │
```

### 子任务标签

```text
│ 改动 × │ 空状态验收 × │ ＋                          │
│ ─────────────────────────────────────────────────── │
│ 空状态验收 · 测试 · 没跑起来                        │
│ ─────────────────────────────────────────────────── │
│ (测) 测试                                           │
│ 正在核对空状态的验收语句…                           │
│ ⚠ 这一步没跑起来（退出码 1）           [重试]       │
│ ─────────────────────────────────────────────────── │
│   ┌───────────────────────────────────────────┐     │
│   │ 说点什么，或 @ 一个成员…               ↑ │     │
│   └───────────────────────────────────────────┘     │
```

子任务标签的输入与操作和主对话区一致，子任务在别处没有说话的地方。

### 空白标签

点加号新开：

```text
│ 改动 × │ 开发 × │ 新标签 × │ ＋ │              │
│ ─────────────────────────────────────────────── │
│                                                 │
│            这个标签要看什么                     │
│                                                 │
│      ┌─────────────────────────────────┐        │
│      │ ⇄  改动                         │        │
│      │    这段对话期间变了什么         │        │
│      ├─────────────────────────────────┤        │
│      │ ▸  项目文件                     │        │
│      │    浏览完整项目树               │        │
│      └─────────────────────────────────┘        │
│                                                 │
│      成员的完整输出和子任务                     │
│      从左边的对话里点开                         │
```

### 标签全部关闭

```text
│ 新标签 × │ ＋ │                                   │
│ ─────────────────────────────────────────────── │
│            这个标签要看什么                     │
│            改动 · 项目文件                      │
```

右侧栏不因为关掉最后一个标签而收起。

### 标签溢出

```text
│ ‹ │ 改动 × │ 开发 × │ 空状态验收 × │ 开发 2 × │ ＋ › │
```

标签条只横向滚动，不折叠成菜单，也不自动关闭旧标签。加号始终可达。

### 窄窗口

```text
┌ 会话区（被覆盖）───────────────────────────────────┐
│ 改动 × │ 开发 × │ ＋                       [×] │
│ ────────────────────────────────────────────────── │
│ ▾ sites/marketeam                                  │
│     index.html                             +12 -3  │
└────────────────────────────────────────────────────┘
```

窗口不足以三栏并排时，右侧栏覆盖会话区，并出现独立的关闭按钮。

## 区域与信息

### 标签条

- 标签条是「用户打开过的东西」的列表，不是固定分类。同类内容可以并存多个标签，用户能一边看某成员的输出、一边看它造成的改动。
- **没有常驻标签，每个标签都可以关闭。** 右侧栏是用户自己组装的工作台，不预设哪一样必须一直在。
- 标签有两种来源：主对话区的点击，以及标签条上的**加号**。
- 加号新开一个**空白标签**，由用户在标签内选择要看什么。空白标签本身也是一个正常标签，可以关闭。
- 主对话区点击产生的标签**去重**：重复点击同一个来源不新建标签，聚焦到已经存在的那一个——那个动作的意思是「带我去看那个东西」。
- 加号产生的标签**不去重**：用户手动开就是明确要一个新的，例如同时开两个改动标签、各自停在不同文件上做对照。
- **关掉最后一个标签不关闭右侧栏**，留下一个空白标签。否则关最后一个标签等于整栏消失，用户会误关而不敢关。
- **过程标签的标题就是成员名**；同一个成员在这段对话里的第二个及以后的过程标签追加序号，例如「开发」「开发 2」「开发 3」。产品无法从第三方 Agent 的输出里取到「这一步在做什么」的可靠描述，不得凭猜测生成描述性标题——标签标题宁可平淡，也不能是编出来的。
- 子任务标签用子任务名，空白标签用「新标签」。标题过长时截断，悬停显示完整标题。

### 空白标签与类型选择

- 空白标签的内容是一句提问和一组类型，用户选完之后这个标签就变成对应类型，标题随之更新。
- 类型只有两种：**改动**（这段对话期间变了什么）和**项目文件**（浏览完整项目树）。它们的共同点是**不需要绑定任何具体对象就能打开**。
- 项目文件夹不是 git 仓库时，类型选择里只有「项目文件」，并说明改动为什么不可用。不静默少一项。
- 过程和子任务**不进类型选择**。它们必须绑定某一个步骤或某一个子任务，让用户在空白页里翻一个长列表去找「开发的第 3 步」是反的——那些东西在主对话区里点，比在这里找快得多。空白标签应当说明这一点，而不是让用户以为这里少了东西。
- **类型选择的边界写死在这两种。** 终端、预览、浏览器不得从这里长出来：加号菜单一旦存在，往里添东西就是顺水推舟，而那会把这一栏推成一个小型编辑器，正是它的非目标。用户核实一次往往要跑测试、起服务，这个需求真实存在，但属于另一次独立裁决，不该由类型选择悄悄承接。

### 改动标签

- 顶部一句话说明**这里显示的是什么**：「这段对话期间，项目发生了这些改动」。
- **基线是这段对话开始时的项目状态，全程累计**。用户的心智是「这段对话把我的项目改成了什么样」，不是「最后这一步改了什么」。这条之所以成立，是因为一段对话的工作空间在发出第一条消息后就锁定了（见[主页面会话区](./main-conversation.md)）——从头到尾在同一个地方工作，基线才有意义。
- **必须说明看的是哪一边**，用词是「项目文件夹」与「独立工作空间」，不出现路径、目录名或其他机器词。选独立工作空间时还要多说一句后果：**这些改动在一份隔离副本里，你的项目文件夹没有被动过**。用户真正需要知道的不是文件在磁盘哪个位置，而是「我切到编辑器为什么看不到」——回答这个问题才算说清楚了。选独立工作空间的恰恰是最谨慎、最需要被说服的那批用户。
- **项目文件夹不是 git 仓库时，不提供改动标签**。这类项目本来就无法使用独立工作空间（见[主页面会话区](./main-conversation.md)），也没有可靠的方式判断「这段对话期间变了什么」；与其给一份可能不准的清单，不如不给。此时空白标签的类型选择里只剩「项目文件」，并说明为什么少了一项。
- **不做改动归因**。产品无法区分某个改动来自团队成员还是用户自己在编辑器里手改的，因此绝不声称「团队改了这些文件」。这不是措辞洁癖：右侧栏存在的唯一理由是建立信任，猜错一次归因，用户对整个区域的信任就没了。
- 只列出有改动的文件，并给出每个文件的增删行数。第一诉求是「改了什么」，铺开整个项目树会把它淹没；要浏览未改动的文件，用「项目文件」类型另开一个标签。
- 文件内容以行为单位区分新增、删除与未改动，用户不需要在脑子里做对比。

### 项目文件标签

- 浏览项目的完整文件树并查看文件内容，包含没有被改动的文件。
- 与改动标签共用同一套文件内容呈现；有改动的文件在这里同样以行为单位标出变化，用户不必为了看一处 diff 再开一个改动标签。
- 同样必须说明看的是项目文件夹还是独立工作空间的副本。

### 过程标签

- 承载**单个成员单个步骤的完整过程**。这里的「完整」不是原样展示 JSONL，而是从 Codex 自己的过程记录中恢复全部用户有意义的事件：本轮启动时看到的公开时间线、Agent 可见 Markdown、命令及结果、工具 / 函数 / MCP 调用及结果、文件操作、错误与诊断。协议噪音、内部 id、token 统计、系统提示词、Agent persona、团队调度规则、附件内部路径和原始 reasoning 不显示。
- 「本轮输入」按主对话区的说话方式还原为用户与各 Agent 一来一回的公开时间线，顺序与正文必须和该次执行启动时一致。不得把 persona、团队规则与公开时间线拼成一个原始 prompt 文本块，也不得只显示最新触发消息。
- 「本轮执行过程」按时间顺序呈现全部受支持事件；Agent 正文沿用安全 Markdown，命令 / 工具 / 文件 / 错误使用可读的结构化事件，不做内容摘要、不截头、不截尾。遇到尚未支持的新事件类型时必须显示「暂不支持的过程事件」及类型，不得静默丢失后让用户以为过程完整。
- **同一步骤被重试时，每次执行在同一个标签内依次追加并标明第几次**，不覆盖上一次。用户靠对照前后两次的报错判断团队是在往前走还是原地打转——这个判断决定他继续放手还是自己上手，是这个标签最有价值的用途之一。
- 每次重试都保留自己启动时的「本轮输入」，因为两次执行看到的公开时间线可能已经不同。
- 内容来自 Codex 自己的过程记录文件，不把 Moebius 临时 stdout tail 或会话最终回复当成完整过程的替代品。关联或文件不存在、重复、损坏或不可读时，明确显示过程记录不可用；主对话区的最终回复仍照常保留，但不在这里降级冒充过程。
- 只读，但**内容必须可以选中和复制**，文件路径同样可以复制。「只读」指不能改，不是不能拿走：用户读懂错误之后的下一个动作永远是复制它——贴给团队、贴到别处、或者拿路径去自己的编辑器里打开。为这个理由把用户推回命令行，比不提供这一栏还糟。

### 子任务标签

- 显示该子任务的名称、负责成员和当前状态，以及它自己的推进内容。
- 子任务标签打开后，主对话区的子会话卡片上对应行标记为当前正在查看。
- **子任务标签的说话方式与主对话区一致**：同样有输入框，同样可以用 `@` 提及成员，同样能中断正在工作的成员。子任务对用户来说就是一段小一号的对话，不该另学一套规矩。
- 子任务的推进操作（重试、停下）同样在子任务标签内直接可用，与主对话区一一对应。这是本页唯一一处提供推进操作的地方——因为主对话区的子会话卡片只给状态，不给操作入口，不在这里给用户就没地方给。
- 子任务标签不提供改动视图。子任务改的文件同样算进这段对话的改动，去改动标签看；单独给子任务算一份改动会立刻带出「父子改动怎么合并」这类没必要的问题。

## 操作与反馈

### 打开与关闭右侧栏

- 显示/隐藏按钮位于主内容区右上角，任何时候可用，包括新对话还没有任何消息时。
- 打开与关闭不影响会话区的滚动位置、草稿和正在进行的推进。
- 状态跨对话切换和应用重启保留。

### 选择文件

- 点击文件树中的文件，在下方显示其内容；当前选中的文件在树中保持可见。
- 切换文件不改变标签，也不新建标签。
- 文件过大或不是文本时，说明无法显示的原因，不静默留白。

### 关闭标签

- 关闭标签只关闭这一个视图，不影响对话本身，也不影响正在进行的推进。
- 关闭子任务标签不取消该子任务。这一点必须让用户看得出来，否则用户会不敢关。

### 内容更新

- 正在工作的步骤，其过程标签实时追加输出。用户停在底部时自动跟随；向上滚离底部后暂停跟随并累计新内容提示，点击「到最新」或手动回到底部才恢复。
- 过程标签首次打开定位最底部，向上滚动按需加载更早事件；加载前页不得让当前阅读内容跳位。切换标签、关闭再打开或应用重启后恢复该标签最后一次成功保存的阅读位置；从主对话区再次点击同一来源只聚焦已有标签，不重置位置。
- 改动标签在一轮工作结束时刷新。**团队正在工作期间必须说明「这份列表截至上一轮结束」并提供手动刷新**，否则用户开着它盯着看，只会看到一个纹丝不动的列表，第一反应是它坏了或者团队根本没动文件。
- 改动标签不做实时跟随。文件在一步的中途处于半成品状态，实时刷新会让列表反复抖动，也会把注意力从会话区拽走。
- 用户正在阅读某个文件时刷新，不得跳走或丢失当前阅读位置；有新的改动时给出可点击的提示，由用户决定何时看。
- 用户在别的标签上时，右侧栏不因为有新内容而自动切换标签。抢焦点会让用户不敢把它开着。

## 页面状态

| 状态 | 表现 | 可用操作 |
| --- | --- | --- |
| 关闭 | 不占宽度，右上角按钮可用 | 打开 |
| 空白标签 | 一句提问和两种类型 | 选类型、加号、关闭标签 |
| 只剩空白标签 | 右侧栏保持打开，不自动收起 | 选类型、加号、关闭右侧栏 |
| 改动·对话还没开始 | 说明团队尚未工作，不说成「没有改动」 | 加号、关闭标签 |
| 改动·跑过但没有改动 | 说明这段对话期间文件没有变化 | 加号、关闭标签 |
| 改动·有改动 | 文件树 + 选中文件的行级对比 | 选文件、加号、关闭标签 |
| 改动·团队正在工作 | 说明列表截至上一轮结束，提供手动刷新 | 刷新、选文件、关闭标签 |
| 改动·加载中 | 保持结构的加载占位，不说成空态 | 等待、关闭标签 |
| 项目文件 | 完整项目树 + 选中文件的内容 | 选文件、加号、关闭标签 |
| 改动·读取失败 | 说明原因并提供重试，不显示半份结果 | 重试、关闭 |
| 改动·项目文件夹不可用 | 说明文件夹已不可用、当前无法读取改动 | 关闭；修复入口在会话区 |
| 改动·项目不是 git 仓库 | 不提供此标签，类型选择里说明原因 | 改开项目文件标签 |
| 过程·进行中 | 输出实时追加 | 阅读、关闭标签 |
| 过程·已结束 | 内容定格 | 阅读、关闭标签 |
| 过程·内容为空 | 说明这一步没有产生输出 | 关闭标签 |
| 过程·记录不可用 | 说明 Codex 过程记录文件找不到、重复、损坏或不可读，不显示 fallback 冒充完整过程 | 关闭标签 |
| 过程·离底阅读 | 保持当前位置，显示新内容数量与「到最新」 | 继续阅读、到最新、关闭标签 |
| 子任务·进行中 | 显示子任务内容与当前状态 | 说话、提及成员、停下、关闭标签 |
| 子任务·没跑起来 | 与主对话区使用同一套说法 | 重试、说话、提及成员、关闭标签 |

三种不可继续状态（项目文件夹不可用、团队已删除、团队需要修复）由[主页面会话区](./main-conversation.md)定义并承担恢复入口。右侧栏在这些状态下只说明自己读不到什么，不重复提供恢复按钮，避免出现两个事实源。

窄窗口下右侧栏覆盖会话区时，恢复入口正好被自己盖住。此时右侧栏必须提供一个回到会话区的方式，而不是让用户自己想到「先关掉我」——项目文件夹失效通常发生在用户刚挪过目录之后，而挪完目录第一件事就是打开应用看看还认不认得。

## 弹层与危险操作

右侧栏**不改变任何文件**，不提供任何弹层与危险操作。

- 不提供编辑、保存、撤销、回滚、丢弃改动。
- 不提供删除文件、删除标签以外的删除动作。
- 不提供任何 git 动作（提交、推送、切分支、创建分支）。
- 内容可以选中、复制。不改变东西不等于不能拿走。

改动、项目文件、过程三类标签**全部只读**。用户对主对话区某一步的干预（重试、停下、说话、提及成员）留在主对话区，右侧栏不重复提供。

**子任务标签是例外**，它承载子任务的对话推进（输入、提及成员、重试、停下）。这不是破例：主对话区的子会话卡片只给状态、不给操作入口，子任务在别处没有可说话的地方，不在这里给就没地方给。子任务标签改变的是那段子对话的推进，仍然不改变任何文件。
- 界面不得出现暗示「可以把改动还原回去」的控件或文案。产品当前阶段只解决「看得见」；「可退」是尚未裁决的更大问题，见「待讨论」。这与[主页面会话区](./main-conversation.md)对「停下」的规定一致——已经产生的改动不会被回滚，界面不得暗示可以撤销。

## 响应式与窗口行为

- 三栏并排时，右侧栏可由用户拖拽调整宽度，宽度跨对话与重启保留。
- 宽度不足以三栏并排时，右侧栏覆盖会话区并提供独立关闭按钮；关闭后回到会话区原滚动位置。覆盖期间若出现只能在会话区处理的状态（三种不可继续状态），右侧栏必须给出回到会话区的方式。
- 文件树、文件内容、过程输出各自独立滚动，互不牵动。
- 过程输出使用虚拟化的时间顺序列表：首次只呈现最底部可视窗口，向上滚动加载更早页；无论过程记录多大，DOM 节点数都只随可视窗口与缓冲区增长，不随完整文件行数线性增长。
- 长行不换行，横向滚动，不撑宽整个页面。行号列在横向滚动时保持可见。
- 侧边栏关闭时右侧栏行为不变，两者互不影响。

## 关联流程

以下流程将在对应流程 PRD 建立后链接，本文档只记录右侧栏的职责：

- 团队接力推进一个目标直到结束——右侧栏提供用户在每一轮结束时核实结果的证据。
- 从没跑起来、卡住或停下的状态恢复——右侧栏提供判断原因所需的完整过程与错误细节，恢复动作本身在会话区。
- 团队拆出子任务并各自推进——右侧栏是查看单个子任务的唯一位置。

## 指标与验收

右侧栏服务的产品指标是**任务成功率**与**非预期人工干预率**：用户不必为了确认「改了哪些文件、这一步报了什么错」而退回命令行。

指标口径到此为止，不扩大到「用户完全不必离开产品」。用户的核实动作本来就不止读改动——跑测试、起服务、亲眼看一次页面，这些都在右侧栏的能力之外，也不应该成为它的目标。把「不退回命令行」整个扣在这一栏头上，只会诱导后来的人往里塞终端和预览，最后变成一个小型编辑器，正好是它的非目标。

可核查的完成标准：

1. 右侧栏默认关闭，界面为两栏；开关状态在切换对话与重启应用后保持。
2. 标签条在切换对话时随对话切换，并在重启应用后恢复。
3. 一轮工作结束后，主对话区的结果卡片能一步打开改动标签。
4. 改动标签列出的是相对这段对话开始时的累计改动，不是最后一步的改动。
5. 改动标签的说明文案不声称改动来自团队成员。
6. 改动标签说明当前看的是项目文件夹还是独立工作空间的副本。
7. 项目文件夹不是 git 仓库时不提供改动标签，类型选择只剩「项目文件」并说明原因。
8. 文件内容能以行为单位分辨新增、删除与未改动。
9. 改动标签只列改动文件；浏览未改动文件通过「项目文件」类型另开标签完成。
10. 团队正在工作时，改动标签说明列表截至上一轮结束并提供手动刷新。
11. 「对话还没开始」「跑过但没有改动」「还在读取」三种情况的措辞互不相同。
12. 点击某成员那一步的完整输出入口，新建一个过程标签；它从 Codex 过程记录恢复本轮输入的完整公开时间线，以及 Agent Markdown、命令、工具、文件、错误等全部用户有意义的执行事件，不展示原始 JSON、系统提示词、persona、团队规则、token 统计或原始 reasoning。
13. 过程标签的标题为成员名，同一成员的第二个及以后的标签带序号，标题中不出现产品自行编写的步骤描述。
14. 同一步骤重试后，多次执行在同一标签内依次可见并标明第几次；每次执行保留自己启动时看到的公开时间线。
15. 重复点击主对话区同一来源不产生第二个标签，而是聚焦已有标签；通过加号可以开出多个同类型标签，不去重。
16. 每个标签都可以关闭，关闭后不影响对话与推进；关掉最后一个标签时右侧栏保持打开并留下一个空白标签。
17. 标签条提供加号，新开的空白标签只给「改动」和「项目文件」两种类型，不出现终端、预览或浏览器。
18. 空白标签说明成员完整输出与子任务从主对话区点开，而不是让用户以为类型选择里少了东西。
19. 有新内容时不自动切换用户当前所在的标签。
20. 用户正在阅读文件时发生刷新，阅读位置不丢失。
21. 全区找不到任何编辑、保存、撤销、回滚或 git 操作；改动、项目文件、过程三类标签只读，子任务标签提供与主对话区一致的输入与推进操作。
22. 文件内容、过程输出和文件路径都可以选中复制。
23. 窄窗口下右侧栏覆盖会话区并可独立关闭，关闭后会话区回到原滚动位置。
24. 过程标签首次打开位于最底部；停在底部时跟随新内容，向上阅读后暂停跟随并显示新内容数量，回到底部后恢复；加载更早事件、切换标签与重开标签都不丢阅读位置。
25. 大型 Codex 过程记录通过反向分页与虚拟列表从头可读到尾，DOM 规模保持有界；未支持的新事件有可见占位，不被静默丢弃。
26. Codex 过程记录关联或文件不存在、重复、损坏或不可读时，过程标签明确显示不可用，不使用 Moebius stdout tail、最终 Agent 回复或按时间猜出的其他记录冒充。

## 已知隐患

以下问题已经识别，本版**接受它带来的后果**，不作为待讨论挂起，也不因此推迟交付：

### Codex 过程记录可能不可用

过程标签读取 Codex 自己维护的过程记录，不复制一份内容归 Moebius 保管。用户清理 Codex 数据、切换 Codex 数据根、文件损坏或未来 Codex 改变记录格式时，历史步骤可能无法读取。本版接受这个后果：页面明确说明「Codex 过程记录文件已不可用」，不拿残缺 tail、最终回复或按时间猜出的另一份文件冒充。

这项取舍换来的是单一内容来源：Codex 拥有过程记录，Moebius 只保存一次运行到 Codex thread 的稳定关联。

## 待讨论

以下问题本版**明确不作答**，不得由实现自行决定：

### 改动的「可退」

本页只解决「看得见」。以下四件事仍然悬空，需要一次完整裁决：按「停下」之后已经产生的半截改动怎么办；用户看到不该有的改动时能否在产品内还原；选择独立工作空间的用户如何把成果带回项目；默认工作空间直接改真实文件夹这一默认值是否需要首次提醒。这组问题跨越会话区、右侧栏和项目设置，可能需要一份流程 PRD 承接。

### 只看这一轮的改动

改动标签目前只有「全程累计」一个视角。长对话里逐轮验收时，用户每一轮都要重新扫同一棵树才能找出这轮新增的部分，返工越多越费劲。是否增加「本轮／全程」切换未裁决。**本版明确不做**：同类产品（编辑器的源代码管理面板、Codex 的评审视图）同样只给全量对比，用户对这套心智是熟的，不构成使用障碍。

### 改动规模很大时的表现

一次涉及几百个文件、或单个文件改动上万行时，改动标签如何呈现与截断，未讨论。日常的一次依赖安装就足以让锁文件和构建产物占满列表，是否需要排序、折叠或过滤，同样未裁决——本版按原样照列。

### 在外部编辑器中打开

目标用户有自己的编辑器。是否提供「在编辑器中打开这个文件」的出口，以及它是否会削弱右侧栏本身的存在理由，未裁决。

### 归属其他页面的衔接问题

**「非预期人工干预」的定义边界**。用户为了跑测试、起服务、看一眼真实页面而离开产品，算不算非预期干预，属于总 PRD 的指标定义问题，不由本页承担。本页只把自己的指标口径收窄到「确认改了什么、报了什么错」。

## 现状参考与产品缺口

右侧栏容器、过程标签与 Codex rollout 阅读链路已经落地。新 run 在收到 `thread.started` 时把 `runId → Codex thread` 稳定关联追加进 Moebius session JSONL；过程标签据此直接读取 Codex rollout，恢复公开时间线、投影友好事件，并通过反向分页、活动增量和虚拟列表从末尾持续向上阅读。升级前没有稳定关联的旧 run 不做猜测回填，打开时明确显示过程记录不可用。改动 / 项目文件 / 子任务标签仍沿用各自独立的数据源与交互。

## 非目标

- 不做代码编辑器。右侧栏不提供编辑、保存与格式化。
- 不做终端、预览与浏览器。加号的类型选择只有改动与项目文件两种，这条边界写死；跑测试、起服务、看真实页面的需求属于另一次独立裁决，不从这里长出来。
- 不做版本控制客户端。不提供提交、推送、分支、暂存或历史浏览。
- 不做改动归因。不声称某个改动由哪个成员造成。
- 不做撤销与回滚。本版只解决看得见。
- 不做跨对话的改动汇总。右侧栏只对当前这段对话负责。
- 不做 Codex rollout 的第二份内容存储。Moebius 只保存稳定关联，记录缺失时如实显示不可用。
- 不把原始 Codex JSONL、系统提示词、Agent persona、团队调度规则、token 统计或原始 reasoning 暴露给用户。
- 不承载主对话的推进操作。重试、停下、说话、提及成员留在会话区；例外是子任务标签，子任务在别处没有说话的地方。
- 不承载子会话的管理。子任务不进入侧边栏，也不在右侧栏被创建、重命名或删除。
- 不服务「离开很久后回来快速读懂全部经过」的场景，界面也不暗示自己能做到。
