## 页面目标

首启引导不是欢迎，是一次让新用户「信这支团队会替我接力」的最短演示——从环境就绪、看见团队、看它怎么协作，到确认准备开工。

走完引导，用户带走的不是「配置完成」的绿勾，而是**「这支团队会替我接力，我不必当中转站」**的认知。

引导每台电脑只出现一次；走完最后一步落进[主页面会话区](./main-conversation.md)的新建对话形态，此后再打开应用不再重复。

## 用户场景

- 用户下载 agent-moebius 并首次打开，此前用过 codex，还没试过多 Agent 团队协作。
- 用户在同一台电脑上曾完整走完引导，重开应用应直接进主页面，不重复。
- codex 未装或不可用时，用户在第一步就得到具体的下一步指引，而不是退回终端猜为什么装了 App 也开不了工。
- 用户走完引导后自然落入新建对话页，此时项目仍为空——加项目由那页承接，不是引导的职责。

## 入口与去向

### 入口

- 应用首次启动，检测到本机**从未完成过引导**时自动进入。判据待讨论（见「待讨论」）。
- 引导期间侧边栏不出现，主内容区就是引导本身。
- 中途关闭应用后重开，回到引导——具体回到哪一步待讨论。

### 去向

- 走完第 4 步点「开始使用」，落进[主页面会话区](./main-conversation.md)的新建对话形态。
- 落到新建对话页时项目仍为空，由该页承接添加项目。
- 落到新建对话页时**引导中所选定的团队应等同于「上一次成功创建会话时使用的团队」**（见 [agent-teams.md:361](./agent-teams.md)），作为该对话的默认预选团队。

## 页面结构

引导为一屏一步的全屏页，共四步；应用启动即引导，本身不占屏而是一次跳转。

│  │ 产品发布团队                                      │    │

│  │ 研究员 · 内容作者 · 品牌审校                      │    │
│  │                                                    │    │

│  └────────────────────────────────────────────────────┘    │
│                                                            │

- 首问固定聚焦工作目标：「你希望这支团队长期替你完成什么工作？」用户不需要先提供成员名称或组织结构。
- 缺少会改变成员构成的关键信息时，AI 每轮只追问一个问题；信息足够时直接给团队方案，不为显得周到而继续采访。
- 方案必须包含团队名称、一句话用途、2–6 名成员、唯一主 Agent、每名成员的稳定 slug 与可生成完整 `AGENT.md` 的结构化职责 / 交棒规格，以及一段供第 3 步播放的接力示例。
- 用户可以继续用自然语言调整方案。只有点击「创建并选中」才创建正式用户团队；此前的对话和方案不是团队，也不出现在 Agent 团队首页。
- 「返回选团队」只退出第 2 步的建队子流程，不是四步引导的全局回退；再次进入时恢复尚未确认的对话与最后一版有效方案。
- 创建成功后回到第 2 步团队列表，新团队成为当前选中项；随后「继续」进入第 3 步。

## 区域与信息

### 顶部标题区（每屏）

- 主标题 22px semibold，居中，说这一步在干什么。
- 副本 13px sub 色，居中，一句话说清动机；第 4 步的副本可省。

### 主体区（每屏）

- 单栏居中，最大宽度约 `max-w-lg`（≈512px）。
- 第 1 步是检查行组；第 2 步是团队卡组；第 3 步是动画容器；第 4 步是大 ✓。

### 底部操作条（每屏）

- 顶部一条 `border-t border-line` 与主体隔开。
- 左侧步骤点：四个圆点，当前步为 `bg-ink` 实心，其余为 `bg-line-strong`；右侧配 tnum 文字 `n / 4`。
- 右侧主 CTA：「继续」或第 4 步的「开始使用」，`Button size="lg" variant="default"`。
- 第 1 步 codex 缺失态额外显示 `Button variant="outline" size="lg"` 的「重新检查」，位于主 CTA 左侧。

## 操作与反馈

### 前进

- 每步主 CTA 前进到下一步；第 4 步主 CTA 出引导，落到新建对话页。
- 上一步的成果携带到下一步：第 1 步的通过状态、第 2 步的所选团队都对下一步继续有效。
- 高保真单 HTML 中的四步前进和第 4 步「开始使用」全部在同一文件内切换可见状态；最后一步必须直接渲染出新建对话形态并让所选团队肉眼可见，不调用正式应用路由，不以提示框或完成文案代替去向。

### 回退

- 四个主步骤之间不提供回退。第 2 步 AI 建队子流程提供「返回选团队」，只回到同一步的团队选择状态，并保留未确认的建队对话与方案。

### 中途关闭

应用关闭 = 中断引导。下次打开回到引导的哪一步（从头 vs 从中断处继续），**待讨论**。

### 第 1 步复制安装命令

- codex 缺失时给出的安装命令是可复制的一整行。
- 用户在自己终端粘贴并执行，回到应用点「重新检查」。
- 「复制」按钮成功后的短暂反馈形态待细化。

### 第 2 步 AI 建队

- 每个建队草稿拥有独立的 Codex thread；第一轮使用 `codex exec`，后续调整使用明确 thread id 的 `codex exec resume`。
- 页面展示的对话、最后一版有效方案和状态由应用自身保存；Codex thread 只提供模型上下文，不是产品状态唯一事实源。thread 丢失时，应用可以用已保存对话重建一次，不得因此丢失用户输入或已经验证的方案。
- 一轮运行中输入框和「创建并选中」不可重复提交；用户可以退出子流程，运行结果返回时只有仍匹配当前草稿和 turn revision 才能提交，过期结果必须丢弃。
- Codex 超时、进程失败、resume 失败或输出非法时，页面保留已有对话和最后一版有效方案，显示可重试错误；不得无限自动重试，也不得把失败推进为已创建。
- 「创建并选中」只接受当前显示且仍匹配 proposal revision 的有效方案。创建期间按钮进入单次进行中状态；失败后回到同一方案并提供重试。
- 创建成功后不得留下未完成团队；失败、取消或应用崩溃恢复时，团队首页不得出现只有部分成员的 AI 团队。

### 第 3 步重播与继续

- 进入第 3 步后演示自动播放一次；「重新播放」会从第一棒重新开始，不改变已选团队。
- 动画播放期间「继续」始终可用。用户继续后立即结束演示并进入第 4 步，不等待剩余动画。
- 动画播完后停在主 Agent 收尾的完成画面，不自动跳到第 4 步，避免用户还没看清就被带走。

## AI 建队技术约束

AI 建队采用**短提示契约 + 结构化输出 + 确定性校验 + 应用状态机**。专业 LLM 可以补全角色内容、职责和协作方式，但不得自行猜测输出协议、权限、状态迁移或提交时机。

### 最小提示契约

固定 developer instructions 只保留模型在生成时必须知道的语义：

```text
你是 agent-moebius 的团队设计器，只负责把用户目标转成团队方案。
缺少会改变成员构成的关键信息时，只追问一个问题；信息足够时直接给方案。
方案必须包含 2–6 名成员、唯一主 Agent、唯一稳定 slug、每名成员的完整职责与交棒规格，以及一段可播放的接力示例。
不得读写文件、运行命令或声称团队已经创建。
只返回指定 schema；phase 只能是 clarifying 或 proposal。
```

不会影响生成内容的约束不继续堆进提示词：

- `phase`、字段类型、必填项与数组形状由 Codex `--output-schema` 约束。
- 成员数量、slug 唯一性、主 Agent 引用、接力成员引用、结构化成员字段和应用渲染后的 `AGENT.md` frontmatter 由 TypeScript validator 判定。
- `idle → running → clarifying / proposal / failed → committing → selected` 由应用状态机判定。
- 超时、重试、thread 重建、过期响应丢弃和 proposal revision 由 service 层判定。
- 团队目录与记录的原子创建由 team store 判定；Codex 永远不获得团队写权限。

同一条会影响生成成功率的关键规则，提示词负责告诉模型怎么做，schema 负责限制输出形状，validator 负责判断是否真的做到；不得只依赖 validator 反复拒绝模型猜测，从而把流程拖入无界重试。

### Codex 执行边界

- AI 建队复用受控 Codex driver，但使用独立 execution profile，禁止直接复用包含 `--yolo` 的普通 Agent 执行参数。
- execution profile 使用只读 sandbox、隔离且不含项目素材的工作目录、显式 output schema 与有界 idle / max-duration timeout；不加载项目 `AGENTS.md`、用户 MCP 或会改变团队设计器行为的个人指令。
- 固定角色约束使用 developer instructions；本轮用户输入只作为 user prompt。不得使用 `model_instructions_file` 替换 Codex 自带基础指令。
- Codex 只返回 `clarifying` 或 `proposal` 两类结构化结果。应用在 schema 之外再次做业务校验；校验失败最多执行一次带明确错误列表的修复 turn，仍失败则进入可见、可重试的失败状态。
- Codex 不直接创建目录、写 `team.json`、写 `AGENT.md`、登记团队或改变 onboarding 选择。用户确认后，应用把最后一版有效方案写入临时团队目录，完整重读校验后再切换为正式用户团队并登记。

| 第 2 步 · AI 团队已选中 | 回到团队列表，新用户团队呈选中态 | 是 |
| 第 3 步 · 动画播放 | 所选团队成员按接力顺序逐步高亮，当前交棒方向可见，已完成步骤保留极简记录 | 是 |
| 第 3 步 · 动画完成 | 停在测试复核通过、主 Agent 带证据收尾的完成画面，提供「重新播放」 | 是 |
| 第 3 步 · 减少动态效果 | 无持续位移，以逐步淡入和静态高亮表达同一接力顺序 | 是 |
| 第 4 步 · 准备就绪 | 大 ✓ + 单一主按钮 | 是 |

## 指标与验收

### 产品指标

- **完成引导 / 打开应用** 的比例（不完成就强关的比例）。
- 从打开 App 到落到新建对话页的中位时长。
- 第 1 步 codex 缺失态触发率（决定 App 下载页是否需要更明确提示先装 codex）。
- 第 2 步进入 AI 建队、看到有效方案、确认创建和中途退出的漏斗。
- AI 建队每轮超时率、非法输出率、一次修复成功率与可见重试成功率。
- 引导完成后 24h 内实际发出第一条对话消息的用户比例。

### 验收标准

1. 本机从未完成引导时打开 App，主窗口进入第 1 步而不是新建对话页。
2. 本机曾完成引导时打开 App，直接进新建对话页，不再看到引导。
3. 第 1 步 codex 缺失时「继续」按钮 disabled；「重新检查」通过后按钮恢复可点。
4. 第 1 步检查项**只查 codex**（是否安装、是否可运行），不检查 gh / GitHub 相关工具，也不检查 claude code 或其他 CLI。
5. 第 2 步默认选中内置开发团队；选中状态由 accent 描边 + sel 底 + 蓝色实心圆点表达。
6. 第 2 步「跟 AI 聊出一支新团队」在同一步内展开，不跳路由；首问聚焦长期工作目标，AI 每轮最多追问一个会改变成员构成的问题，用户可退出并恢复未确认草稿。
7. AI 方案包含 2–6 名成员、唯一主 Agent、唯一稳定 slug、每名成员可生成完整 `AGENT.md` 的结构化职责 / 交棒规格与接力示例；未确认前不得创建团队，确认后必须整支团队原子可用并成为当前选中项。
8. 第 3 步是引导的必经步；必须用约 10 秒的连续动画演示第 2 步所选团队从主 Agent 派工、专业成员执行、发现问题、修正、再次复核到主 Agent 带证据收尾的接力过程；AI 团队使用其已验证方案中的角色与接力示例，不得退回开发团队硬编码或静态占位。
9. 第 4 步只有一个主 CTA「开始使用」，点击落到新建对话页且已带上第 2 步所选团队作为该对话的默认预选团队。
10. 底部步骤点 1..4 与右侧文字 `n / 4` 同步；当前步为 ink 实心，其余为 line-strong。
11. 所有屏的顶部标题区、底部操作条布局一致；主体最大宽度 `max-w-lg`。
12. 引导期间侧边栏不出现，主内容区就是引导本身；引导完成后恢复主页面三栏布局。
13. 引导中不出现任何 GitHub / gh / PR / issue 相关字样。
14. 所有色值走 [DESIGN.md](../../../packages/console-ui/DESIGN.md) 的令牌（`bg-canvas` `text-ink` `text-pass` 等），无裸 hex。
15. 亮暗双主题下所有屏可读、按钮状态可区分。
16. 第 3 步支持重新播放，播放期间也可继续；减少动态效果模式保留等价信息但不做持续位移。
17. 高保真原型是可离线直接打开的单个 HTML，所有前进操作都在同一文件内切换状态，能够从第 1 步完整走到带有所选团队的新建对话状态，不依赖外部脚本、样式、字体、图片或应用路由；AI 建队在原型中使用确定性 mock，不调用真实 Codex；原型与正式产品代码双向不共享源码。
18. AI 建队使用独立 Codex thread、developer instructions、output schema 和独立只读 execution profile；普通 Agent 的 `--yolo` 配置不得泄漏到建队调用。
19. 非法结构、超时、resume 失败和创建失败都保留用户对话及最后有效方案、有限结束并提供重试；不得无限自动重试、静默丢草稿或创建半成品团队。
20. Codex thread、原始 JSONL、schema 路径、运行目录和内部错误不得出现在 renderer 可见 DTO；页面只接收安全消息、方案、状态和可执行恢复动作。

## 现状参考与产品缺口

现有实现里没有独立的欢迎 / 引导界面——冷启动直接渲染三栏操作台的新建对话页，项目为空时只有一行小字「还没有项目，从上面的项目按钮添加一个」；环境自检（codex/gh）完全不阻断、也不在主窗口显示。

相较本文定义，当前实现的主要缺口：

- **无引导页**：无任何一步；主窗口直接进新建对话页。
- **环境自检沉默**：[`desktop/src/env-doctor.ts`](../../../desktop/src/env-doctor.ts) 检查 codex / gh / gh auth，结果只进独立诊断页；主窗口对 codex 缺失零提示，用户要发消息才以失败 run 暴露。
- **多余的 gh 检查**：`env-doctor.ts` 现同时查 `gh --version` 和 `gh auth status`；按本文与项目决定（本项目 desktop-only，不做 GitHub 侧执行），这两项应清除，只保留 codex 相关。
- **`isFirstRunOnboarding` 判据不稳**：[`sidebar-preference.ts:31-35`](../../../desktop/src/console-page/sidebar-preference.ts) 用「projects 空」当首次，加删项目会反复触发；本文需要稳定的「本机曾完成引导」判据。
- **没有 AI 建团队路径**：Agent 团队页现有空白创建与「复制并编辑」，但没有从工作目标生成完整团队的能力；本文定义的「跟 AI 聊出一支新团队」是新能力。
- **新建对话页缺「已带上引导中选定团队」的传参链路**：现在冷启动预选团队走 [agent-teams.md:367](./agent-teams.md) 「首次预选软件自带第一支」，本文需要它变成「引导中选定的团队」。

## 非目标

本引导明确不做：

- codex 之外的任何 CLI 检查（gh / claude code / node 等）。
- 让用户在引导里编辑内置团队的成员或 `AGENT.md`（属于 [Agent 团队](./agent-teams.md)页的职责）。
- 添加第一个项目（属于[主页面会话区](./main-conversation.md)新建对话形态的职责）。
- 授权 / 隐私声明 / 用户协议弹层（若产品需要，另议）。
- 教用户完整用法——那是官网 / 文档的职责，引导只做「让他信这支团队会接力」这一件事。
- 演示通用的「什么是多 Agent」抽象概念——第 3 步演的是用户刚选的**具体这一支团队**，不做概念铺垫。
