# 主页面会话区

> 状态：讨论中草案
>
> 所属页面：主页面
>
> 范围：定义主内容区的会话页——包括还没有消息时的新对话形态、上下文选择、时间线、用户与团队的往来、子会话入口和异常表现。[主页面侧边栏](./main-left-sidebar.md)、[主页面右侧栏](./main-right-sidebar.md)、[Agent 团队](./agent-teams.md)、[全局搜索](./search.md)、[设置](./settings.md)分别定义。
>
> 产品基线：[agent-moebius 产品需求文档](../prd.md)

## 页面目标

会话页是用户和一支 Agent 团队的唯一工作面。用户在这里描述目标、看着团队推进、在需要时插话或改变方向、确认结果，直到主 Agent 结合团队完整时间线给出本轮可见收尾。

「新建对话」不是一个动作，而是这个页面还没有任何消息时的样子。产品不设独立的新建对话窗口：新对话页和有历史的会话页是同一个页面的两种状态，区别只在于有没有消息，以及进入时是否已经带上了项目。

会话页只回答三个问题：

1. 这段对话在哪个项目、用哪支团队、以什么方式动我的文件？
2. 团队现在做到哪了，说了什么？
3. 我想改变方向时，话该说给谁？

会话页不是运行日志、任务看板或工作流图。用户不需要理解内部编排结构，就能读懂团队在做什么。

## 用户场景

- 用户点击侧边栏「新建对话」，还没想好放在哪个项目，先在页面上选。
- 用户从某个项目行进入，项目已经选好，直接描述目标。
- 用户还没有任何项目，需要在这个页面里添加第一个。
- 用户在团队推进途中想补一句话或改变方向。
- 用户想让某个正在工作的成员停下来换个做法。
- 用户想看看团队拆出来的某个子任务进行到哪了，看完回到主线。
- 用户遇到一步没跑起来，想重试或换个人做。
- 用户想中途换一支团队。

## 入口与去向

### 入口

- 从[主页面侧边栏](./main-left-sidebar.md)顶部「新建对话」进入，**不预选项目**。
- 从侧边栏项目行的新建入口进入，带上该项目。
- 点击侧边栏中任意一段已有对话进入。
- 从[全局搜索](./search.md)的结果跳转进入。

从顶部进入时不替用户预选项目：项目在发出第一条消息后就永久锁定，替用户猜一个会让一次没看清的点击变成一段归属错误且无法纠正的对话。团队可以预选，因为它随时能改。

进入新对话页时，侧边栏顶部「新建对话」呈现选中态，项目列表中不新增任何行。

### 去向

- 发出第一条消息后，这段对话才被创建，侧边栏才出现对应的行。
- 点击时间线中的子会话卡片，在[主页面右侧栏](./main-right-sidebar.md)打开该子任务。
- 团队按钮通往[Agent 团队](./agent-teams.md)的修复或管理入口。
- 切换到其他对话或页面时，未发送的草稿保留。

## 页面结构

### 新对话（从顶部进入，未选项目）

```text
┌ 侧边栏 ────────┬──────────────────────────────────────────────────────┐
│ ＋ 新建对话  » │  新对话                                              │
│ ⌕ 搜索         │                                                      │
│ ◇ Agent 团队   │                                                      │
├────────────────┤            描述你的目标，团队会开始推进              │
│ 项目           │                                                      │
│ ▾ 项目 A    ＋ │                                                      │
│    对话 1      │   ┌──────────────────────────────────────────────┐   │
│ ▸ 项目 B    ＋ │   │ [▱ 选择项目 ▾] [◇ 开发团队 ▾]                │   │
│                │   │ 描述你的目标…                             ⊘ │   │
├────────────────┤   └──────────────────────────────────────────────┘   │
│ ⚙ 设置         │     选择一个项目后才能发送                           │
└────────────────┴──────────────────────────────────────────────────────┘
```

项目未选定时，工作空间和分支不显示，发送禁用，草稿照常可以输入。输入框下方常驻显示禁用原因，不把原因藏在禁用按钮的悬停提示中。

### 新对话（从项目行进入，项目已带上）

```text
│   ┌──────────────────────────────────────────────┐   │
│   │ [▱ 项目 A ▾] [默认工作空间 ⇄] [main]         │   │
│   │ [◇ 开发团队 ▾]                               │   │
│   │ 描述你的目标…                             ↑ │   │
│   └──────────────────────────────────────────────┘   │
```

### 项目按钮展开

添加项目发生在这里，不在侧边栏。

```text
   ┌──────────────────────┐
   │ ✓ 项目 A             │
   │   项目 B             │
   │ ──────────────────── │
   │ ＋ 添加项目…         │
   └──────────────────────┘
   ┌──────────────────────────────────────────────┐
   │ [▱ 项目 A ▴] [默认工作空间 ⇄] [main]         │
   │ 未发送的输入草稿                          ↑ │
   └──────────────────────────────────────────────┘
```

### 一个项目都没有时

```text
│   ┌──────────────────────────────────────────────┐   │
│   │ [▱ 选择项目 ▾] [◇ 开发团队 ▾]                │   │
│   │ 描述你的目标…                             ⊘ │   │
│   └──────────────────────────────────────────────┘   │
│     还没有项目，从上面的项目按钮添加一个              │
```

此状态与项目未选定使用同一版式，只根据是否存在可用项目改变禁用原因，不另设一套空态页面。

### 团队推进中

```text
│ 官网落地页验收                                              │
│ ─────────────────────────────────────────────────────────── │
│ 你                                                          │
│ 帮我把落地页的空状态补上验收语句                            │
│                                                             │
│ (经) 开发经理                                               │
│ 这个改动只碰落地页，@开发 来做。                            │
│                                                             │
│ (开) 开发                                                   │
│ 运行 pnpm test -- landing                                   │
│                              [完整输出 →]      [停下]       │
│ ─────────────────────────────────────────────────────────── │
│   ┌──────────────────────────────────────────────────────┐  │
│   │ ▱ 项目 A  默认工作空间  main  [◇ 开发团队 ▾]         │  │
│   │ 说点什么，或 @ 一个成员…                          ↑ │  │
│   └──────────────────────────────────────────────────────┘  │
```

成员这一行只显示它当前最新的一句，下一句到来时原地替换。「完整输出」把这一步的全部输出送进[主页面右侧栏](./main-right-sidebar.md)的过程标签。

已有会话的标题使用与窗口控制区相同的 46px header 高度和不透明 sticky 行，放在时间线滚动容器内并吸附到主内容区顶部；标题文字与消息行文字使用同一条左边界，并随统一 header 容器自然垂直居中。窗口使用隐藏标题栏，不为系统标题栏增加纵向留白。侧边栏关闭时，恢复按钮必须位于 macOS 红绿灯安全区右侧，不得与红绿灯碰撞。

上下文条第三格显示真实分支名（如 `main`、`feat/x`；detached HEAD 显示确定性兜底值），不可点击，不使用「当前分支」「会话分支」等字面占位词。对话已经开始后，项目与工作空间同样是不可点击的文本，只有团队仍可改选。

### 一轮结束

```text
│ (开) 开发                                                   │
│ 已补上空状态验收语句。                                      │
│ ┌─────────────────────────────────────────────┐             │
│ │ ✓ 已结束 · 2 个文件有改动          [查看]  │             │
│ └─────────────────────────────────────────────┘             │
```

```text
   ┌──────────────────────────────────────┐
   │ ✓ 默认工作空间                       │
   │   直接改项目文件夹里的文件           │
   │ ──────────────────────────────────── │
   │   独立工作空间                       │
   │   把改动隔离在一份副本里             │
   └──────────────────────────────────────┘
```

项目文件夹不是 git 仓库时，在同一菜单内禁用并解释：

```text
   ┌──────────────────────────────────────┐
   │ ✓ 默认工作空间                       │
   │   直接改项目文件夹里的文件           │
   │ ──────────────────────────────────── │
   │   独立工作空间                  （灰）│
   │   这个项目文件夹不是 git 仓库，       │
   │   无法隔离改动                       │
   └──────────────────────────────────────┘
```

### 选择独立工作空间的说明

工作空间只在发出第一条消息之前可选。选择独立工作空间时说明它的后果：

```text
   ┌──────────────────────────────────────────────┐
   │ 换成独立工作空间                             │
   │                                              │
   │ 副本基于项目当前所在的提交，不包含你还没     │
   │ 提交的改动。                                 │
   │                                              │
   │                      [取消]  [换过去]        │
   └──────────────────────────────────────────────┘
```

对话开始后工作空间锁定，没有切回默认工作空间这条路径，也不存在对应的确认弹层。

### 团队按钮展开

```text
   ┌──────────────────────────────────────┐
   │ ✓ 开发团队                           │
   │   营销团队                           │
   │ ──────────────────────────────────── │
   │ 这段对话用的是开始时载入的那份团队    │
   │ 内容，之后在团队页的修改不影响它      │
   └──────────────────────────────────────┘
```

### 运行中改选团队

```text
│   ┌──────────────────────────────────────────────────────┐  │
│   │ ▱ 项目 A  默认工作空间  main  [◇ 营销团队 ▾]         │  │
│   │ 当前这一步跑完后换成营销团队                         │  │
│   │ 说点什么，或 @ 一个成员…                          ↑ │  │
│   └──────────────────────────────────────────────────────┘  │
```

按钮显示用户刚选择的目标值；当前步骤尚未结束时，下方一行说明该值会在当前步骤跑完后生效。四项上下文里只有团队可以这样改选——项目与工作空间在发出第一条消息后已经锁定。

### 四种事实

```text
一步没跑起来
│ (开) 开发                                                   │
│ 正在补空状态验收语句…                                       │
│                                                             │
│ ⚠ 这一步没跑起来（退出码 1）                     [重试]     │
│ ─────────────────────────────────────────────────────────── │

一步卡住了
│ ⚠ 这一步卡住了                                   [重试]     │
│ ─────────────────────────────────────────────────────────── │

你按了停（中性，不触发红点）
│ ⏹ 你让这一步停下了                                          │
│ ─────────────────────────────────────────────────────────── │

反复重试仍未成功（触发红点，不给重试）
│ ⚠ 这一步反复没跑起来，已经不再重试                          │
│   你可以说点什么，或换一个成员接手                          │
│ ─────────────────────────────────────────────────────────── │
```

四条彼此可区分，且都由落库的事件类型驱动，因此刷新或重启后仍在。

最后一条不给「重试」按钮：产品已经替用户试过多次，此时提供下一步出路，不再把重复动作交回用户。

文案说「这一步没跑起来」，不说「开发失败了」；这四种事实来自机器故障或用户动作，不代表团队协作失败。

### 三种不可继续状态的共同表现

```text
项目文件夹不可用
│ 官网落地页验收                                              │
│ ─────────────────────────────────────────────────────────── │
│ （历史只读）                                                │
│ ─────────────────────────────────────────────────────────── │
│   ┌──────────────────────────────────────────────────────┐  │
│   │ [▱ 项目 A ⚠]  默认工作空间  main  [◇ 开发团队]       │  │
│   │ 项目文件夹找不到了，修复后才能继续      [修复…]      │  │
│   └──────────────────────────────────────────────────────┘  │

团队已删除
│   │ [▱ 项目 A]  默认工作空间  main  [◇ 开发团队 ⚠ ▾]     │  │
│   │ 这支团队已经被删除，改选一支团队后可以继续           │  │

团队需要修复
│   │ [▱ 项目 A]  默认工作空间  main  [◇ 开发团队 ⚠ ▾]     │  │
│   │ 这支团队需要修复            [去修复]                 │  │
```

三种状态共用同一套骨架：历史只读、输入框被说明文案替换、对应的上下文按钮标红。区别只在说明文案和恢复动作。

「团队已删除」通过标红的团队下拉改选一支可用团队；「团队需要修复」在修复完成后自动恢复，不要求用户回到对话再操作一次。

### 侧边栏状态点的来源

```text
│ ▾ 项目 A       │
│    落地页验收 ● │  ← 红：这段对话里有没跑起来 / 卡住 / 反复重试仍未成功
│    构建脚本   ● │  ← 蓝：没人在工作，且最后一条没提及任何人
│    文案打磨   ◌ │  ← 闪：有成员正在工作
│    需求梳理     │  ← 无点：最后一条提及了某个成员，团队内部仍在传递
```

用户按下「停下」之后是无点，不是红点。正常完成也是无点。

### 子会话卡片

```text
│ (经) 开发经理                                               │
│ 这个目标要动三个模块，我拆成三段并行推进。                  │
│ ┌─────────────────────────────────────────┐                 │
│ │ ▸ 落地页文案          开发    进行中    │                 │
│ │ ▸ 空状态验收          测试    没跑起来  │                 │
│ │ ▸ 构建脚本            开发    已结束    │                 │

```

点击任意一行在[主页面右侧栏](./main-right-sidebar.md)开出该子任务的标签，父会话留在视野中：

```text
┌ 侧边栏 ──┬ 主会话 ────────────────────┬ 右侧栏 ─────────────┐
│          │ (经) 开发经理              │ 改动 × │ 空状态验收 ×│
│          │ ┌──────────────────┐       │ ─────────────────── │
│          │ │ ▸ 落地页文案     │       │  该子任务的内容、   │
│          │ │ » 空状态验收     │       │  状态、输入与操作   │
│          │ │ ▸ 构建脚本       │       │  由右侧栏文档定义   │
│          │ └──────────────────┘       │                     │
└──────────┴────────────────────────────┴─────────────────────┘
```

子会话不在侧边栏重复出现，侧栏只保留父会话入口：

```text
│ ▾ 项目 A       │
│    官网落地页  │
```

窗口收窄时，上下文按分支、工作空间、团队、项目的顺序逐项收敛：

```text
宽    [▱ 项目 A] 默认工作空间 [main] [◇ 开发团队 ▾]
↓     [▱ 项目 A] 默认工作空间        [◇ 开发团队 ▾]
↓     [▱ 项目 A]                     [◇ 开发团队 ▾]
↓     [▱ 项目 A]
窄    （上下文项全部隐藏）
```

窄窗口展开子任务时右侧栏覆盖整个主内容区：

```text
┌ 右侧栏 ────────────────────────────┐
│ 空状态验收 ×                   [×] │
│ ────────────────────────────────── │
│  该子任务的内容、状态、输入与操作  │
│  由右侧栏文档定义                  │
│                                    │
└────────────────────────────────────┘
```

## 区域与信息

### 页面标题

- 显示这段对话的标题；还没有消息时显示「新对话」。
- 标题取用户第一条消息的开头文字，在发出第一条消息时确定，之后不随对话内容变化。
- 长标题单行截断，悬停显示完整标题。
- 本版不提供手动修改标题的入口。用户第一句话的开头文字高度雷同时的定位问题记入「待讨论」。

### 上下文

输入框上方按固定顺序显示这段对话的四项上下文：

1. **项目**：这段对话在哪个本地文件夹工作。对话必须归属一个可用项目。
2. **工作空间**：这段对话以什么方式改动文件。「默认工作空间」直接改项目文件夹里的文件；「独立工作空间」把改动隔离在一份副本里，不弄脏原文件夹。
3. **分支**：只显示不可点击，显示**真实的分支名称**，不使用「当前分支」这类字面词。用户要凭它在自己的编辑器或命令行里找到团队正在改的地方。
4. **团队**：负责这段对话的 Agent 团队。

工作空间是**这段对话**的设置，不是项目的设置。同一个项目下，一段对话可以直接改文件，另一段对话可以隔离改动，互不影响。

项目文件夹不是 git 仓库时无法隔离改动，「独立工作空间」不可选并说明原因。

按[Agent 团队](./agent-teams.md)的规则，一段对话使用的是创建时载入的那一份团队内容。用户之后在团队页修改成员的 `AGENT.md`，不会改变已经开始的对话。团队按钮必须让用户能看出这一点，否则用户会反复修改文件并困惑于行为为何不变。

### 时间线

时间线是一条按时间顺序的聊天记录，混合用户、团队成员和必要的系统记录。

每条记录只显示发言者和内容。时间只在悬停时显示。

**能被对话内容本身表达的，不给状态标记。** 成员在回复、在 @ 别人，本身就说明了交棒；成员正在吐字，本身就说明了运行中。产品不额外渲染「已交棒」「已完成」「运行中」这类过程标记，也不让它们在对话结束后留在历史里。

只有对话内容表达不了的四种事实才升格为可见状态：**一步没跑起来、一步卡住了、用户按了停、反复重试仍未成功**。这四种全部来自机器故障或用户动作，不是团队协作的失败，文案上说「这一步没跑起来」，不说「开发失败了」。

正在工作的成员**只轮播它当前最新的一句**，随着工作推进不断被下一句替换，不在时间线里堆积。这一句的作用是让用户知道它在动、动到哪一步了，不是让用户逐条读完。

**完整输出不进时间线**。正在工作的记录上提供一个查看完整输出的入口，点击后在[主页面右侧栏](./main-right-sidebar.md)开出该步骤的过程标签，包含原始错误在内的全部输出都在那里，可以一直留着对照。这一步结束后入口仍然保留，历史记录随时可以重新展开。

这样切分的理由：产品尚未稳定，用户需要监督的是交棒流程本身走没走对，而这件事看最新一句就够了；真要逐行读输出的时刻是少数，且往往需要跟改动对照着看——那正是右侧栏的场景。把全量输出留在时间线里，会让用户每次翻阅历史都要滚过大段机器输出，也让右侧栏的过程标签变成同一份内容的第二份拷贝。

本版不服务「离开一段时间后回来快速读懂全部经过」这个场景，也不在界面上暗示自己能做到。

### 运行中的操作条

- 只挂在当前正在工作的那一条记录末尾，提供「停下」和「完整输出」。
- 不显示计时。
- 这一步结束后「停下」消失；「完整输出」保留在历史记录上，用户之后仍然可以调出这一步的全部输出。

### 结果卡片

- 一轮工作结束、没有成员在继续接力时，时间线末尾出现一张结果卡片，说明**这段对话期间有几个文件发生了改动**，并提供查看入口。
- 卡片只给数量与入口，不在时间线里铺开文件清单。铺开就又变成了噪音，而看清单本身是[主页面右侧栏](./main-right-sidebar.md)的职责。
- 点击查看入口打开右侧栏并落在改动标签。
- 措辞不声称这些改动由团队成员造成。产品无法区分改动来自成员还是用户自己动手，理由与右侧栏一致。
- 没有任何文件发生改动时，卡片如实说明这一点，不省略——「跑完了但什么都没改」也是用户需要知道的结果。
- 项目文件夹不是 git 仓库时不出现结果卡片，因为这类项目没有改动视图。

### 子会话卡片

- 团队把目标拆成多个子任务时，时间线中出现一张卡片，逐行列出子任务、负责成员和**该子任务当前的状态**。
- 子任务的状态必须出现在卡片上。「能被对话内容本身表达的不给状态标记」这条规则不适用于子会话：子会话的内容不在主时间线上，主时间线表达不了它。卡片是唯一的聚合入口，不给状态用户就只能挨个点开才知道跑完几个、有没有一个挂了。
- 子会话不出现在侧边栏。子任务是团队的内部产物，不是用户主动管理的对象；让它们进入侧边栏会使侧栏长度随团队的拆分行为膨胀。
- 点击卡片行在[主页面右侧栏](./main-right-sidebar.md)开出该子任务的标签，父会话留在视野中。标签内的内容、状态表现、输入与操作由右侧栏定义，本页只负责卡片何时出现、显示什么、点击后打开什么。

### 输入框

- 用户输入自然语言，可以用 `@` 提及当前团队的成员。
- 提示文案在新对话时为描述目标，在已有历史时为继续说话或提及成员。
- 未发送的草稿在切换对话、切换页面、窗口缩放和**重新启动应用**后都保留。新对话页虽然不创建对话，草稿本身仍然必须留住——描述目标是用户在这个产品里花时间最多的动作。

## 操作与反馈

### 创建一段对话

- 进入新对话页不创建任何东西。用户点开又退出，不留下空白对话。
- 发出第一条消息时才真正创建这段对话，侧边栏随即出现对应的行并选中。
- 创建过程中防止重复提交；创建失败时保留全部已填内容并显示可理解的原因。

### 选择项目与添加项目

- 项目按钮列出全部可用项目，末尾是「添加项目…」。
- 从侧边栏顶部进入时项目为未选定状态，用户必须显式选择或添加。
- 「添加项目…」通过系统文件夹选择器添加本地项目，成功后直接成为这段对话的项目。
- 添加被取消或失败时不创建项目，也不改变当前选择，页面其他输入保留。
- 所选文件夹已经绑定一个活动项目时不重复添加，提示用户该文件夹已被使用。
- 项目在发出第一条消息后锁定，显示为不可点击的文本。归属一旦形成就不再改变。

### 选择工作空间与团队

- **工作空间只能在发出第一条消息之前选择，之后与项目一同锁定**，显示为不可点击的文本。
- 锁定的理由：[主页面右侧栏](./main-right-sidebar.md)以「这段对话开始时的状态」为基线回答「这段对话改了什么」，而这个问题只有在一段对话从头到尾都在同一个地方工作时才有答案。中途换地方，改动会分散在两处，用户唯一想问的问题就变成没法回答的问题。
- 选择独立工作空间时，隔离副本基于项目当前所在的提交，**不包含用户尚未提交的改动**。界面必须说清楚这一点。
- **团队仍然可以在对话进行中切换。** 换人不改变工作现场，与工作空间不是一回事。
- 切换团队时如果有成员正在工作，**当前这一步跑完再生效**，不中止它、不留下半截改动。这与[主页面侧边栏](./main-left-sidebar.md)对项目文件夹失效的处理保持一致。
- 生效后不重放已经完成的步骤。
- 团队预选上一次成功创建对话时使用的团队；没有历史选择或上次团队已不可用时，回退到软件自带的第一支团队。
- 切换团队保留当前对话和全部已有上下文，由新团队接管之后的推进。

### 说话与提及

- 用户不提及任何人时，消息发给当前团队的主 Agent。主 Agent 是这支团队接收新任务、接回团队结果并向用户收尾的主理人。
- 用户提及某个成员时，消息先直接发给该成员，但不改变主 Agent 对本轮的最终责任；该成员完成一次性回答或成员接力后，控制权仍回到主 Agent，不设直接点名例外。
- 成员回复中有合法 `@` 时按显式交棒继续；成员回复没有合法 `@` 且该成员不是主 Agent 时，系统只按团队结构把下一棒交回主 Agent，不根据正文内容猜测结论。主 Agent 回复没有合法 `@` 时，本轮自然结束。
- 主 Agent 可以在接回控制权后继续派工、询问用户或自由形成收尾文案，产品不规定固定成员顺序、固定验收阶段或固定收尾模板。
- 从专业成员完成回复到主 Agent 接回并完成收尾之间，整段接力仍属于进行中；侧边栏不得提前显示蓝点，子会话不得提前显示“已结束”，结果卡片也不得提前出现。
- 如果当前成员被停下、没跑起来、卡住，或项目 / 团队已经不可用，系统保留真实异常与恢复入口，不伪造一条主 Agent 成功收尾；“最终回主 Agent”约束正常完成的成员接力，不覆盖失败和用户主动停止。
- 提及一个**正在工作**的成员，等于让它停下当前这一步，带着新的指令重新开始。主 Agent 转达用户的话时具备同样的效果。
- 这是用户已有的中断权的代行，不改变团队成员之间的扁平关系。

### 专业判断与程序状态

- 测试、复核、“通过”“不通过”和“验收”等词都可以出现在普通 Agent 对话中，不因角色或关键词自动生成程序状态、系统提示、返工子任务或父任务汇合。
- 专业成员仍应按各自 `AGENT.md` 提供判断、证据和下一步建议；主 Agent 读取这些内容并对会话下一步负责。
- 当前版本不提供机器可判定的子任务语义完成或自动汇合。复杂并行任务是否全部满足目标暂由主 Agent判断，并列入待讨论的结构化 Agent 裁决协议缺口。
- 创建子会话属于明确的程序副作用，可以继续使用受控结构化事件；其中任务检查点是可选上下文，不要求 formal acceptance statements。旧输出中的验收语句只兼容显示为任务检查，不触发验收入账或自动汇合。

### 停下

- 「停下」中止当前这一步，是中性事实，不是失败。
- 停下之后可以继续说话，开启新的一轮。

### 重试

- 一步没跑起来、卡住或反复失败时，系统记录旁提供「重试」。
- 用户也可以不重试，直接说话或换一个成员接手。
- 重试到底重跑什么、会不会重复产生文件改动，记入「待讨论」。

### 侧边栏状态点的触发

会话页的事实决定[主页面侧边栏](./main-left-sidebar.md)上这段对话显示哪种点。本节是红点触发来源的唯一定义处，侧边栏文档引用本节，不重复列举：

| 侧边栏的点 | 会话页的事实 |
| --- | --- |
| 红点（需要你处理） | 一步没跑起来、一步卡住、反复重试仍未成功；以及三种不可继续状态（项目文件夹不可用、团队已删除、团队需要修复） |
| 蓝点（有新结果） | 没有任何成员在工作，最后一条消息没有提及任何人，且用户尚未查看这些新结果 |
| 闪烁点（正在运行） | 有成员正在工作 |

只要最后一条消息提及了某个成员，就属于团队内部仍在传递，既不是红点也不是蓝点。

三种不可继续状态同时触发红点和会话页的只读表现，两者不互斥：用户不在这段对话里的时候，红点是唯一能把他叫回来的东西；他回来之后看到的就是只读表现和对应的恢复入口。

蓝点的清除条件由[主页面侧边栏](./main-left-sidebar.md#选择对话)定义——打开这段对话并成功展示最新结果后清除。本节给的是它何时**出现**，两者合起来才完整；只按出现条件实现会让蓝点永不消失。

用户按下「停下」不触发红点：那是用户自己刚做的动作，不需要产品回头提醒他。

红点只由异常触发，因此用户被红点带回这段对话时，页面上一定有一条说明发生了什么的系统记录，不会出现「被叫回来却看不出要我干什么」的情况。

让成员主动声明「我需要用户裁决」是更精确的红点来源，需要另行约定，记入「待讨论」。界面上不出现「等你回话」这类词，也不为此增加额外视觉元素。

## 页面状态

| 状态 | 表现 | 可用操作 |
| --- | --- | --- |
| 新对话·项目未选定 | 引导语，工作空间与分支不显示 | 选项目、添加项目、改团队；发送禁用 |
| 新对话·项目已选定 | 引导语与四项上下文 | 改项目、改工作空间、改团队、发送 |
| 团队推进中 | 聊天记录，末条实时输出并带操作条 | 说话、提及成员、停下 |
| 有新结果待看 | 页面无特殊表现，侧边栏出现蓝点 | 说话、提及成员 |
| 一步没跑起来 | 系统记录说明原因并提供重试，侧边栏红点 | 重试、说话、提及成员 |
| 一步卡住 | 同上，与没跑起来分开表述 | 重试、说话、提及成员 |
| 已停下 | 中性系统记录，不触发红点 | 说话、提及成员 |
| 反复重试仍未成功 | 系统记录说明已停止重试，侧边栏红点 | 说话、提及成员 |
| 项目文件夹不可用 | 历史只读，项目按钮标红并提供修复 | 修复项目文件夹；输入与发送禁用 |
| 团队已删除 | 历史只读，团队按钮标红 | 改选团队；输入与发送禁用 |
| 团队需要修复 | 历史只读，团队按钮标红 | 去修复或改选团队；输入与发送禁用 |
| 加载中 | 保持时间线结构的加载占位 | 等待结果 |
| 加载失败 | 保留页面框架，显示原因与重试 | 重试、切换对话 |

### 三种不可继续状态的共同规则

项目文件夹不可用、团队已删除、团队需要修复使用同一种表现：历史只读、输入与发送禁用、对应上下文按钮标红。区别只在说明文案和恢复路径。

**恢复条件必须明确**：

- 项目文件夹修复成功后，这段对话恢复输入、发送和团队推进能力，历史与选择不变。
- 团队被删除后改选另一支可用团队，这段对话恢复推进能力；已有历史保留，新团队从当前上下文接手。
- 团队修复完成后，这段对话自动恢复推进能力，不需要用户再操作一次。

**这三种情况在有成员正在工作时发生**：已经拥有有效隔离副本、能够安全完成的执行跑完当前这一步再停止；依赖已经不可用的项目文件夹或团队内容、无法安全继续的执行立即停止并留下可读的系统记录。产品不得在执行已经无法继续时仍显示成员正在工作。

## 弹层与危险操作

- 添加项目使用系统文件夹选择器，不删除、移动或修改用户文件。
- 「默认工作空间」直接改动项目文件夹里的文件。
- 「独立工作空间」把改动隔离在副本中，副本基于项目当前所在的提交，不包含用户尚未提交的改动。
- 从独立切回默认前必须说明之后的改动会直接落在项目文件夹里。
- 「停下」中止当前这一步，已经产生的文件改动不会被回滚，界面不得暗示可以撤销。
- 本页不提供删除对话、删除消息或删除文件的能力。

## 响应式与窗口行为

- 时间线是页面唯一的主要滚动区域；页面标题和输入框保持可达。
- 有新内容且用户停留在底部时自动跟随；用户已经向上翻阅时不打断，提供回到底部的方式。
- 窗口变窄时上下文按钮逐项收敛，项目和团队最后收敛。
- 右侧栏在窄窗口下的覆盖行为由[主页面右侧栏](./main-right-sidebar.md)定义；从覆盖状态关闭后，会话区回到原滚动位置。
- 侧边栏关闭时，页面左上角提供「打开侧边栏」按钮。
- macOS 集成标题栏下正确避让系统交通灯并保留可拖拽区域。
- 长文本换行不撑破布局；代码与命令输出横向滚动，不撑宽页面。

## 待讨论

以下问题已经在评审中被提出，本版**明确不作答**，不得由实现自行决定：

### 用户拿不回 Agent 动过的东西

「看不见」这一半已经解决：一轮结束的结果卡片说明有几个文件发生改动，[主页面右侧栏](./main-right-sidebar.md)承载文件清单与行级对比。产品不设独立的交付结果陈列区仍然是有效结论——右侧栏是按需打开的证据区，不是常驻陈列区。

「拿不回来」这一半仍然悬空：「停下」之后已经产生的半截改动怎么办；用户看到不该有的改动时能否在产品内还原；选择独立工作空间的用户如何把成果带回项目；首次使用时默认值恰好是最不可逆的那个，是否需要提醒。

这四件事共同决定用户敢不敢让团队动他的真实项目，跨越会话页、右侧栏和项目设置，需要一次完整裁决，可能由一份流程 PRD 承接。

### 出问题时给用户的信息不够

- 「一步卡住」的判据是什么，以及在没有常驻计时的前提下，卡住之后是否显示已经多久没有动静。
- 「重试」的语义：重跑这一步，还是让成员重新思考；上一次已经产生的文件改动会不会被重复写入；产品自身的自动重试次数是否对用户可见。

### 用户想补一句话而不打断

当前规则下，用户在团队工作期间只有两个选择：不说话，或者承担一次会留下半截改动的中断。是否提供「排队到当前这一步结束后再送达」的补充方式，尚未裁决。

### 其他

- 成员如何主动声明「我需要用户裁决」，以及它是否成为红点的来源。
- **结构化 Agent 裁决协议**：当前不自动判断子任务是否通过或多个子任务是否可以汇合。未来若引入机器处理，必须使用显式、版本化、绑定任务标识和裁决权限的结构化事件（例如受控 JSON），并定义证据、幂等、重试与错误可见性；不得重新回到自然语言关键词推断。
- 是否允许用户修改对话标题。自动取第一句开头文字在真实使用中可能高度雷同。

## 关联流程

以下流程将在对应流程 PRD 建立后链接，本文档只记录会话页的职责：

- 添加第一个项目并开始第一段对话。
- 团队接力推进一个目标直到结束。
- 从没跑起来、卡住或停下的状态恢复。
- 团队拆出子任务并各自推进。
- 会话进行中切换团队或改变工作空间。

## 指标与验收

### 产品指标

- 用户从进入新对话页到发出第一条消息所需的操作次数。
- 因为界面没说清而重复发送同一指令的次数，目标为零。
- 点开又退出新对话页产生的空白对话数量，目标为零。
- 用户提及成员纠正方向后，团队真正改变行为的比例。
- 红点对话中真正存在异常的比例，目标为全部；正常完成的对话被误标为红点的次数，目标为零。

### 验收标准

1. 从侧边栏顶部进入新对话页时不创建任何对话，且**项目为未选定状态**；退出后侧边栏不新增行，也不留下空白对话。
2. 从项目行进入时该项目已被选中，且与顶部进入是同一个页面。
3. 发出第一条消息后对话才被创建，侧边栏出现对应行并选中，标题取自该条消息的开头文字。
4. 项目未选定或一个项目都没有时，草稿可以输入、团队可以改选，但发送被禁用；从项目按钮可以添加项目并直接用于本次对话。
5. 项目与工作空间在发出第一条消息后一同锁定为不可点击文本；团队在对话进行中仍可切换。
6. 切换团队时，正在工作的成员跑完当前这一步再生效，不产生半截改动。
7. 同一个项目下的两段对话可以分别使用默认工作空间和独立工作空间，互不影响。
8. 项目文件夹不是 git 仓库时，「独立工作空间」不可选并说明原因。
9. 分支显示真实分支名称，用户可以据此在自己的编辑器或命令行中定位。
10. 时间线中不出现「已交棒」「已完成」「运行中」这类过程状态标记；时间只在悬停时显示。
11. 正在工作的成员只轮播当前最新的一句，输出不在时间线中堆积；其记录末尾提供「停下」和「完整输出」且不显示计时；这一步结束后「停下」消失，「完整输出」保留在历史记录上。
12. 没跑起来、卡住、已停下和反复重试仍未成功四种事实可见且彼此可区分，刷新页面或重启应用后仍然可见。
13. 界面文案不出现运行目录、工作目录、数据库路径或内部标识。
14. 用户不提及任何人时消息发给主 Agent；用户直接提及其他成员时先由该成员执行，但本轮最终仍回到主 Agent 可见收尾；提及某个正在工作的成员会让它停下并带着新指令重新开始。
15. 团队拆出子任务时时间线出现卡片，**每行显示子任务、负责成员和当前状态**；子会话不出现在侧边栏；点击卡片行在右侧栏开出该子任务的标签，父会话仍可见，关闭后回到原滚动位置。
16. 侧边栏红点只由没跑起来、卡住、反复重试仍未成功触发；蓝点只在后端确认没有待处理控制工作且存在未读最终结果时触发，不再由“最后一条是否提及成员”另行推断；用户按下「停下」不触发红点。任务正常完成不产生红点。
17. 项目文件夹不可用、团队已删除、团队需要修复三种情况使用同一种不可继续表现；各自的恢复动作完成后，这段对话恢复输入、发送和推进能力。
18. 上述三种情况在有成员正在工作时发生，能安全完成的执行跑完当前一步再停止，无法继续的立即停止并留下可读记录；界面不再显示成员正在工作。
19. 未发送的草稿在切换对话、切换页面和重启应用后仍然保留。
20. 团队按钮能让用户看出这段对话使用的是创建时载入的团队内容，之后在团队页的修改不影响它。
21. 窗口变窄时上下文按钮逐项收敛，项目和团队最后收敛；窄窗口下右侧栏覆盖整个主内容区。
22. 一轮工作结束后，时间线末尾出现结果卡片，说明有几个文件发生改动并能一步打开右侧栏的改动标签；没有文件改动时如实说明；措辞不声称改动由团队成员造成。
23. 非主 Agent 有合法 `@` 时按显式接力继续，无合法 `@` 时下一棒确定性回到主 Agent；主 Agent 无合法 `@` 时本轮结束，不产生自我循环。
24. 任意 Agent 正文出现“验收”“通过”或“不通过”都不会触发本地程序化验收、吞掉同消息交棒或生成缺少验收语句的系统提示。
25. 非主 Agent 无 mention 回复已经落库但主 Agent 尚未收尾时，会话和子会话仍显示进行中；只有主 Agent 无 mention 回复完成处理后才进入本轮结束状态并产生新结果提示。
26. 会话、侧边栏、子会话卡片和结果卡片使用同一个“是否还有控制工作待处理”事实；该事实不等于任务成功或验收通过。运行中切换团队后由新团队主 Agent接管后续收尾；失败、卡住、停下或不可用状态不被伪装成成功收尾。
27. 本地结构化 child descriptor 没有 formal acceptance statements 时仍能创建子会话；可选检查点只显示为“任务检查参考”，不产生验收事实或自动汇合。

## 现状参考与产品缺口

现有本地对话操作台已实现单条时间线、实时输出、停下、失败与卡住记录、项目选择、工作空间切换和成员提及。这些是现状参考，不是本文的需求上限。

相较本文定义，当前实现的主要缺口：

- **工作空间是项目级设置**：现在切换一次会波及该项目下的所有对话；本文要求它是每段对话各自的设置。
- **本地对话仍会回退读取共享的 Agent 素材目录**：本文与[Agent 团队](./agent-teams.md)一致，Agent 只来自团队，没有脱离团队的全局来源。
- 输入框上方没有团队按钮，团队无法在对话中改选。
- 分支显示的是「当前分支 / 会话分支」字面词，不是真实分支名。
- 从项目行新建会立即创建并落盘一段空白对话，而不是进入不落盘的新对话页；从顶部进入没有对应形态。
- 添加项目不在项目按钮中。
- 时间线渲染「已完成」「已显示」等过程状态标记，未按本文收敛。
- 系统记录中仍会出现运行目录等机器信息。
- 子会话在侧边栏平铺，没有卡片入口、卡片状态，也没有承接它的右侧栏。
- 对话标题、侧边栏状态点的触发与联动尚未建立。
- 三种不可继续状态的恢复条件与运行期行为尚未实现。

`docs/wireframes/pages/console.md` 中与主内容区有关的内容由本文接管，该文件不再新增会话页事实。原[新建对话窗口](./new-conversation.md)定义的独立窗口被本文取代：新建对话与会话页是同一个页面，添加项目和团队选择改由页面内的上下文按钮承接。
