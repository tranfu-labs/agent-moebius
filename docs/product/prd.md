## 产品命名

产品的统一品牌名为 `Moebius`，面向用户的产品名称、仓库内技术标识、应用配置与本地运行目录都必须围绕这一名称保持一致，避免同一产品同时存在多套名称。

这次命名切换是一次明确的破坏式变更：产品不继续识别旧名称对应的应用目录、环境覆盖或运行协议。当前 worktree 的磁盘路径与 GitHub remote 属于外部仓库状态，不由应用命名规则修改。

## 产品定位

Moebius 是一款面向 Codex、Claude Code 同类用户的本地多 Agent 协作产品。

用户用自然语言定义 Agent 及其团队，让多个专业角色围绕同一任务自动推理、执行、审查、交棒和验收。产品的目标不是同时调用更多 Agent，而是让 Agent 形成可以持续推进和自我校正的团队，尽可能减少人类充当消息中转站、流程监督者和重复验收者。

## 目标用户

核心用户是已经使用或愿意使用 Codex、Claude Code 等编码 Agent，并希望任务执行进一步自动化的人。

这类用户通常：

- 能够使用自然语言描述任务、角色职责和项目规则。
- 已经接受 Agent 读取本地项目、运行命令和产出修改。
- 不满足于单轮问答或一个 Agent 独立完成全部工作。
- 希望方案、专业审查、实现和验收能够由不同 Agent 自动接力。
- 希望减少重复补充上下文、手动复制结果、指定下一角色和催促下一步。
- 仍然要求过程可见、失败可恢复、结果可验收。

当前不对用户职业、团队规模或商业属性做进一步细分。

## 核心问题

单个编码 Agent 可以完成复杂工作，但一项任务从提出到可靠交付，通常仍需要用户持续参与：

1. 把需求交给执行 Agent。
2. 检查方案是否完整，是否遗漏模块影响。
3. 找其他角色提供架构、测试或用户视角的审查。
4. 把审查意见重新转述给执行 Agent。
5. 等待实现后再次安排测试和验收。
6. 发现问题时恢复上下文并组织返工。
7. 持续判断当前该由谁接手、为什么卡住、是否真的完成。

用户因此承担了团队调度、上下文传递和质量监督三种额外职责。角色和任务越多，人工干预越频繁，Agent 的自动化价值越容易被协调成本抵消。

## 价值主张

Moebius 让用户配置的是“Agent 团队”，而不是需要手工绘制的固定工作流。

团队中的 Agent 根据自然语言职责和协作规则自行判断、交棒、检查和修正。每支团队的主 Agent 对一轮会话拥有编排与收尾责任：它可以把工作交给任意成员，成员之间可以继续自然接力，但控制权最终回到主 Agent，由主 Agent 结合完整时间线决定继续派工、询问用户或给出可见收尾。用户主要参与目标表达、关键裁决和最终结果确认，不再负责在每个步骤之间复制信息和指定下一步。

产品提供四项核心价值：

1. **自然语言定义角色**：Agent 的职责、判断方式、协作边界和交棒条件以自然语言表达。
2. **团队自动接力**：一个 Agent 完成当前职责后，可以把任务继续交给合适的团队成员。
3. **专业分工保证质量**：不同 Agent 承担方案、架构、执行、测试和复核等不同职责，而不是重复回答同一个问题；专业结论作为主 Agent 判断下一步的输入，不由正文关键词直接驱动程序状态。
4. **过程可见且可恢复**：用户能看见当前角色、任务阶段、等待对象、失败原因和交付结果，并能从中断处继续。

## 核心产品概念

### Agent

Agent 是一个职责明确的专业角色，例如开发、开发经理、测试、产品经理、用户画像或设计师。

Agent 由自然语言文件定义，内容可以包括：

- 角色职责与能力边界。
- 应当读取和遵守的事实。
- 判断结果质量的方法。
- 与其他角色的协作方式。
- 何时交棒、交给谁，以及何时等待用户。

Agent 属于 Moebius 的应用级配置，不从属于某一个具体项目。同一个 Agent 可以加入多个团队。

### Agent 团队

Agent 团队是一组围绕同类任务协作的 Agent，是产品中由用户选择和管理的一等对象。

例如，开发团队可以包含：

```text
开发团队
├─ 开发
├─ 开发经理
└─ 测试
```

产品团队可以包含：

```text
产品团队
├─ 产品经理
├─ 用户画像
└─ 设计师
```

团队定义“有哪些专业角色共同负责这类任务”。团队成员的自然语言职责和交棒规则在实际任务中形成协作过程。

“工作流”不是当前需要用户单独创建、绘制或维护的一等对象。工作流是 Agent 团队处理具体任务时自然形成的运行过程。主 Agent 是会话主理人而不是其他成员的组织层级；它固定的是控制权归属与最终收尾，不固定成员执行顺序或专业判断方法。

### 项目

项目是用户允许 Moebius 工作的本地目录，也是会话、项目上下文和交付结果的工作现场。

项目提供：

- 本地工作目录。
- 代码、文档和其他项目事实。
- 会话与交付结果的归属。
- Agent 执行任务时需要的项目上下文。

项目不拥有 Agent 或 Agent 团队。同一套应用级团队可以服务多个项目。

### 会话

会话是用户在某个项目中围绕一个目标展开的持续任务空间。

每个会话：

- 属于一个项目。
- 创建时默认选择一个 Agent 团队。
- 保留用户、Agent 和系统共同形成的上下文。
- 记录 Agent 执行、交棒、等待、失败、返工和验收结果。
- 允许用户在需要时手动切换团队。

当前产品心智中，“发起任务”等同于“创建或进入一个会话并描述目标”，不另外向用户暴露一套与会话并列的顶层任务概念。复杂目标内部可以产生子任务或子会话，但其具体呈现方式仍待讨论。

切换团队时保留当前会话及已有上下文，由新团队接管切换后的任务推进；系统不新建会话，也不重新执行已经完成的历史步骤。

## 视觉语言原则

产品界面采用近黑底暗色优先的 dashboard 视觉语言，令牌级细节以 `packages/console-ui/DESIGN.md` 为包内事实源，本节只定义全局原则：

- 画布近纯黑、卡面微亮、描边可见，层级靠亮度与描边表达；圆角基线 14px。
- 运行状态一律渲染为状态 pill（状态图标 + tinted 底 + 同色描边），不使用圆点 + 裸文字。
- 状态色相：running 琥珀、pending 蓝、waiting 紫、completed/displayed 中性灰；等待类状态允许使用专属色相。
- 交互 accent 唯一靛蓝 `#5E6AD2`，亮暗双主题一致。
- 绿 / 红只用于验收裁决与危险事实；唯一例外是未读计数允许使用红色圆角标。
- 所有状态色令牌亮暗双主题成对定义，不允许单主题取值。

## Agent 与团队配置

Agent 和团队保存在 Moebius 的应用数据目录中，作为可跨项目复用的应用级配置。

产品应允许用户：

- 在首次启动时配置或选择 Agent 团队。
- 使用一套可以立即开始工作的默认团队。
- 查看一个团队包含哪些 Agent。
- 修改已有 Agent 和团队。
- 新建适合其他任务领域的 Agent 和团队。
- 在具体会话中选择默认团队，并在后续手动切换。

具体文件组织、团队定义格式和编辑方式尚未讨论，不在本版草案中提前规定。

## 运行环境与数据目录

产品层统一使用“Moebius 应用数据目录”描述 Agent、团队、配置和本地运行数据的归属。

- 打包后的生产环境默认使用 `~/.moebius`。
- 当前桌面开发环境默认使用项目仓库根目录作为数据目录。
- 开发、测试或高级使用场景可以通过 `MOEBIUS_DATA_ROOT` 显式覆盖数据目录。

产品界面不应把开发环境路径当作普通用户的默认心智。开发环境与生产环境必须保持数据目录边界清晰，避免测试配置、开发数据和真实用户数据相互污染。

## 核心用户旅程

当前确认的主旅程为：

```text
首次启动
  → 检查本地运行环境
  → 配置或选择 Agent 团队
  → 确认团队包含的 Agent
  → 添加本地项目
  → 在项目中创建会话
  → 默认选择一个 Agent 团队
  → 用户描述目标
  → 主 Agent 组织团队成员自动协作和交棒
  → 必要时等待用户裁决或允许用户切换团队
  → 专业成员把检查结果与执行证据交回主 Agent
  → 主 Agent 向用户收尾或继续推进
```

该旅程的正常路径已经确认；首次启动细节、页面承载方式、失败分支和恢复路径仍待后续讨论。

## 开发域 MVP

开发域是验证 Agent 团队价值的第一条样板场景。

默认开发团队至少包含：

- **开发 Agent**：理解需求、形成方案、实施修改并提供实现证据。
- **开发经理 Agent**：检查架构选择、模块边界、依赖影响和长期维护风险。
- **测试 Agent**：检查方案是否可验证，并在实现后执行独立测试或复核，把发现和证据交回主 Agent。

目标协作过程为：

```text
用户提出开发任务
  → 开发经理作为主 Agent 判断如何推进
  → 开发、测试和其他成员按各自 AGENT.md 自然接力
  → 专业成员给出方案、实现、测试、复核或返工意见
  → 接力链最终回到开发经理
  → 开发经理结合完整时间线继续派工、询问用户或收尾
```

MVP 必须证明：

- 用户发起任务后，多数协作节点可以自动推进。
- 每个 Agent 承担明确且不重复的职责。
- 上游审查意见能够进入后续执行，而不是停留在对话中。
- 方案、实现、测试和复核结论保留可追溯的对话与执行证据。
- 专业成员发现问题时能够携带依据交棒返工，且最终回到主 Agent 裁决。
- 中断或失败不会迫使用户从头描述任务。
- 用户不需要持续充当团队调度器。

当前版本不把 Agent 正文中的“验收”“通过”“不通过”等自然语言解释为机器控制事件，也不维护由这类文本自动生成的结构化验收事实。子任务是否满足目标、多个子任务是否可以汇合，暂由主 Agent 根据共享时间线与专业证据判断。未来如需可靠自动汇合，应另行设计由 Agent 显式输出、带版本、任务标识、裁决权限与证据字段的结构化事件协议；在该协议建立前，不用关键词推断补洞。

## 长期愿景

长期目标是让用户只参与真正需要产品判断和个人裁决的部分。

以产品设计为例：用户先通过对话确认产品目标和用户画像，后续由产品经理、用户画像、设计、开发和测试等 Agent 自动讨论并补全需求、交互、视觉、技术和验收细节；发现矛盾时自动回到对应角色修正，最终形成达到约定质量标准的可用应用。

开发域是第一条样板团队，不是产品能力的最终边界。后续可以通过新的 Agent 团队扩展到产品设计等其他领域。

## 成功指标

### 非预期人工干预率

一次任务中，因为重复补上下文、手动催办、纠正错误交棒或恢复卡住流程而需要真人介入的节点数，占全部协作节点的比例。

产品主要降低非预期人工干预。产品取舍、危险操作和用户明确保留的审批属于预期干预，不应被当作自动化失败。

### 自动闭环率

从用户描述目标到主 Agent 给出可见收尾，无需非预期人工干预的任务占比。

### 首次交付有效率

主 Agent 首次向用户收尾后，无需因遗漏、错误或未满足目标而重新打开返工的任务占比。

### 返工率与返工轮次

至少经历一次专业成员指出问题或主 Agent 判断未达目标并退回执行角色的任务占比，以及每项任务从首次方案到最终收尾的平均返工次数。

### 无效协作率

Agent 重复提问、重复审查、没有新增信息的交棒或错误路由，占全部 Agent 协作行为的比例。

### 任务成功率

在约定条件内完成用户目标、提供必要证据并由主 Agent 向用户收尾的任务占比。

自动化指标必须与任务成功率共同观察。减少人工干预不能以降低结果质量、隐瞒失败或省略任务所需的测试与复核为代价。

## 当前非目标

当前阶段不包含：

- 商业化、付费方案和收入模型。
- 按职业或团队规模建立精细用户分层。
- 建设复杂的可视化工作流编辑器。
- 要求用户学习平台专用工作流 DSL。
- 一开始覆盖所有知识工作领域。
- 以 Agent 数量或对话轮次作为产品价值指标。
- 为追求完全无人监督而牺牲过程可见性、恢复能力和结果质量。
