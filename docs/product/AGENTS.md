# agent-moebius 产品文档约定

## 用途

本目录承载 agent-moebius 的产品事实：产品为谁解决什么问题、由哪些页面和流程组成、用户可以完成什么，以及达到什么结果才算产品成立。

产品文档必须从完整产品和用户体验出发，不以现有代码、既有 OpenSpec、当前页面或短期实现成本反向限制需求定义。现有实现可以作为现状证据，但不能代替产品判断。

本目录不记录具体技术方案，不承担开发任务拆解，也不复制运行时实现规格。

## 文档结构

```text
docs/product/
├── AGENTS.md                 # 产品文档编写规则、模板与事实源边界
├── prompt.md                 # PRD 采访提示词，开新会话前整段粘给 codex/Claude
├── prd.md                    # 总 PRD
├── pages/                    # 页面级 PRD，一页一个文件
│   └── <page>.md
└── flows/                    # 跨页面流程 PRD，一个完整目标一个文件
    └── <flow>.md
```

目录按实际产品逐步建立。不得为了补齐结构而提前创建没有经过讨论的空页面或空流程文档。

## 三层产品事实

### 总 PRD

`docs/product/prd.md` 定义整个产品，至少回答：

- 产品定位、目标用户、核心问题与价值主张。
- 产品形态、核心概念与产品原则。
- 完整产品的信息架构与一级页面清单。
- 关键用户旅程和全局产品规则。
- MVP 范围、后续产品范围与明确非目标。
- 产品成功指标与非功能要求。
- 页面 PRD、流程 PRD 的索引和相互关系。

总 PRD 必须说明产品由哪些页面和模块组成，但不在一个文件里穷举每个页面的全部控件、状态和异常分支。

### 页面级 PRD

`docs/product/pages/<page>.md` 定义一个具体页面。页面级 PRD 同时承载该页面的产品要求和低保真至中保真界面表达，不再把页面结构单独外包给 Wireframe 文档。

每个页面级 PRD 至少回答：

- 用户为什么来到这个页面，离开时应完成什么。
- 页面从哪里进入，可以去往哪里。
- 页面包含哪些区域、信息和操作。
- 每个操作在什么条件下可用，触发后发生什么。
- 默认、空、加载、运行、等待、成功、失败、禁用等状态如何表现。
- 弹窗、菜单、抽屉、确认和危险操作如何处理。
- 窄视口或窗口尺寸变化时如何调整。
- 与其他页面和流程如何衔接。
- 该页面服务哪些产品指标，如何判断需求已被满足。
- 哪些能力明确不属于该页面。

一个页面只维护一份当前有效的页面级 PRD。页面内的弹窗、抽屉和局部模式默认归入该页面；只有具备独立入口、独立用户目标或跨多个页面复用时，才拆成独立页面或流程文档。

### 流程 PRD

`docs/product/flows/<flow>.md` 定义用户为了完成一个目标，如何跨越两个或以上页面、系统状态或参与角色。

每个流程 PRD 至少回答：

- 流程服务的用户目标与开始条件。
- 用户、桌面应用和 Agent 分别承担什么职责。
- 涉及哪些页面，按什么顺序流转。
- 正常路径、关键分支、异常路径和恢复路径。
- 何时自动推进，何时等待用户，等待什么。
- 中断、重试、退出和重新进入时如何继续。
- 什么状态代表流程完成，什么证据证明完成。

同一项规则若只影响单页，留在页面 PRD；若决定多个页面之间的衔接或一个端到端目标的完成条件，写入流程 PRD。

## 页面级 PRD 内容模板

页面文档按下列结构编写。没有内容的章节必须说明“不适用”及理由，不得静默省略关键状态。

```markdown
# <页面名称>

## 页面目标
<用户为什么来到这里，完成后得到什么>

## 用户场景
<哪些用户在什么情况下使用此页面>

## 入口与去向
<入口、退出、返回和跳转关系>

## 页面结构
<ASCII 界面图或等价的结构表达>

## 区域与信息
<逐区说明信息层级、内容来源和存在理由>

## 操作与反馈
<逐项说明触发条件、动作、结果和反馈>

## 页面状态
<默认、空、加载、运行、等待、成功、失败、禁用等状态矩阵>

## 弹层与危险操作
<弹窗、菜单、抽屉、确认、删除和不可逆操作>

## 响应式与窗口行为
<窄视口、窗口缩放、滚动和焦点行为>

## 关联流程
<关联的 flows 文档及本页面在流程中的职责>

## 指标与验收
<页面服务的产品指标和可核查的完成标准>

## 非目标
<明确不在本页面解决的事项>
```

### 页面结构图的要求

页面结构图是页面 PRD 的组成部分，不是装饰性草图。图中出现的区域、主要控件和重要状态必须能在后文找到对应说明；后文定义的核心操作也必须能在结构图或状态变体中找到位置。

页面结构图应表达：

- 页面层级和主要区域。
- 核心信息的相对优先级。
- 主要操作的可发现位置。
- 弹层、空态或关键状态下的结构变化。
- 必要的桌面宽屏与窄视口差异。

页面结构图不负责规定精确像素、最终颜色、字体数值、阴影参数、图像资产或复杂动效的逐帧参数。这些内容由高保真设计、设计系统或后续实现设计承接。

### 可选的 HTML 线框：验收对照件

页面 PRD 可以配一份同名的 `pages/<page>.wireframe.html`，浏览器直接打开，自带样式、不依赖外部资源。

它的职责是**实现完成后的人工验收对照**——把界面做出来之后，打开它对一眼区域关系、信息密度和状态表现对不对。它不是写需求时的依据，也不是给 AI 读的输入。

因此：

- **需求改动一律先落 md**。任何产品判断、区域归属、状态规则的变化都写进页面 PRD 正文，HTML 随后跟进。
- **HTML 允许落后于 md**，两者冲突时以 md 为准，HTML 文件头必须写明这一点。
- **HTML 里不得出现 md 中没有的事实**。它只是把正文已经写清的东西画出来，不承担新增定义。
- 页面 PRD 正文的字符图**不因为有了 HTML 而省略**。字符图服务评审与 diff，HTML 服务肉眼验收，两者读者不同。

两份内容描述同一批界面，长期必然出现不一致。这是明知的取舍：验收对照的价值大于维护两份的成本，代价由「以 md 为准」这条规则兜住。

### 可选的高保真交互原型

需要在正式开发前评审完整视觉、动效和跨状态交互时，页面或流程 PRD 可以链接同目录的 `<page>.prototype.html` / `<flow>.prototype.html`。

- 高保真原型是可离线直接打开的单 HTML 设计交付物，不是产品事实源或正式实现。
- 产品决策仍然先写入 PRD；原型只投影已经确认的事实，冲突时以 PRD 为准。
- authoring 源码、第三方动效组件、单文件构建、双向隔离与验证规则统一由仓库根 `prototypes/AGENTS.md` 管理。
- 原型不得 import 正式产品代码，正式产品也不得复用原型源码；两者只共同读取 PRD。
- Storybook 继续服务正式组件的开发展示，不承担完整页面高保真原型职责。

## 与其他事实源的边界

| 事实源 | 负责回答 | 不负责回答 |
| --- | --- | --- |
| `docs/product/prd.md` | 产品是什么、为谁服务、由哪些部分组成 | 页面控件细节、技术实现 |
| `docs/product/pages/` | 具体页面让用户完成什么，如何组织、交互和反馈 | 跨页面完整流程、底层实现 |
| `docs/product/flows/` | 用户如何跨页面和角色完成一个目标 | 单页全部视觉细节、底层实现 |
| `docs/roadmap/` | 哪些产品能力先做、后做，阶段如何验收 | 产品长期事实、页面完整定义 |
| `openspec/changes/` | 一次具体开发准备改变什么 | 完整产品应当是什么 |
| `openspec/specs/` | 当前已经实现并验证的行为是什么 | 尚未实现但产品必需的需求 |
| `docs/architecture/`、`docs/adr/` | 技术上如何实现以及为何这样实现 | 用户需求和页面产品判断 |
| `agents/` | 项目运行时如何用自然语言定义 Agent 行为 | agent-moebius 自身的产品 PRD |
| 高保真设计与设计系统 | 最终视觉语言、令牌、资产和精细动效 | 产品范围与业务取舍 |

产品文档可以引用这些事实源作为依据，但不得复制一份随后独立漂移的实现规则。产品要求与当前实现不一致时，应明确记录为产品缺口，后续再由 roadmap 和 OpenSpec change 承接。

## 旧 Wireframe 的迁移规则

`docs/wireframes/` 是项目早期为了快速实现采用的临时页面事实源，不是新的产品文档体系中的长期并列事实源。

迁移按页面逐步进行：

1. 对应页面 PRD 尚未建立时，旧 Wireframe 继续作为现状参考。
2. 编写页面 PRD 时，必须阅读对应 Wireframe，辨别哪些内容仍有效、哪些只是历史实现选择。
3. 有效的页面结构、状态和交互被吸收到页面 PRD，并补齐用户目标、操作规则、异常路径、指标与验收。
4. 页面 PRD 经确认后，成为该页面唯一的产品事实源；后续需求不再向对应旧 Wireframe 增加新事实。
5. 已迁移页面的旧 Wireframe 应标明其内容已由哪个页面 PRD 接管，避免继续被误用。
6. 所有页面完成迁移后，再单独决定整体退役 `docs/wireframes/`，不得在迁移未完成时一次性删除历史信息。

迁移不是把旧字符图原样搬到新目录。页面 PRD 必须重新从用户目标和完整产品出发判断页面，而不是把当前实现包装成需求。

## 需求进场：采访必须产出三段对照

产品事实源只有在需求进场时被读取和更新，才不会退化成一次性初稿。任何走 OpenSpec 变更流程的需求，采访环节 MUST 产出下面三段，并在方案落盘时把「期望」写入对应 PRD：

1. **现状**：这个需求现在被记录成什么？（读对应页面 / 流程 PRD 原文；读不到即为 PRD 缺口，当场说明，NEVER 拿旧 Wireframe 或现有实现顶替）
2. **期望**：改完之后应该被记录成什么？（用产品语言写决策与理由，这段话就是 PRD 的新句子）
3. **落点**：这句话写进哪个文件？（只影响单页 → `pages/<page>.md`；决定跨页衔接或端到端完成条件 → `flows/<flow>.md`）

第 1 段的答案直接决定本次变更性质，不需另行判断：

| 现状 | 性质 | 后续 |
| --- | --- | --- |
| PRD 已写清楚，实现没做到 | 实现缺口 | 不动 PRD，直接开 change |
| PRD 没写 | PRD 缺口 | 先补 PRD，再开 change |
| PRD 写了但这次要改 | 产品意图变更 | 改 PRD，同时记录旧决策为何被推翻 |

**写入时机**：采访确认决策，落盘时写入 PRD（与 `openspec/changes/` 同时），NEVER 推迟到归档——实现期间正是最需要 PRD 指导的时候。产品决策在采访中尚未拍板的探索性变更，「需求基线」标「待定」，决策定下后回填。

## 与 `openspec/specs/` 的分工：同一规则的两种投影

同一条规则在 PRD 与行为规格中都出现是正常的，两者不是粗细之分，而是投影不同。落笔时问：**「用户需求没变、只是实现方式变了，这句话要不要改？」**

- **不用改 → 属 PRD**：决策、理由、产品原则、指标。例：「红点优先于蓝点，因为人的阻塞比未读更紧急」「闪烁必须克制，不造成持续视觉干扰」。
- **要改 → 属 spec**：可判定的判据、字段名、枚举字面量、Scenario。例：`MUST apply the priority red > blue > blink`、辅助名称的具体文案。

由此得出 PRD 的写作纪律：

- PRD MUST NOT 写判据形式——不写实现字段名、不写枚举字面量、不写优先级表达式、不写可执行的 GIVEN/WHEN/THEN。需要精确判定的部分交给 `openspec/specs/`。
- `openspec/specs/` 的 Requirement 用 `Source: docs/product/...#<小节>` 指回 PRD，不复述决策理由。因此**页面 PRD 的小节标题是被外部引用的锚点，MUST 保持稳定**；确需改名时同步更新引用它的 spec。
- 两边都不得出现对方内容的原样副本。发现同一句话在两处并存，删掉投影错误的那一份，而不是同步维护两份。

## 编写与评审原则

- 一次只建立或评审一个产品文件，确认后再进入下一个文件。
- 先定义完整产品，再标注当前实现状态；不得只描述已经存在的功能。
- 页面必须覆盖正常状态和非正常状态，不能只写理想路径。
- “支持某功能”不是完整需求，必须说明入口、条件、结果、反馈和失败处理。
- 删除、移除、覆盖、外发等高影响操作必须说明对象、后果、确认和恢复能力。
- 产品语言面向用户心智，不使用数据库表、进程、cwd、runDir 等实现术语替代产品概念。
- 页面结构、正文规则、流程文档和验收标准必须相互可追溯。
- 发现产品文档与现有实现不一致时，记录缺口，不为了让文档看起来“已完成”而降低产品要求。

## 建立顺序

产品文档按以下顺序逐个讨论和建立：

1. 本 `AGENTS.md`：确认文档边界和模板。
2. `docs/product/prd.md`：定义完整产品、信息架构和一级页面。
3. 根据总 PRD 的一级页面清单，逐个建立页面 PRD。
4. 根据页面之间的真实关系，逐个建立跨页面流程 PRD。
5. 最后对照现有 roadmap、OpenSpec、Wireframe 和实现形成产品缺口清单；缺口排序属于后续产品规划，不回写为虚假的已实现事实。
